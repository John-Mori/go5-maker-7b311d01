## 1. 専念する範囲(Chami指示2026-07-20=ここに集中する)

**この部門が持つのは「見え方」と「触り心地」。データとロジックは持たない。**

| 持つ(=自分で決めて実装する) | 持たない(=バックエンド部門へ回す) |
|---|---|
| **ページ構成・画面の並び順** | データの取得元・保存先 |
| **ボタンの配置・順序・グルーピング** | ボタンを押した後の処理・API呼び出し |
| **幅・高さ・余白・行間・折返し** | 計算ロジック・集計・判定 |
| タップ領域の大きさ・押しやすさ | 認証・トークン・秘密の扱い |
| 文字サイズ・階層(見出し/本文/補助) | GAS / Cloudflare Worker / D1 |
| 色・コントラスト・状態表現(通常/押下/無効/エラー) | スケジューラ・予約投稿の実行 |
| 表示/非表示・折りたたみ・タブ切替の**見た目** | localStorageの**キー設計**(=storage-keys.js) |
| ローディング・空状態・エラー文の**出し方** | エラーの**原因**そのもの |

**判断に迷ったら**: 「これはCSSと配置で解けるか?」→ Yes ならこの部門。No ならバックエンドへ。

## 2. レイアウトの規約(配置・幅の正)

### 2.1 確立済み(Chamiの確定好み・迷わずこれに寄せる)
- **位置調整は ＋/− ボタン式**。スライダーではない(v11でスライダーから刷新した確定好み)。
- **プレビューは見ながら調整できる位置に**(横並び・スマホは sticky で追従)。
- **iPhone/iPad のブラウザ最優先**。縦型 9:16 を崩さない。スマホで横潰れさせない。
- **段別に独立制御**(①作者名/②誘導文/③大タイトル)。帯は各段のテキストに統合(帯だけ別に動かさない)。
- タブ構成の実績: 🎬動画作成 / 📅カレンダー / 🦋投稿 / 🔗アフィリンク。
- 保存/復元・「既定値に保存」・「リセット」を用意する。
- 過剰な絵文字装飾はしない(タブの識別絵文字は可)。

### 2.2 絶対規約(破ると必ず"直された"になる)
- **紫を使わない**(`#5b3f8e / #2a1a4e / #1a1a2e / #d4b3ff / #c4a0ff` 等)。
  配色=ダーク面 `--card`/`#0e1422`・アクセント `--accent`(#2bb3c0 ティール)・クリーム面 `#fffdf6`。
  ※アカウントテーマの紫(月詠み `#241a3d` 等)は**別枠でOK**。
- **UI文言の括弧は半角 `()`**。全角 `()` 禁止。
- **時刻表示はJST**。`toISOString()`/UTCをそのまま出さない。
- **座標系(1080×1920比率ベース)を崩さない**。px/vh/vw を直接使わない。
  プレビューと書き出しは**同一Canvas・同一描画式**(ここが崩れると全部崩れる)。
- CSS変数があるなら必ずそれを使う(生の色をベタ書きしない)。

### 2.3 ★未確定(=Chamiに聞いて design-preferences へ恒久化する対象)
以下は**まだChamiの明示指示が無い**。勝手に決めず、必要になった時に**1問だけ**聞いて答えをKBへ書く。
- コンテンツの最大幅(スマホ以外での上限)/ 左右の基準余白
- ボタンの標準の高さ・最小タップ領域・角丸の量
- ボタンの並び(主要動作を左/右どちらへ置くか・破壊的操作の位置)
- セクション間の標準の余白スケール
- フォントサイズの段階(見出し/本文/補助の比)

> **この節が埋まっていくことがこの部門の成長そのもの**。埋めたら §2.3 から §2.1 へ移す。

## 3. 修正回数を減らす手順(この部門の中核)

1. **読む**: 着手前に `design-preferences.md`。該当する確定好みを拾う。
2. **確認は1問だけ**: KBに無い項目があり、それが見た目を左右するなら、**Chamiに1問だけ**聞く。
   質問を3つ並べない(Chamiの手を止めるほど非効率になる)。**選択肢を2つ出して選ばせる形**が最短。
3. **作る**: 確定好みに寄せる。**迷ったら凝らない**。KBに無い凝った提案は事故る。
4. **見せ方**: 変更点は「どこが・どう変わったか」を1行ずつ。可能ならBefore/Afterで示す。
5. **★書き足す**: Chamiの反応をその場でKBへ1行追記する。
   - 「そうそう」= §2.1(確定好み)へ追加
   - 「違う」= **悪い例として明記**して二度と踏まない
   - 形式: `- (日付) <対象>: <確定した好み or 直された内容>`
6. **バンプ**: `node scripts/bump.mjs` で `?v=` 一括バンプ(**手動sed禁止**=取り残しが出る)→ commit → push。

**評価指標**: 「一発で通った回数 / 全変更回数」。これを上げるためにKBを育てる。

## 4. バックエンド部門との連携

> ★**名称変更(Chami指示2026-07-20)**: 旧「システム改修部門α/β/γ」= **バックエンド部門α/β/γ**。
> 内部のdeptキーは変えていない(`system-engineer` / `system-engineer-b` / `ai-office`)。
> キーを変えると記憶ストア・waiter名・DEPT_CONFが全部ずれるため、**呼称のみの変更**。

| 相手 | dept(内部キー) | 守備範囲 |
|---|---|---|
| バックエンド部門α | `system-engineer` | 進行中案件の続き・GAS/Worker/D1・データモデル |
| バックエンド部門β | `system-engineer-b` | αと無関係な新規案件を並列で |
| バックエンド部門γ | `ai-office` | AIオフィス側の設計・改修 |

### 4.1 回す時(フロント→バックエンド)
**回す判断**: 見た目を直すのに**データの形・APIの返り値・保存キーの設計**を変える必要が出たら、そこで止めて回す。
自分で押し通さない。回す時は4点セットを必ず添える:
1. **目的**(何を実現したいか) 2. **出力形式**(どんな形で返してほしいか)
3. **情報源**(どのファイル/画面の話か) 4. **境界**(どこまでやってほしいか・どこは触らないでほしいか)

### 4.2 受ける時(バックエンド→フロント)
バックエンドが機能を足したら、**その画面の見え方はこちらが決める**。
「とりあえず置いた」UIをそのまま残さない。§2の規約に通してから出す。

### 4.3 衝突を避ける(絶対)
- **着手前に必ず所有権黒板**: `python scripts/ownership.py check "<キーワード>"`
  - exit=2(誰かが着手中)なら**実装しない**=所有者と調整。
  - exit=0(空き)なら `python scripts/ownership.py claim <topic> --owner frontend --doc <path> --note <要約>`
    してから着手。完了で `release`。
- **同じファイルを2部門が同時に触らない**(1領域=1オーナー・RULES §3)。
  `index.html` / `style.css` は特に競合しやすい。**claimせずに触らない**。
- 大きい/横断の改修は着手前にmain箱(研究室HQ)へ相談。

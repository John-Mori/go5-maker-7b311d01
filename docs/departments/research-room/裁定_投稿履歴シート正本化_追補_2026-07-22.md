## 裁定A: 端末をまたぐ draft — **(b)を採る。ただし条件付き**

### 論点(HQ指摘)
設計書§2の表で **draft は localStorage のみ**。だがChamiの原要求は「端末に依存しない履歴」。
生成/X投稿をした端末と、YouTube投稿+「投稿完了」を押す端末が違うと、
後の端末に draft が無く**ボタンを押せずシートに1行も残らない**=現行より悪化する。

### 裁定
**(b) draft は端末ローカルのまま。UIで「この端末で完了させてください」と明示する。** 理由:

1. **不可逆を増やさない(C-003)。** (a)案(draft をシートの仮フェーズ用タブへ)は、
   confirmed へ昇格する時に**仮行の後始末**が要る。消せば不可逆、残せばゴミ——
   今回消そうとしている「ゴミデータ」を別の場所に作ることになる。
2. **移行中に書き込み経路を増やさない。** 本改修の本体は**書き込み経路を3つから1つへ減らす**ことだ。
   その最中に新しい書き込み先(仮タブ)を足すのは、減らす作業と増やす作業の同時進行になる。
   事故の切り分けができなくなる。
3. **測っていない事象に構造を足さない。** Chamiは 2026-07-21 に
   「パソコンから投稿したことないです。**スマホからしかない**」と明言している。
   端末をまたぐ draft は**一度も観測されていない**。★起きないとは言わない。**観測されていない**だけだ。

### ★この裁定に付く条件(無視するな)
- HQがChamiへ「実機運用が同一端末で完結しているか」を確認中。
  **Chamiが「端末をまたぐ」と答えたら、(b)のまま実装に入ってはならない。** (a)へ格上げして設計をやり直す。
- (b)の実装要件: **draft を黙って捨てない。** 画面から消える経路を作らないこと。
  UI文言は「この端末で完了させてください」の趣旨を必ず出す(沈黙は不可)。

## 裁定C ★差し戻し: `writeRecord_()` 先頭のバリデーションは **採用しない**

### 設計書§2の記述
```javascript
// gas/コード.gs: writeRecord_() の先頭に追加
if (!f.youtubeUrl) return jsonOut_({ ok: false, error: 'youtube_url_required' });
```

### これは**データ喪失を起こす**。実物を読んで確認した

`writeRecord_()` は**共有の書き込み口**で、呼び出しが**5箇所**ある(`gas/コード.gs` を grep):

| # | 呼び出し元 | youtubeUrl を送るか | バリデーション追加後 |
|---|---|---|---|
| 1 | `gas/コード.gs:452` doPost(ウィザード経路) | 送る | OK |
| 2 | `gas/コード.gs:647` `syncHistory_()` 一括同期 | **行による**(draft・旧行は空) | **拒否される** |
| 3 | `gas/コード.gs:1287` `runReservations` **無人予約** | **送っていない**(payloadに項目自体が無い) | **全件拒否** |
| 4 | `bluesky.js:1207` **アカウント矯正**(INC-112の修復機構) | **送っていない** | **拒否される** |
| 5 | `bluesky.js:2018` リビルド | **送っていない** | **拒否される** |

`bluesky.js:1083 recordToSheet()` の payload(1095〜1101行)には **`youtube_url` の項目自体が存在しない**。
4と5はここを通るので、例外なく拒否される。

### ★最も重い: #4 は**行が消える**

`bluesky.js:1207` 付近はこう書かれている:
```javascript
recordToSheet({ account: mv.to, ... });                       // 正チャンネルへ再upsert
if (it.postUri) jsonpGet(gasUrl + '?action=delete&channel=' + mv.from + ...);  // 誤チャンネルの行を削除
```
**delete は再upsertの成否で gate されていない**(jsonpの撃ちっぱなし・コールバックで待っていない)。
よって「**再upsertが拒否される → 誤チャンネルの行は削除される**」=
**行が消えて、どこにも書き戻されない。不可逆。**
INC-112の修復機構そのものが、データを消す装置に変わる。

### さらに: 設計書§7の記述が事実と食い違う
> 「無人予約(Phase5): GAS側の `handleReserve_()` は別経路なので影響なし」

`runReservations` は `gas/コード.gs:1287` で **`writeRecord_()` を呼んでいる**。別経路ではない。
同じく「`sync_history` は変更しないので問題なし」も、`syncHistory_()` が `:647` で
`writeRecord_()` を呼ぶ以上、成り立たない。★設計書のこの2行は**訂正が要る**。

### 裁定
1. **`writeRecord_()` の中にバリデーションを置くことを禁じる。** 共有の書き込み口を塞ぐな。
2. 必須チェックは**ウィザード経路の入口だけ**に置く。
   `doPost` の分岐で「ウィザードからの確定記録」であることを識別できる印(例: `op` か専用フラグ)を
   クライアントから送り、**その経路に限って** youtube_url 必須とする。印の設計は改修αに任せる。
3. **#4 の delete を gate する**のは、この改修の範囲外だが**別件として起票しろ**。
   バリデーションを入れなくても「再upsertが失敗したら行が消える」構造は既に存在している。
   ★これは今日見つけた**既存の穴**だ。今すぐ壊れているとは言わない(現状は拒否条件が無いので発火しない)。
   だが**発火条件を1つ足すだけで不可逆事故になる**。起票して可視化しておくこと。

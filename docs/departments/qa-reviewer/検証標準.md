# 検証標準 (品質-QA部門・8条)

> 正本。全ての検証・判定はこの8条に従う。根拠=改善設計書_品質QA窓運用_2026-07-17.md §4 (Chami承認 2026-07-17)。
> 各条の初出根拠は2026-07-16〜17の実測 (括弧内)。

## 8条

1. **現物主義** — 報告・戻り値・ログの「OK」ではなく成果物そのものを見る。LastTaskResult=0 は「起動した」の証明であって「中身が入った」の証明ではない。(バックアップ検証: 宛先の228ファイル実見で初めて合格)
2. **配線確認** — 仕組みは「存在する」ではなく「呼ばれている」まで確認する。ガードを足した確認と、呼び出し側が引数を渡している確認は別物。(known_depts既定値frozenset(): 渡し忘れならsweep恒久no-op化、の未然検出)
3. **時刻連鎖** — 「動いた」は独立した記録の時刻を鎖にして示す。時計系が違う記録はオフセットを明記。(タスク01:58:41→ログ01:58:43→現物)
4. **対策殺し確認** — 防御を足す変更では、守られるべき機能がまだ生きているかを必ず試す。防御の追加は守るべきものの停止と紙一重。(sweepガード検証で「本物の脈なし箱が回収されるか」を本命に置いた)
5. **再現主義** — 不具合の主張には再現手順を付ける。再現できないものは「未再現」と明記して確度を下げる。
6. **非破壊とサンドボックス** — 本番状態を変えない。代替 (GO5_LOCAL_DIR等) があれば必ずそちらで再現する。本番に触らざるを得ない検証は事前にちゃみのchatペイン承認。共有常駐の再起動はしない。
7. **検証の署名** — 判定には「何を見たか」(ファイル:行・時刻・件数・msg_id) を必ず書く。**「検証済み」という言葉自体は証拠ではない**。報告もまた次の検証対象。(データ整理の自己申告・研究室の「やった≠できている」)
8. **条件は期限つき** — APPROVED WITH CONDITIONS の条件には期限と再検証日を付ける。範囲外で見つけた問題は「指摘 (non-blocking)」として条件と区別して所有部門へ渡す。

## 判定の様式

```
判定 = APPROVED / APPROVED WITH CONDITIONS / REJECTED / ESCALATED
検証署名 = 見たもの一覧 (ファイル:行・時刻・件数・msg_id)
条件 (AWC時) = 内容 / 期限 / 再検証日
範囲外指摘 = non-blocking と明記して所有部門へ
```

## 検証依頼テンプレ (依頼側に渡す様式)

```
【検証依頼】
- 依頼元 / 期限・優先度 (P0〜P3):
- 対象: 成果物の場所 (ファイル・commit・タスク名・バージョン)
- 主張: 「何ができている」と言うか (箇条書き・1項目1主張)
- 証拠の所在: ログ・設計書・実行記録のパス
- 触ってよい範囲: read-onlyのみ / サンドボックス可 / 本番可 (可の場合は承認の所在)
- 差し戻し先: 不合格時に誰へ返すか
```

- 複数依頼が重なった時の優先順位はドンナがP0〜P3で裁く
- QA自身の誤判定も振り返り対象 (判定が後に覆ったらSTATUSへ記録し、害があればインシデント.mdへ)

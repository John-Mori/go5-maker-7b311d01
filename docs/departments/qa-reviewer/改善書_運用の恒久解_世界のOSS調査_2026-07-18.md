## 0. 結論 (先に要点)

**わたしたちは、分散システムの教科書を1章ずつ事故で再発見している。**
INC-85/86/93/94/98/103は全て、成熟したOSSが何年も前に構造で解決済みの古典的な故障パターンである。恒久解は「規約を増やすこと」ではない——規約は人もAIも踏む (提案者本人が5分後に踏んだ実測が2件ある)。**故障が起こり得ない部品に土台を替えること**である。

推奨は3段階 (詳細§4-6):
1. **土台の置換 (最優先)**: ポーリング→push受信 / JSONLファイル箱→トランザクショナルキュー / ログオン依存の常駐→OSサービス化
2. **労働力の置換**: 対話窓の常設→ヘッドレスの部門ワーカー (メッセージが来た時だけ起きて課金される)
3. **任意の強化**: 編成フレームワーク・記憶永続化・トレース可視化 (必要になってから)

## 2. 恒久解の要件 (何が「恒久」の条件か)

1. **push受信**: Discord発言が1秒以内にプロセスへ届く (ポーリング周期という概念を消す)
2. **消えないキュー**: 取り出し=リース (期限つき占有)・完了=ack・リース切れ=自動再配布。プロセスがいつ死んでもメッセージは失われず、二重処理もしない。**これが実装されれば Q2/Q3 の全インシデント類型が構造的に不可能になる**
3. **ログオン不要の常駐**: 再起動後、ログオン前に自動復帰。クラッシュは自動再起動+記録
4. **部屋=ワーカー**: 25部屋それぞれに「メッセージが来た時だけ起きる」ヘッドレスの応対者。アイドル費用ゼロ・人格と文脈は毎回正しくロード
5. **ルーターの役割回復**: 研究室は裁定と横断だけ。定常応対はワーカーが吸収
6. **人間承認の非同期化**: 承認待ちがワーカーを止めない (キューに積んで別件を処理)
7. **測って判定**: 全hopのレイテンシが数字で見える (体感論争の根絶・P6の一般化)

## 3. 世界のOSSでの標準解 (Sonnet 4班の実地調査・2026-07-18時点・全て現物確認済み)

### 3.1 受信の即時化 (Q1)
- **discord.py** (Rapptz/discord.py・16.1k★・v2.7.1=2026-03・活発): Gateway WebSocketで`on_message`が**push受信** (15秒周期という概念が消える)。しかも**Webhookの別名義送信 (username/avatar上書き)・リアクション付与・添付取得が全部標準API** — つまり現在の鳩+persona_send+react.py+添付退避の4機能が1ライブラリに揃っている。純PythonでWindows可。
- 代替: **Anthropic公式Discordプラグイン** (anthropics/claude-plugins-official内): Claude CodeをDiscord bot化し reply/react/edit/添付DLで「着信→起床→応対」を**公式サポート**。わたしたちのinbox_waiter路線の公式実装が存在した。

### 3.2 消えないキュー (Q2/Q3)
- 結論: 「クレーム→リース→ack・リース切れ自動再配布」をフル装備した既製SQLite品は**世界に存在しない** (litequeue 228★は半分まで・persist-queueはFIFOのみ)。選択肢は2つ:
  - **(a) SQLite自作** (数百行・WALモード+`UPDATE…WHERE lease期限切れ RETURNING`が定石・litequeueのソースが良い実例): 依存ゼロ・バックアップは既存Drive機構でファイル1個・学習資産になる
  - **(b) NATS JetStream** (nats-io/nats-server・20.3k★・CNCF・**Windows単体exe 18MB**): AckWait以内に未ackなら自動再配信・MaxDeliverで上限 — **Q3「生きてるが忙しい」が"リース中=処理中"として仕様レベルで表現される**。ただし常駐+運用+学習が1つ増える
- どちらでも、INC-85/86/94/103の**故障類型そのものが構造的に不可能**になる。

### 3.3 ログオン不要の常駐 (Q4)
- **NSSM** (nssm.cc): 10年更新なしだが枯れて定番。Windowsサービス化=ログオン前自動起動+クラッシュ自動再起動+ログ管理。代替: WinSW (停滞)・**Servy** (2026新興・ヘルスチェック統合・実績浅)。

### 3.4 部門ワーカー (Q5/Q6/Q7)
- **anthropics/claude-agent-sdk-python** (公式・7.7k★・**ほぼ毎日更新**・Windowsネイティブ): `ClaudeSDKClient`+hooks+in-process MCP+**セッションresume** — 「キューから取る→部門の文脈と人格を積んで処理→ackして寝る」ワーカーの正統な土台。**Pro/Maxサブスク枠で動く** (別課金化は一時停止中・要継続確認)。
- 編成系の却下判定: **claude-flow** (64.9k★) は効果の数字が自社発表のみ=ハイプ・不採用 / **claude-squad・agent-farm**はtmux前提=Windows不可 / CCManagerのみ要実地検証。

### 3.5 耐久実行・承認の非同期化 (Q8/Q9)
- **LangGraph** (langchain-ai・37.5k★・活発): SQLite checkpointer 1ファイルで状態永続化・**`interrupt()`で人間承認待ちでも状態が消えない** (Q9の恒久解)。Docker不要・pipのみ。
- 軽量タスク耐久: **huey** (SQLiteバックエンド・依存ゼロ) が最軽量。Temporalは単機に重すぎ・HatchetはDocker越し=過剰。
- **AutoGenはメンテナンスモード入り** (後継Microsoft Agent Frameworkへ過渡期) — 今は様子見。

### 3.6 人格・記憶・観測 (Q7/Q10)
- **CrewAI** (54.2k★・活発): Agent=role+goal+backstory+tools — **部門+人格manifestと1:1対応する設計**。枠組みごと採用せずとも、この構造をワーカーの人格ロード様式として真似る価値がある。
- **Letta** (旧MemGPT): Postgres必須=単機に過剰・見送り。**Langfuse**: 6サービス構成=重すぎ・トレースが欲しくなったら**Arize Phoenix** (単一プロセス) が軽量代替。当面はP6のsla_reportで足りる。

### 3.7 丸ごと置換の検討 (統合ゲートウェイ)
- **OpenClaw** (383k★・2026年最も伸びたOSS): Discord多部屋・**agent単位のwebhook別名義**・常駐・cron・Claude接続 — **機能要件はほぼ完全一致の唯一の本命**。しかし**セキュリティが壊滅的**: 2026年だけでCVE 138件超 (2〜4月は平均15時間に1件)・認証なし管理者奪取CVSS9.9・スキル市場に悪意あるスキル1400件超・**Microsoftが2026-02-19に「標準の個人PCで動かすべきでない」と公式警告**。全人格と全データが1台に集中するわたしたちの構成では許容不能 → **非採用。ただし設計 (agent単位のwebhook identity構造) は自作の参考に盗む**。
- 新興代替: **QwenPaw** (20.4k★・Windows AppContainerサンドボックス=OpenClawが炎上した問題への設計的回答) / **AstrBot** (Persona Settings=部屋別人格切替) — 部品取りの参考。**n8n**はルーティングの骨格のみで核心 (別名義・多セッション) は結局自作 → 丸ごと置換にならない。

## 4. 候補アーキテクチャ 3案

### 案A: 土台の置換 (最小・恒久解の核)
```
Discord ⇔ [gatewayボット(discord.py・NSSMサービス)] ⇔ [キュー(SQLite lease方式)] ⇔ 既存のClaude窓/研究室
```
- gatewayボット1本が: push受信 (1秒) → キューへ投入 → 送信印 → 添付local保存 → 受領スタンプ → 応答のWebhook別名義送信、を全部担う。**鳩・waiter・sweep・persona_send・react・P1/P4の後継を1プロセスに統合**
- キューは「クレーム+リース+ack」。窓が死ねばリース切れで自動再配布=**INC-85/86/94族の絶滅**
- 常駐は全てNSSMサービス=**Windows Update再起動→ログオン前に自動復帰** (既知の弱点の根絶)
- 既存のClaude窓・規約・人格運用はそのまま。移行はstrangler方式 (新gatewayが旧JSONLにも書き、並走で検証してから切替)

### 案B: 労働力の置換 (Aの上に載せる本命)
```
[キュー] ⇔ [部門ワーカー×N (claude-agent-sdk-python・常駐だが処理時のみ課金)] + [研究室=裁定専任の対話窓]
```
- 各部屋のワーカーが: claim → 人格manifest+STATUS+直近文脈をロード (CrewAIのrole/goal/backstory様式) → 応対 → ack → 寝る。**アイドル費用ゼロ・25部屋全部に「本人」が常駐するのと等価**
- 研究室は裁定と横断だけに戻る (INC-84の最終形・ちゃみの「研究室は総括する場所」の実装)
- ames型事故の消滅: ワーカーは毎回正しい人格を構造的にロードする (代打という概念自体が消える)
- 承認が要る作業はLangGraphの`interrupt()`様式でキューに戻す=**承認待ちがワーカーを止めない** (Q9)

### 案C: 既製ゲートウェイへの乗り換え — **非採用**
- OpenClawが機能的に唯一の完全一致だが、セキュリティ状況 (§3.7) により個人PC1台構成では許容不能。QwenPaw/AstrBotは部品取りの参考に留める。**自作継続が最も安全** — インシデント.mdという学習資産も活きる。

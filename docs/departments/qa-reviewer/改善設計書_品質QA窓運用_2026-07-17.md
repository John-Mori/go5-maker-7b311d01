## 3. 欠点と対策 (各: 事象 / 根本原因 / 対策 / 受け入れ条件)

### A-1. 報告様式 — 結論が埋まった
- **事象**: 初回の照合報告が長く、ちゃみから「すまん、結論よくわからん」(msg 1527327505437167626)。伝わらない報告は書かなかったのと同じ。
- **根本原因**: QA内部様式 (網羅・根拠列挙) のままスマホのDiscord読者へ出した。読者と様式の不一致。
- **対策**: Discord報告テンプレを固定する。
  1. **1〜2文目 = 結論・判定** (「〜でしたわ」で言い切る)
  2. **「ちゃみが決めること」があれば直後に番号付き選択肢** (無ければ書かない)
  3. 詳細・根拠は下段へ。本文はおよそ15行以内、超える詳細はファイルに書きパスを示す
- **受け入れ条件**: 以後の報告5件で、ちゃみからの意味確認の再質問がゼロ。

### A-2. 未検証の提案が「正しい顔」で流通した
- **事象**: 「mvで退避」を提案→ちゃみが実装OK→**提案者本人が5分後に退避先の罠 (local/inbox/内) を踏んだ** (INC-86続報)。
- **根本原因**: 一度も実行していない手順を、実行済みと同じ顔で送った。
- **対策**: 手順・対策の提案には**検証ラベルを必須**とする: **[実測済み] / [サンドボックス済み] / [机上]**。[机上]の案は「未検証」と明記して出す。
- **受け入れ条件**: ラベル無しの手順提案がゼロ。[机上]案を渡す時は必ず検証手段を併記。

### A-3. 取りこぼしを自分で検知できなかった
- **事象**: 強奪に2度とも「後から」気づいた (処理しようとしたら箱が空)。
- **根本原因**: waiterは鳴動時に `WAITER:MESSAGE total=N` を出しているのに、回収数との突合をしていなかった。**検知材料は既に手元にあった**。
- **対策**: 起床時の不変条件チェック: **waiterのtotal=N と退避できた行数を照合**。N>回収数なら即座に main箱・processed台帳をmsg_idで捜索し、結果を報告してから作業に入る。
- **受け入れ条件**: 不一致の検知から報告まで1ターン以内。「後から気づく」の再発ゼロ。

### A-4. 起動チェックリストの不備
- **事象**: 今セッションは HQの`PRIORITY.md`(存在確認済み・未読)・`local/requests.jsonl`(**38件中16件openを未読**)・インシデント.md関連カテゴリ を読まずに開窓した。CLAUDE.mdと規約が要求する手順の欠落。
- **根本原因**: qa-reviewerには**BOOT.mdが無い** (docs/departments/qa-reviewer/ 自体が今日まで不存在)。毎回その場のプロンプトで起動していた。
- **対策**: `docs/departments/qa-reviewer/BOOT.md` を新設。チェックリスト:
  1. cwd確認 (go5-maker直下でなければ停止して要請)
  2. session_label書き込み → waiter背景起動
  3. HQ `PRIORITY.md` → `local/requests.jsonl` のopen → 自部門STATUS.md → インシデント.md関連カテゴリ
  4. 受信箱: **mv退避(local/_work/) → 即再武装 → 既読印 → 処理 → 着手印**
  5. ちゃみのchatペイン発言はミラー規約に従う (※本日、名義送信が分類器に止まる事象あり→C-3参照)
  6. Python出力は `PYTHONIOENCODING=utf-8` を常用 (今日のcp932文字化けの再発防止)
- **受け入れ条件**: BOOT.mdが存在し、次回開窓の初回ターンで全項目に言及がある。

### A-5. 未クローズ案件の台帳が無い

### A-8. 規律4点 (小さいが実測で崩れたもの)
1. **口調の■退行**: 分析が重い報告ほど「■見出しレポート体」に寄った。これは記録済みの既知の退行パターン (メモリ: 人格の声はクロードへ退行する)。→ 結論と判定はドンナの声で言い切り、■の骨組みは要点列挙の最小限に留める。
2. **進捗印の徹底漏れ**: 3段化 (送信📮/既読✅/着手👀) の導入後、既読を一度も押していない。強奪でmain経由になったメッセージも、Discord上の実体は残っている=**印は押せる**。→ 起床手順 (A-4) に組み込み。
3. **アイドル起床の沈黙**: TTL満了時は再武装のみ・報告ゼロ (今日は維持できた。明文化して継続)。
4. **委譲ゼロ**: 今日の探索・照合を全て自窓でやった。文脈をきれいに保つため、大量grep・横断探索はサブエージェント (model明示: haiku/sonnet) へ寄せ、判断だけ窓に残す (CLAUDE.md §5準拠)。
- **受け入れ条件**: 処理済み全msg_idに✅が付く / アイドル起床の出力ゼロ行 / 3ファイル超の探索は委譲。

## 4. 検証標準 (8条・正文ドラフト)

> 承認後 `docs/departments/qa-reviewer/検証標準.md` へ切り出す。全て今日の実測に根拠を持つ。

1. **現物主義**: 報告・戻り値・ログの「OK」ではなく成果物そのものを見る。LastTaskResult=0 は「起動した」の証明であって「中身が入った」の証明ではない (バックアップ検証: 宛先の228ファイルを実見して初めて合格にした)。
2. **配線確認**: 仕組みは「存在する」ではなく「呼ばれている」まで確認する。ガードを足したことの確認と、呼び出し側が引数を渡していることの確認は別物 (`known_depts` 既定値frozenset()——渡し忘れならsweep恒久no-op化、の未然検出)。
3. **時刻連鎖**: 「動いた」は独立した記録の時刻を鎖にして示す (タスク01:58:41→ログ01:58:43→現物)。時計系が違う記録はオフセットを明記する。
4. **対策殺し確認**: 防御を足す変更では、**守られるべき機能がまだ生きているか**を必ず試す (sweepガード検証で「本物の脈なし箱が回収されるか」を本命に置いた)。防御の追加は、守るべきものの停止と紙一重。
5. **再現主義**: 不具合の主張には再現手順を付ける。再現できないものは「未再現」と明記して確度を下げる。
6. **非破壊とサンドボックス**: 本番状態を変えない。代替 (GO5_LOCAL_DIR等) があれば必ずそちらで再現する。本番に触らざるを得ない検証は、事前にちゃみのchatペイン承認を得る。共有常駐の再起動はしない (研究室の所有)。
7. **検証の署名**: 判定には「何を見たか」(ファイル:行・時刻・件数・msg_id) を必ず書く。**「検証済み」という言葉自体は証拠ではない**——報告もまた次の検証対象である (データ整理の自己申告検証・研究室の「やった≠できている」10件目、の両方が今日の根拠)。
8. **条件は期限つき**: APPROVED WITH CONDITIONS の条件には期限と再検証日を必ず付ける (今日のデータ整理AWCには期限を付け忘れた=本条の初出根拠)。範囲外で見つけた問題は「指摘 (non-blocking)」として条件と明確に区別して所有部門へ渡す。

判定語彙は組織共通の4値 (APPROVED / APPROVED WITH CONDITIONS / REJECTED / ESCALATED) を維持。

## 5. 検証依頼テンプレ (受け側の様式・B-3の実体)

```
【検証依頼】
- 依頼元 / 期限・優先度 (P0〜P3):
- 対象: 成果物の場所 (ファイル・commit・タスク名・バージョン)
- 主張: 「何ができている」と言うか (箇条書き・1項目1主張)

- 触ってよい範囲: read-onlyのみ / サンドボックス可 / 本番可 (可の場合は承認の所在)
- 差し戻し先: 不合格時に誰へ返すか
```

QAの返答様式: 判定 (4値) + 検証署名 (何を見たか) + 条件 (期限つき) + 範囲外指摘 (non-blocking)。

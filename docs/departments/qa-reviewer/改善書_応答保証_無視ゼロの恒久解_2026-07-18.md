- 起草: 品質-QA部門 / 状態: **Chami承認待ち (実装はgo待ち)**
- 依頼: Chami「Discordで送ったものがそのまま無視される問題を恒久的に解決する方法を探して (治っていないと思われる)」
- 調査体制: 前回同様 — 診断と執筆=Fable / GitHub実地調査=Sonnet並列2班 (現物確認済み)

## 0. 結論

**ちゃみの体感は正しい。治っていない。** 診断で現行犯を捕まえた: **1.2時間前のちゃみの「アメス！ローカルLLM教育部門に転送してくれ」が、誰にも読まれない箱で眠っていた** (発見後すぐ研究室へ回した)。

なぜ治らないか: これまでの対策 (P1受領スタンプ等) は**個別の穴を1つずつ塞ぐ**方式で、経路の一部しか覆っていない。今日見つかった穴 (for_claude箱) のように、**内部の仕組みで保証しようとする限り、新しい穴が生え続ける**。

恒久解は発想の転換: **内部状態を信じるのをやめ、Discord自体 (=最終出力) で「人間の発言にbot返信が付いたか」を監視し、付いていなければ自動で再投入・段階エスカレーションする**。どの内部部品がどう壊れても「無視」という症状そのものを検知して自己回復するため、**未知の穴にも効く**。これが「恒久的」の意味である。

## 2. 世界の裏付け (Sonnet 2班・GitHub/一次資料の実地確認)

1. **内部が動いていても返らない事故は世界でも起きている**: OpenClaw issue #4864 — 「handlerは132秒動いたとログに出るのにDiscordには何も返らない」。内部状態の監視だけでは無視を検知できない実例。
2. **答責の標準形は「外形監視」**: SREのsynthetic/blackbox monitoring — 内部でなく**最終出力そのもの**を観測する。2026年のAIボット監視も「応答が実際に完了したか」を追う潮流 (cekura.ai等)。
3. **inbox/outbox突合**: 受信の記録と送信の記録を同一DBで突き合わせ、`受信 LEFT JOIN 送信` で**未応答をSQL1本で機械抽出** (transactional outboxパターン・microservices.io)。
4. **escalation ladder**: GoAlert (Target製OSS)・IncidentRelay (SQLite対応) — 「N分沈黙→次の段階へ自動昇格→最後は人間へ」の状態機械。
5. **応対側の規律** (Discord LLMボット上位の実装から):
   - 重い処理をハンドラ内で待たない (受信即ACK・本処理は別タスク=OpenClaw #4864の教訓)
   - typing表示を8秒ごと再送する「心拍」(picoclaw #390) — 無音区間を作らない
   - **例外は握りつぶさず必ず「失敗した」と返信** (discord.pyの既定はstderrに吐くだけ=無視と区別がつかない)
   - 処理中の追加入力は破棄せず必ずキューへ (OpenClawのCommand Queue)
6. **SQS由来でうちのキューに足りない概念**: maxReceiveCount超過の自動DLQは実装済み・**DLQからの再投入経路 (redrive) とDLQ発生時の警報が未実装**。

## 3. 恒久解の設計 (4層・実装はgo待ち)

### L0: 書き手側の修正 — 今回の直接原因の根治 (研究室指摘・実装go時の最優先)
今回の現行犯には**既知の裁定違反という直接原因**があった: 「受付AIのエスカレ先はfor_claude箱でなくmain箱へ」は**2026-07-17に裁定済み** (memory: escalation-to-main-box)。qwen受付 (local_responder) がまだ旧経路へ書いているのは**裁定の未反映バグ**である。
- 修正: qwen受付の回送先をmain箱 (discord_inbox.jsonl) へ変更 + **for_claude箱の完全退役** (読み手を増やすのでなく、書き手を直す)
- 効果: main箱ならwaiterのチャイムが鳴る=今回のような無音滞留が構造的に起きない
- 教訓: **裁定は「決めた」だけでは効かない。全ての書き手コードに反映されたかを機械で照合するまで終わらない** (grepで旧経路への書き込み残存を検査するチェックをQA回帰に追加する)

### L1: 答責ウォッチドッグ (unanswered watchdog) — 恒久解の本体
**Discord APIだけを見る独立常駐** (内部の箱・脈・キューに一切依存しない=どの部品が死んでも動く):
```
毎2分: 全29chの直近履歴を取得 (既存sla_reportと同じ読み取り専用API)
  → 人間の発言で、M分(案:3分)経ってもbot返信が無いものを抽出
  → ①その部屋へMk.II「未応答を検知。再処理する」(重複防止台帳つき)
  → ②リースキューへ再enqueue (msg_id冪等=既に処理中なら無視される・安全)
  → ③15分たっても未応答なら研究室へSendMessage/incident chへ警報
  → ④30分でちゃみへメンション通知 (最後の砦)
```
- **なぜ恒久か**: 症状 (無視) そのものを検知するので、for_claude箱のような**まだ知らない穴が将来生えても検知・回復する**。個別の穴塞ぎ競争が終わる。
- NSSMサービス化 (再起動・ログオン前も生存) とセットで導入。

### L2: 経路の一本化 (案A段階2・保留中のものがこの問題の根治でもある)
全レーン (dept/llm/gemini/for_claude) をリースキューへ集約すれば「読む者が居ない箱」という概念自体が消える。**for_claude箱の構造穴は段階2で自然消滅する** — 段階2再開時のRelease Gateに「for_claude廃止」を追加する。
それまでの応急 (小): for_claude箱をwatchdogの滞留監視対象に追加 (現在は完全な盲点)。

### L3: 応対側の規律 (案Bワーカーの設計要件として固定)
§2-5の4点 (ハンドラで待たない/typing心拍/例外は必ず返信/追加入力はキューへ) + DLQ発生時のDiscord警報とredrive手順。ワーカー実装時のQA受け入れ条件にする。

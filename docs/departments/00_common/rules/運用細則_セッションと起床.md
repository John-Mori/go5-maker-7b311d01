## セッション分離マップ (2026-07-15 Chami方針=文脈混在でトークンを無駄にしない・1セッション=関連ch群)
> Discordのchは細かく分かれていても、**セッションは役割単位でまとめる**(1セッションが複数の関連chを担当してよい)。起動=各BOOT.md。
| セッション | 担当ch(dept) | 人格 |
|---|---|---|
| **研究室(main)**=常時open・キャッチオール | 総合-受付(router)・**研究室-コーチングルーム(research-room)**・横断裁定・**部門窓が開いていない全部屋の代打** | アロンソ/アメス(+その部屋のキャラで応対) |
| 改修(system-engineer) | 改修-依頼(system-engineer) | デブライネ/咲季 |
| AI office改修(ai-office) | システム改修設計-ai-office(ai-office)=**改修とは別セッション**(Chami指定) | デブライネ/咲季 |
| 人事(hr) | 人事-補強(hr-room)+人事-コンテキスト(hr-context)=**2ch1セッション** | ククール/琴葉 |
| 学習(learning-coach) | 学習3部屋(learning-coach)=**3ch1セッション**(質問ルームは複数あるが運用1つ) | 4コーチ |
| 改善提案(kaizen-analyst) | 改善提案部門(kaizen-analyst) | **アスナ(専任・2026-07-15着任)**/アメス補佐 |

| その他部門(product-scout/copy-director/shorts-analyst/qa-reviewer/report-notify) | 各ch | 各人格・必要時にスポーン |
- 運用: 新窓でgo5-makerを開き「あなたは○○部門、docs/departments/<dept>/BOOT.mdに従って」。複数dept担当時は各deptで `inbox_waiter.py --name <dept>` を打てば各箱を受ける(新着で即起床)。
- 未起動部門宛ての新着は研究室のmain箱へ集約(取りこぼしなし)。
- **★体制の既定 (2026-07-16 Chami承認 → 同日 Chami指摘で下記のとおり補正)**: **研究室(main)だけが常時open**。部門セッションは**重い作業がある時だけ臨時open**。部門窓が閉じている間、その部屋への発言は研究室が代打として**その部屋のキャラで**応対する。
- **★★在宅時は部門窓を開く (2026-07-16 Chami指摘・実測で裏付け)**: 上の既定のまま運用したら、**1日で研究室が43件を1人で処理し、うち29件(67%)が他部門の代打**になった(部門窓が0個だったため)。Chamiの指摘=「**全部門のチャットを研究室が読んでいる。分けた意味がない。負担が集中し、トークンも並列性も損なう**」は正しい。判断基準を明確化する:
  | 状況 | 体制 | 理由 |
  |---|---|---|
  | **Chamiが在宅で作業中** | **部門窓を開く**(改修/人事/学習など動いている部門) | 並列で速い・安い(各窓は自部門の文脈だけ持てばよい)。研究室は**裁定と横断だけ**に専念 |
  | **Chamiが外出中・就寝中** | **研究室1枚が全部代打** | 窓を開けられない。遅くても**取りこぼしゼロ**が優先 |
  - 仕組みは既にある: **窓が開けば鳩がその部門箱(`local/inbox/<dept>.jsonl`)へ直接配り、main箱には来ない。窓が閉じれば自動でmainへ回帰**(取り残しもsweepで回収)。切り替えに実装は不要=**窓を開けるか否かだけ**。
  - **★窓の開け方の正解(2026-07-16確定)**: **既に居る部門セッションをセッション間メッセージ(研究室のSendMessage)で起こす**。引き継ぎ内容=溜まった案件・進行中案件の重複着手防止・直近の規約変更。起きたセッションは自分でwaiterを起動し、**Discordの自部門chで「窓を開けた」と一言報告**する(Chamiが動いた証拠を見られるように)。**廃案=`claude "<起動文>"`で新窓を立てる方式**(ウィンドウは立つが対話待ちで止まり、waiterまで進まない=2026-07-16実測で失敗)。1部門セッションが近縁の複数dept(改修α+β等)を兼任してよい(waiterを2本立てる)。
  - 研究室の役割(窓が開いている時): **横断裁定・部門間の調停・規約の更新・判断に迷う案件の引き取り**。部門の実務は部門に任せ、**詳細まで把握しにいかない**(Chami明示)。
- **★★★部門一次対応(dispatch-first)が最終形 (2026-07-16 Chami明示・INC-84)**: Chamiの目標=「**部門chに投げたら、まずその部門のセッションが動く。重大なことだけ研究室へ上がる**」。研究室がmain箱で部門宛てを拾った時の行動:
  1. **自分で答えない。その部門のセッションをSendMessageで起こして渡す**(溜まった件・文脈・規約変更を添えて)
  2. 研究室が直接答えてよいのは3つだけ: **①その部門のセッションが存在しない ②Chamiが外出/就寝中(起こしても確認ダイアログを押す人が居ない) ③横断・重大・裁定そのもの**
  1b. ★★**転送・応答の前に必ず処理済み台帳を引く(INC-87・2026-07-17)**: **`python scripts/discord/triage_inbox.py`** を実行し「未処理」だけを扱う。**「main箱に在る」は「未処理」を意味しない**——sweepが部門箱を回収すると**部門が既に処理した控えまでmainへ落ちてくる**ため。これを転送すると部門は二度手間になり、**最悪Chamiへ同じ返信が2回届く**(2026-07-16に改修α2件・データ整理7件・人事2件で実際に発生。processed台帳に同一msg_idが2件記録される形で残っている)。既に部門が処理済みの行は`--drop`でmain箱から除去してよい(processedへの再追記はしない=二重記録を作らない)。
  3. 部門セッションは起きたら **waiterを常駐化(切れたら再武装)** し、自部門chで開窓を一言報告する
  3b. **★起床の正順(INC-85 → INC-86で退避先を訂正 → 2026-07-17に既読を追加・全部門必須)**: 起きたら **①mvで自箱を先に退避(`local/inbox/<dept>.jsonl` → `local/_work/<dept>.jsonl`) → ②即waiterを再武装 → ③読んだら"既読"を押す → ④workファイルを処理(本格的な作業を始める時に"着手")→processedへ**。
  3d. ★★**ドレインしたら記帳するまでが退避(INC-103・2026-07-18)**: main箱・部門箱を空けたら、**同じ手で処理済み台帳(`local/discord_processed.jsonl`)へ記帳する**まで手を止めない。箱から消えた=処理された、ではない——深夜にmain箱をドレインした誰かの手の中で、Chamiのアイコン差分2便(5枚)が処理も記帳もされず3.5時間消えた実例。恒久対策は台帳ステートマシン化(α発注3)。
  3c. ★★**再武装はチャイム起床に限らない(INC-98・2026-07-17)**: 長作業中のセッションはワークフロー通知・部門便など**チャイム以外の理由で何度もターンが始まる**。そのたびに自分のwaiterの生存を確認し、死んでいれば張り直す(=**どんな理由で起きたターンでも、終える前に耳が生きていることを確かめる**)。実例: 研究室が設計書作業に没頭した朝、main waiterがTTL全滅したまま9時間誰も張り直さず、Chamiの「大至急」(Discord招待の取り消し)が3時間無応答→Chamiの結論が「この場面でClaude Codeは用途が違ったな」になった。**箱への配達は20秒で成功していた——死んでいたのは耳だけ**。恒久対策は運用設計書§2-2(presence併用)。waiterの起動は**ハーネス管理の背景実行で行う**(シェルの&や別窓spawnは終了通知が届かず耳が死ぬ=改修αの実測と同穴)。
  - ★**進捗印は3段(Chami指示2026-07-16「既読の意味がおかしい」→(A)3段化を選択)**: **送信(sendms・📮)=鳩が配達時に自動**(届いた証明。Claudeが見たとは限らない) / **既読(kidoku・✅)=セッションが起床して読んだ直後に自分で押す** / **着手(chakusyu・👀)=本格的な作業を始める時に押す**。これでChamiの画面から**未達/届いたが無人/読んだ/着手済み**の4状態が判別できる。
    - 旧実装は鳩が配達時に「既読」を押していた=**"届いた"であって"読んだ"ではなかった**(改修βが調査・Chamiの指摘は正確だった)。2段案(既読をClaudeへ移すだけ)は「届いたが無人」の判別が消えるため却下(鳩の故障とセッション不在が同じ見た目になる)。
    - コマンド: `python scripts/discord/react.py --channel <ch名かID> --msg <msg_id> --emoji 既読` / `--emoji 着手`
  - ★**退避先は必ず `local/_work/`(local/inbox/ の外)**。`local/inbox/`の中へ退避すると、sweepが`*.jsonl`を全走査しファイル名=dept名と解釈するため**退避ファイルごと食われて中身が黙って消える**(QA/data-orgが独立に実測=INC-86)。sweep側にも台帳のdept以外は触らないガードを入れたが、規約でも退避先を名指しする。
  - 併せて**sweepのTTLを90秒→600秒へ延長**(INC-86): waiterの脈は新着を配達した瞬間に止まるため、90秒では正常に働いている窓を不在と誤判定していた。誤判定の害(取りこぼし・二重対応・研究室の負荷)>検知遅延の害(箱に残るだけ)。waiterはTTLで必ず死ぬので偽生存にはならない。旧順(読む→処理→最後に再武装)は、**waiterの脈が新着到達と同時に止まる**仕様のため、90秒を超える処理中にsweepへ箱ごと奪われる(QA実測2026-07-17・研究室の代打67%の主因)。sweepは空箱を触らない=mv先行で構造的に解決する(INC-76のmv先行ルールの部門箱適用)
  4. セッションが存在しない部門は、Chamiが新しい窓を1回開いて起動文(コードブロック単独メッセージで提供)を貼れば誕生する。以後は研究室が起こせる

## 部門専用セッションの開設手順 (コンテキスト逼迫時・専門作業が続く時)
1. 新セッションで宣言「○○部門として作業する」
2. 読む順: CLAUDE.md → 本ファイル → 自部門の.claude/agents/○○.md → chami-principles.md → 自部門docs → dept_tasks(自部門・open)
3. 作業 → 結果をdept_tasks.result+dept_eventsへ記録 → 終了(次のどのセッションでも続きが読める)
- 主要4部門(system-engineer/product-scout/copy-director/shorts-analyst)が対象。qa-reviewer/kaizen-analyst/learning-coachはスポーン型

## 版スコープ規約 (全部門集知・Chami発令2026-07-14)
- 設計・依頼で **PC版(パソコン版)/スマホ版に限定** する時はChamiがその旨を明言する。
- **何も言われなければ「共通改修」**(PC版・スマホ版の両方に適用)と認識する。
- ただし改修内容に **PC版/スマホ版に固有の部分が含まれ、それが書き換わる可能性がある時は、研究室が事前に指摘する**(黙って片方の固有実装を壊さない)。実装前に受け入れ条件表へ「この変更が触る版・固有部分の有無」を1行足すこと。

## 部門報告の消化規約 (INC-75の再発防止・2026-07-12)
1. **修正案の全項目消化**: 調査部門の報告にある「修正案・提案」は1項目ずつ「実施した/不採用(理由)」を実装時の報告に明記する。黙って落とさない
2. **仕様→受け入れ条件表**: Chamiが仕様を示したら箇条書きの受け入れ条件表に変換し、デプロイ前検証はその表を全行チェックする(「動いた」ではなく「全条件を満たした」)

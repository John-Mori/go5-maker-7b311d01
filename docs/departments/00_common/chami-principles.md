# Chami運用原則 (全部門必読・1ページ)

> 本原則は、注意力ではなく構造でミスを防いできた実務経験(Excel書類の手入力ミスを、VLOOKUP/IF化で手入力箇所自体を減らして解消した等)から導かれた設計要求である。全部門は新規タスク着手時に本ファイルを読むこと。

## 5原則と実装対応

| 原則 | 全部門への実装指示 |
|---|---|
| 1. 入力を減らす(二度入力させない) | 新機能は常に「URL1本→自動展開」「既定値」「引き継ぎ自動化」を第一候補にする |
| 2. 記憶に依存させない | 「前も似たことが…」で始めない。`インシデント.md`検索・decisions.md・D1ログを先に引く |
| 3. 注意力に依存させない | 人の目視確認に頼る手順を作らない。qa-reviewerのチェックリストと自動検証に落とす |
| 4. 同じ訂正を繰り返させない | improvement_requestsの反復を検出→Skill/ルール化を提案(承認制)。同じ指摘を3回受けたら仕組み化を検討 |
| 5. 壊れにくく戻せる | 変更→検証→履歴→ロールバック可能、を標準順に。切替弁(USE_D1式)・git revert・記録済みの変更履歴 |

## 適用例(このプロジェクトでの実績)
- 商品情報の手入力 → FANZA APIから自動取得 ／ 投稿結果の記憶管理 → Sheetsへ記録 ／ 毎回同じ説明 → ルール化・Skill化 ／ 部門への伝達 → dept_events/tasksで自動共有

## 原則の追加について
新しい原則はChamiの経験から導かれた一般化済みのもののみ本ファイルへ追記する(経験の詳細な記録はローカル側で管理し、本ファイルには載せない)。

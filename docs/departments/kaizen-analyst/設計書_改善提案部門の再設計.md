## 0. この部門のミッション(Chamiの言葉が正本)

> 2026-07-14: 「各部門が提案した内容をここで(判断するなり、わかりやすく私に伝えるなりして)提案してほしい。各部門が直接提案する形は一旦なし」
> 2026-07-15: 「ここは私が改善案を言うんじゃなくて、日々私がチャットしている内容や改善を促す内容や頻度であったり、あらゆる多角的な視点から、こんな実装や仕組み・システムにしたらどうか?とか、デザインやUIのアイデアとか、よりChamiが『それいいな!』となってくれそうなものを、あらゆる過去データやChamiの趣向から提案してほしい」

つまり機能は3つ:
1. **集約・翻訳**: 各部門の提案を溜め、判断し、Chamiに分かる言葉で提示する(部門→Chami直行の禁止の受け皿)
2. **能動提案**: Chamiの過去データ・趣向を分析し、Chamiがまだ言語化していない改善を先回りして出す(★本命)
3. **効果測定**: 承認→実装された改善が本当に効いたかを数字で追う

## 1. 現状診断(2026-07-17調査・数字つき)

### 1-2. 機能不全の真因(4つ)
1. **入口が紙の上にしかない**: 「部門提案はkaizen chへ集約」(orchestration.md 07-14)と決めたが、**運ぶ仕組みも習慣も作られなかった**。insight.created イベントの購読表にもkaizen-analystは居なかった。結果、提案は07-12を最後に途絶。
2. **能動提案の起動トリガーがない**: この部門の常駐はチャイム(受信箱の新着)待ち。しかし新着が来ない部屋で待っても仕事は永遠に始まらない(実測: 07-17未明、45分TTL×4回連続で新着ゼロ)。**「待つ」設計と「先回りして提案する」ミッションが矛盾**していた。
3. **提案のライフサイクル管理がない**: 提案番号なし・状態(提示済/承認/却下/実装済/効果測定済)の追跡なし・「Chamiが何を喜び何を却下したか」の記録なし=**趣向の学習が回らない**。
4. **一番豊かなデータ源が分析対象に入っていない**: agent定義のデータソースはD1の4表のみ。だが実際に一番濃いのは **Discordの全発言ログ(Chami発言575件・機微除外後)**・**git log(全修正とChami指摘の履歴)**・**インシデント.md(INC-100まで101件)**。Chamiの趣向はここにある。

### 1-3. 使えるデータ資産の棚卸し(能動提案の燃料)
| データ源 | 量 | 中身 |
|---|---|---|
| local/discord_processed.jsonl ほか | Chami発言575件(機微除外後) | 全部門へのChamiの生の言葉・指摘・要望・温度感 |
| git log(コミットメッセージ) | 14日で420件 | 何を直し、何をChamiに指摘されたかの時系列 |
| インシデント.md | INC-100まで101件 | 失敗の型=再発防止の需要 |
| D1 4表(REQ/CHG/insight/events) | 上表 | 構造化済みの要求・変更・操作 |

## 2. あるべき姿(再設計の骨格)

```
[入口A] 各部門の提案 ──(insight登録と同時にkaizen chへ自動ミラー)──┐
[入口B] Chamiの思いつき(このchへの発言) ─────────────────────┤
[入口C] 能動分析(週次改善便: Chami発言ログ+git+INC+D1を定期分析) ──┤
                                                              ▼
                                    [台帳] 提案台帳(KZ-NNN/PRO-NNN・状態管理)
                                                              ▼
                          [提示] 週次改善便として最大3件・各3行でDiscordへ
                                                              ▼
                     Chami判断(番号で返事) → 承認=改修部へ / 却下=理由を趣向DBへ
                                                              ▼
                                  [効果測定] 実装後2週間で前後比較を報告
```

## 3. 提案と実装状況(P-1〜P-7・全てChami承認済み)

### P-1: 週次改善便(能動提案サイクル) — ✅実装
- **何**: 週1回、アスナが「Chami発言ログ・git log・インシデント.md・D1」をPython前処理→多角分析し、提案を最大3件・各3行(効果/根拠/コスト)・番号付きでこのchへ出す。
- **起動**: TTL起床(45分ごと)時に `local/kaizen/last_weekly.txt` で「前回から7日経過」を判定→自走。新インフラ不要。Chamiが「改善便」と言えば臨時便。
- **実装物**: マーカー `local/kaizen/last_weekly.txt` + 前処理3スクリプト(下記P-5) + BOOT.mdの週次便セクション。

### P-2: 提案台帳とライフサイクル管理 — ✅実装
- **何**: 提案に **KZ-NNN**(能動)/**PRO-NNN**(集約)番号。状態語彙 `proposed → presented → approved / rejected / deferred → implemented → measured`。正本=D1 improvement_insights(status列)、`improvement-findings.md` は人が読む索引。
- **実装物**: findings.md を台帳表に再構成。D1 insert/update は研究室へ依頼(承認後)。

### P-3: 趣向プロファイルの育成 — ✅実装

- **実装物**: Chami発言575件の分析から初期エントリを起こした(承認スタイル=即決・許可疲れ・相関志向・貼らせない志向・表記/デザイン固定ルール)。

## 4. 運用様式

### 4-1. 提案の提示様式(週次便・臨時便共通)
```
📮 改善便 #N (YYYY-MM-DD)
KZ-012: (一言タイトル)
  効果: 何がどう良くなるか1行
  根拠: 観測データ1行(n付き)
  コスト: 小/中/大 + 金銭の有無 + 実装先(改修部/運用のみ/GAS等)
```
- 1便は最大3件(Chamiの判断コストを溢れさせない)。溢れた分は次便へ繰り越し、台帳では管理し続ける。実行後 last_weekly.txt を更新。

### 4-2. 台帳への記録(研究室へ依頼するSQL様式)
```
INSERT INTO improvement_insights(department,finding,evidence,confidence,suggested_action,status)
VALUES('kaizen-analyst','観測(n付き)','参照ID/ログ','low|med|high','提案本文','proposed');
```
- 状態遷移の更新も研究室依頼。md索引(improvement-findings.md)は自部門で即時更新。

### 4-3. 効果測定
- approved→implemented の提案は、実装から2週間後の便で前後比較(user_events/system_changes/該当指標)を1行報告。測れないものは「測れない」と正直に書く。

## X(Twitter)の文字数は「加重280」。日本語1字=2で数える (2026-07-21)
**結論**: X向けの文字数は必ず `BlueskyCore.xWeightedLength()` で数える。素の字数で数えない。
- 典拠: X公式 docs.x.com/fundamentals/counting-characters。Latin・記号=1／**CJK(日本語)と絵文字=2**／
  **URLは長さに関わらず一律23**(t.coに巻かれるため)。上限は文字数ではなく加重280。
- 実害: 1字=1で数えていた実装は「257字書ける」と表示していたが、日本語だけの実際の上限は**127字前後**。
  つまり**書ける量を約2倍に見せて**いた。「余裕あり」表示のままXに弾かれる。
- 現状の余白: フック行・CTA行入りの現行テンプレは実測**251/280**＝**残り約29**しかない。
  文言を足す提案(ハッシュタグ追加・定型文の追加等)は、この29を食い潰さないか必ず実測してから出すこと。
- テスト: `tests/test_bluesky.js` の X-1〜X-6。

## Xのアカウント固定値は core/account.js に一本化する (2026-07-21)
**結論**: Xのハンドル・表示名は `Go5Acct.X_ACCOUNTS` だけに置く。他ファイルに同じ表を作らない。
- acc1=夜半の耽読🌙コミック厳選 `@nox_raptus` / acc2=眠れぬ夜の書架めぐり `@nightowlbkshelf`。
  acc1/acc2 の並びは Bluesky・YouTube と同じ(月詠み=acc1 / 宵桜=acc2)。
- **台帳の正本は `local/consult_intel/🐧さん情報.md`**。core/account.js はコードから読むための写し。
  食い違ったら台帳が正。第2の台帳を作らない(ORG-11)。
- なぜ一本化するか: 実装中に bluesky.js 側にも同じ表が出来かけていた。同じ値の取得口が2つあると
  ハンドル変更時に片方だけ直って静かにズレる＝**INC-112(宵桜の履歴が消えた)の原因と同型**。
- **Xへは自動投稿しない**(Chami明示)。プレビュー+コピーで手動投稿。ハンドルは本文に差し込まず
  「どちらへ出す文面か」の表示に使う(自分宛メンションは無意味＋残り29の文字数を食う)。

## アカウント解決は必ず Go5Acct.current() を通す (2026-07-20)
**結論**: `localStorage('current_account')` 直読みも `window.getCurrentAccount()` 直呼びもしない。
- 実害(INC-112): 書き込み側(bluesky.js)と読み出し側(yt-clicks.js)が別々の取得口を使っていたため、
  ズレると「acc2へ書いてacc1から読む」状態になり**宵桜艶帖の履歴だけが消えて見えた**。
- **両方のフォールバックが acc1** なので、月詠み(acc1)では原理的に症状が出ない。
  → **症状が片方のアカウントでしか出ない時は、まず既定値と一致していないかを疑う。**

## 短縮ドメインの一覧は3箇所に複製されている (2026-07-20)
**結論**: 短縮ドメインを増減する時は**3箇所すべて**を直す。1つでも漏れると計測が静かに全滅する。
- `bluesky.js` の `SHORT.WORKER_HOSTS` / `affiliate-core.js` の `SHORT_URL_HOSTS` /
  **`gas/コード.gs` の `SHORT_WORKER_HOSTS`**。
- 実害(INC-112): 5mgl.com/yoz2.com へ切替えた際に**GASだけ取り残され**、GASが未知ホストを
  無言で捨てるためクリック列が永久に空→画面は「取得⚠️」。現在はGAS側に警告ログを入れてある。
- 予約投稿の待機状況は `<exec URL>?action=reservations_status` で読み取り専用に確認できる(2026-07-21追加)。

## 投稿本文の重複判定は「見出し行」で行う。URL一致では駄目 (2026-07-20)
**結論**: 自動付与ブロックの重複検知は `BlueskyCore.stripAutoBlocks()`(定型の見出し行で判定)を使う。
- 実害(INC-111): 「そのURLが本文にあるか」で判定していたため、本文の旧URL(da.gd)と新URL(5mgl.com)が
  別文字列となりガードをすり抜け、PR行・セール行が**2組ずつ投稿文に出た**。
- 併せて: 「このまま投稿されます」と名乗るプレビューは、実投稿と**同じ関数**で組むこと。
  別々に組むと後処理(短縮URL置換)の差でジワジワ乖離し、プレビュー自体が信用されなくなる。

## 一覧・キャンペーンURLは buildFanzaListLink で包む (2026-07-20)

一覧ラッパを使えばアフィ化できる。🔗アフィリンクタブは no_cid の時に自動でこちらへ回る。

## 作業手順で踏んだ罠(繰り返さない)

- **`?v=` は必ず `node scripts/bump.mjs`**。手 sed は取り残し・並行セッションとの衝突を生む(ORG-09)。
  ※このセッションで実際に手sedをやってしまった。他セッションの正規化に救われたが、以後はbump.mjs。
- **waiter はハーネスの run_in_background で起動する**。シェルの `&` は終了通知が来ず耳が死ぬ。
- **所有権黒板を必ず先に見る**(`python scripts/ownership.py check "<語>"`)。
  X投稿機能は既に別セッションがclaim+実装済みで、危うく二重実装するところだった。
  claimされていたら実装せず、**検証側に回ると価値が出る**(実際そこで文字数バグを見つけた)。
- **GAS を触る時は `gas/` 全体が push される**。他部門が作業中の `gas/競合.gs` を巻き込まないよう
  未コミット変更の有無を先に確認する。
- **readonly の入力欄にリスナーを足しても動かない**。「直した」と報告する前に、その操作が
  実際に可能かをUIで確かめること(X欄で実際にやってしまい、後で訂正した)。

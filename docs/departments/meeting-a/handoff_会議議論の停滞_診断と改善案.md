# 検討引き継ぎ: 会議室(α/β)の議論が途中で止まり全体進行が停止する問題

> 起票: 会議室α(meeting-a) / 2026-07-15 / Chami指示「会議の議論が途中で止まって全体の進行が止まるのを解決したい。研究室へ意見と改善を」
> 宛先: research-room(アロンソ=方針裁定・アメス=整理) → 判断後に system-engineer(実装)へ
> 分類: 横断(常駐アーキテクチャ・進行プロトコル・トークン効率)。実装は会議室では行わない。
> 関連: `docs/departments/shorts-analyst/handoff_イベント駆動ウェイク検討.md`(脈=heartbeatの根が同じ)・INC-091(有限TTL)・orchestration.md §セッション分離マップ

## 0. 結論(先に)
会議室の議論が止まるのは**やる気や内容の問題ではなく、構造の問題**。現行は「複数の部門セッションが同時に起きていて、投稿し合えば議論が進む」ことを暗黙の前提にしているが、**その前提が実データで成立していない**。
→ 提案の芯: **会議を「複数セッションのリアルタイム往復」から「1つの進行役セッションが部門人格をサブエージェントで招集し、有限ラウンドで回して結論+議事録+差し戻しまで一気に出す“ラウンドテーブル方式”」へ変える。** これで「相手が起きていない/終わらない/戻らない」を同時に潰す。

## 1. 実データで裏づけた真因(2026-07-15 21:30 JST 時点)

脈ファイル `local/llm/claude_active_<dept>.txt` の最終更新(=そのセッションが最後に生きていた時刻):

| セッション | 最終脈 | 新鮮(<90s)? |
|---|---|---|
| research-room | **ファイルが存在しない** | ✗(一度も脈なし) |
| meeting-a(会議室α) | **ファイルが存在しない** | ✗ |
| meeting-b(会議室β) | 21:00 | ✗(30分前・失効) |
| product-scout | 09:28 | ✗(12時間前) |
| copy-director | 09:29 | ✗ |

**停滞の因果(5つの故障モード):**
1. **相手が起きていない**: 「商品候補選定部門とタイトル文の部門と会議室で議論して」(Chami 12:10依頼)を実行しようにも、product-scout/copy-director のセッションは起きていない。会議室へ最初の1投稿をしても**応答する相手がいない**→即停止。
2. **進行役がいない**: 会議室(α/β)自体に、議論を前に進め・要約し・結論を迫る**常駐の司会がいない**。meeting-bも1回脈打っただけ。
3. **終端プロトコルが無い**: 「自由議論」に**ラウンド上限・収束条件・決定ルール**が無い。終わり方が定義されていない議論は漂流して死ぬ。
4. **差し戻し経路が無い**: 仮に良い結論が出ても、それを**必要としていた元の作業へ戻す道**が定義されていない→結論が宙に浮き、全体進行が止まる。
5. **正本に未登録**: 会議室(meeting-a/b)は orchestration.md の「セッション分離マップ」にも所有権モデルにも**載っていない**。だから「誰がこの議論を回すのか」の持ち主が居ない。

> 補足: これは shorts-analyst の「イベント駆動ウェイク検討」と同根。脈(heartbeat)が切れると相手が起きておらず、投稿しても応答が来ない。会議は「2つ以上の生きたセッションの同時性」に依存しており、その同時性が最も脆い。

## 2. 改善案: ラウンドテーブル方式(会議の単一セッション・有限ラウンド化)

**思想**: この組織は既に「1セッションが複数人格を演じ分ける」パターンを持っている(研究室=司令塔がアメス/アロンソを演じる)。**同じ型を会議に適用する。** 会議室セッションが議題を受け、関係部門のサブエージェント(product-scout/copy-director 等・モデル明示)を**その場に招集**し、有限ラウンドで意見を集め、司会が収束判定して決定を書く。**同時に複数の生きたセッションを必要としない**ので「誰も起きていない」で止まりようがない。

### 2.1 プロトコル(有限・必ず終端する=INC-091の精神を尊重)
```
convene(招集): 司会が [議題 / 成功条件 / 参加部門 / 最大ラウンド数(既定3) / 決定ルール(収束しなければ司会が裁定)] を宣言
loop(最大 N ラウンド):
    各参加部門サブエージェントが [立場 + 理由(1〜2点)] を返す  ← モデルは部門既定を明示
    司会が議事録へ追記し、収束判定:
        合意 or 2ラウンド連続で新情報ゼロ → break(収束/枯渇)
decide(裁定): 司会が [結論 + 根拠 + 少数意見] を確定
handoff(差し戻し): 議事録を会議室docsへ保存 + 決定を発注元へ返す(Discord+受信箱) + 学習用レコードを1件出す
```
- **必ず終わる**(最大ラウンドで打ち切り)ので、偽の無限ループ(INC-091)を作らない。
- **1セッション完結**なので相手の生存に依存しない=故障モード1・2を解消。
- **収束条件と決定ルール**を明示=故障モード3を解消。
- **handoffを工程に含める**=故障モード4を解消。

### 2.2 会議室αとβの役割分担(案・要Chami/アロンソ裁定)
- 現状βは1回だけ脈があった=何かの試行中と推測。**α=商品×コピー系の創作会議 / β=分析×QA系の検証会議**のように**テーマで分けるとセッション文脈が濁らない**(Chamiの「文脈混在でトークンを無駄にしない」方針と一致)。ただしこれは仮。**実際の使い分けはChamiの意図が正**(§4で確認)。

### 2.3 発注元への差し戻し(故障モード4の具体)
会議の決定は必ず次のいずれかで**発生元へ戻す**:
- 部門間イベント購読表(orchestration.md §購読表)に会議結果の型を1行追加(例 `meeting.concluded` → 発注元部門で dept_task 化)。
- Discordは決定を**発注元のch**へ、会議の顔で1投稿(§0.5 Discord往復の徹底)。

## 3. 「議論を止めない」ための最小運用ルール(コード不要・今日から)
実装(§2)を待たずに効く運用パッチ:
1. **会議は"招集→有限ラウンド→議事録→差し戻し"を1セッションで完結させる**(相手セッションの生存を当てにしない)。
2. **待機中は heartbeat を機械的に再武装しない**(仕事した時だけ脈)。idleの空脈は「10分ごとにClaudeを起こす」トークン垂れ流しの元(memory: idle-let-heartbeat-expire / shorts-analyst handoff §4と整合)。
3. **会議室を orchestration.md のセッション分離マップへ登録**(所有権の空白を埋める)。← これが無いから「誰も回さない」。

## 4. research-room / Chamiへの確認事項(判断が要る点)
1. **会議室の使い分け**(§2.2): α/βをテーマで分けるか、Chamiの想定は別か。
2. **決定ルール**: 部門が収束しない時、司会(会議室)が裁定してよいか。それとも研究室(アロンソ)へ上げるか。
3. **ラウンド上限**: 既定3で妥当か(コスト vs 議論の深さ)。
4. **§2の実装可否・工数**は system-engineer 見積り(購読表への型追加・handoff配線が主)。

## 5. 学習パイプラインとの接続(Chami 12:13「教育係の最大目標」に応える)
Chamiは「日々の全部門のやりとり・私の話・Claudeからの知識を学習して、ローカルで最強の“考えて実行するAI”にしたい」と明言(12:13)。
→ ラウンドテーブルの**議事録を構造化(議題/各部門の立場/決定/根拠/少数意見)**しておけば、そのまま**ローカルLLMの良質な教材**になる。「議論が途中で死ぬ」と教材も濁る=停滞解消と学習強化は**同じ打ち手**。議事録出力に `learning_record` を1件添える運用にする(local完結・[[local-llm-work-logging-learning]])。

## 3. キャラコンテキスト自動収集パイプライン「Context Packet v1」 ★本丸

### 3.1 設計思想
- 「自動」=無人ではなく、**「Chamiが原稿を書かなくても、人事が定型手順で正確に集め、Chamiは確認・修正するだけ」**にする。人格の最終決定権は常にChami(実例: ホイミンの一人称「ぼく」・アーモンドアイのさん付け例外は全部Chami裁定だった)。
- 正確さは**情報源の階層**と**2ソース一致**で作る。モデルの記憶に頼らせない(ハルシネーション遮断)。
- deep researchは**弾数管理**: 既定0発/キャラ。使用条件を§3.5に固定。通常のWebSearch/WebFetchは弾数制限の対象外なので、これを主砲にする。

### 3.2 情報源の4階層(信頼度ルール)

| 階層 | 情報源 | 採用条件 |
|---|---|---|
| **S0** | Chami原稿(`local/persona_context/*_source.md`方式) | 最高権威。そのまま採用(要約は非転記で) |
| **S1** | 公式一次情報: 公式サイトのキャラ紹介・ゲーム内プロフィール・公式X/出版社ページ | **単独で採用可** |
| **S2** | 大型二次資料: Wikipedia日本語版・ピクシブ百科事典・ニコニコ大百科・Fandom | **運営主体が異なる2ソースの一致**で採用。不一致→S3 |
| **S3** | deep research(弾数限定) | §3.5の条件時のみ |

- 独立の定義: 運営主体が異なること。Wikipediaのミラー(weblio等)はWikipediaと同一扱い=1ソース。ピクシブ百科×ニコニコ大百科は独立ペアとして可。公式+任意1つが最強。
- ウマ娘勢の注意: **2層に分けて集める**。①ゲームキャラとしての設定=Cygames公式が正 / ②モチーフの競走馬の史実=JRA・netkeibaが正。混ぜない(台帳の呼称原則「ウマ娘=ちゃみ呼び=ゲームキャラ扱い」と同じ切り分け)。

### 3.3 収集手順(1キャラの流れ)

```
①トリガ  Chamiが配属指示(hr-room) or 既存キャラ増強指示(hr-context)
②検索    WebSearchで公式ページ+S2を特定(2〜3クエリ)
③取得    WebFetchで3〜5ページ(公式優先)
④構造化  調査票(§3.4)に沿ってサブエージェント(sonnet)が要約
          ★グラウンディング命令を必ず付ける:
          「取得テキストに書いていないことは書かない。自分の知識で補完しない。
            全項目に出典番号を付ける。確認できない項目は『未確認』と書く」
⑤検証    事実項目(声優・初出・肩書・作中役割)は2ソース一致を突合。
          不一致・単一ソースは「未検証」ラベル(manifestには使わない)
⑥人事レビュー 著作権チェック(§3.6)+要約整形+呼称の提案化
⑦Chami確認  hr-contextへ「確認パケット」を投稿。
          一人称・口調・呼称は【提案+質問形式】で出す(決めるのはChami)
⑧記録    local/persona_context/<id>_source.md(出典つき要約)
          → sprites.jsonのtraits_note → INDEX.md行 → manifest化は現行フロー(研究室)へ
```

- コスト目安: 1キャラ=検索2〜3回+取得3〜5回+構造化1回(sonnet)。**deep research 1回の数十分の一**。
- 複数キャラ滞留時はサブエージェント並列で1バッチ化。
- 一度集めたら再収集しない(キャラ設定は変わらない)。更新はChami指示時の差分のみ。

### 3.4 調査票(取材項目=Packetの骨格)
テンプレ実体= [ContextPacket_テンプレ_v1.md](ContextPacket_テンプレ_v1.md)

| 区分 | 項目 | 検証ルール |
|---|---|---|
| 基本事実 | 原作/初出年/公式肩書/声優/公式プロフィール要点 | **2ソース一致 or S1単独**。数字・固有名詞はここに集約 |
| 性格の軸 | 性格3〜5軸/行動原理/背景の要約 | 2ソース一致は「事実」、単一ソースの性格描写は「解釈」ラベル |
| 口調 | 一人称/二人称/語尾パターン/敬語有無/テンション | パターンの記述+**代表セリフの引用可**(出典付き・2026-07-17 Chami裁定)。象徴句は原作引用でも創作でもよい |
| 関係性 | 主要キャラとの関係 | 組織内の呼称設計の材料(ハルさん問題の再発防止) |
| NG | やらないこと・言わないこと | Chami確認欄とセット |

### 3.5 deep researchを使ってよい条件(弾の温存)
1. S1が存在しない/発見できない(設定資料集にしか無い等)
2. S2同士が矛盾し、S1でも裁定できない
3. Chamiが明示的に深掘りを要求した
- それ以外は使わない。使ったら引き継ぎ書に「弾を使った理由」を1行記録する。

### 3.6 セリフ・原文の扱い(2026-07-17 Chami裁定で緩和)
- **セリフの引用=可**(Chami裁定「著作権とか言い過ぎ。閉じたここだけの世界の話」)。象徴句に原作の名セリフをそのまま使ってよい(前例=スネークの「待たせたな」・原作名言をChami指定で正式採用済)
- 引用には出典(作品名・できれば場面)を添える——法律の話ではなく**正確さのため**(誰のセリフか後から検証できるように)
- 設定資料の長文丸写しは引き続きしない。理由は著作権ではなく**トークンと可読性**(要約+効くセリフの引用、の組み合わせが一番使いやすい)
- 出典URL+取得日の記録は継続(検証可能性の担保)

### 3.7 初弾=黒川あかね(具体的な最初の運用)
台帳に「新キャラ・設定未定=**Chamiの設定待ち**」で滞留中(data-org配属済)。
→ **待つのではなく、人事がPacketドラフトを先に作ってChamiに出す**。Chamiはゼロから書く代わりに直すだけ。これがこのパイプラインの価値の実証になる。
※2026-07-18追記: Chamiが参照URLを別途提供済み=このキャラの自動収集は不要。manifest化はそのURLを元に進める(台帳更新済)。パイプラインの初弾実証は次の新キャラで行う。

## 4. 呼称・口調ルールの整合運用

### 4.1 呼称マトリクス(P1・済 2026-07-17)
INDEX.mdに「誰が→誰を→どう呼ぶ」の一覧表を新設し、散文で増え続けている呼称ルールを1表に吸収した。
新ルールが来たら**表に1行足す→各manifestは表を参照**。三重持ちを「正本1+参照」に変える。

### 4.2 整合チェッカー ★実装済(2026-07-17)
実体= `scripts/kaizen/check_persona_consistency.py`(読み取り専用・exit 1=要対応)。人事BOOTの起動時手順に組込済。

- **所有を「改修」→「人事」に変更した判断**: ①`scripts/kaizen/`は**Chami承認済の「Python積極利用方針」(許可確認不要)**の指定置き場で、用途に「検証/データ品質確認/重複検出/差分比較」が明記 ②同じ性質の点検スクリプト`check_incident_ids.py`が既にある(前例) ③**読み取り専用でアプリのコードに一切触れない**・点検対象は人事自身の台帳 ④中身は今日デブライネ統一で人事が手作業grepした内容そのもの。→ 「コードは改修へ」の趣旨(アプリを壊さない)から外れないと判断。改修が引き取るなら単独ファイルなのでそのまま移せる。
- 検査項目: ①旧表記の混入 ②禁止句 ③台帳キャラ⇔`persona_avatars.json`の突合 ④アバターURL失効(`refresh_avatars.py --check`へ委譲)
- **実装で判明した設計の甘さ(修正済)**:
  - 判定は**完全一致**でなければならない。`persona_send`は`json.get(persona)`で引くため部分一致では顔が出ない=甘い判定は顔なし投稿を見逃す。「表記ゆれ(近いキーはあるが完全一致が無い)」と「未登録」を分けて報告する。
  - **旧語を"語るために"書いている行は正しい**(Chami発言の引用ログ・ルール定義文・新旧併記)。ここを除外しないと、規約文書と本チェッカー自身が永久に赤くなる。
  - INDEX.mdには名簿表以外の表(呼称マトリクス/改修3部屋)もあるため、**名簿表だけ**を読む(全表を読むと「部屋」を人名と誤認した)。
- **初回実行の成果**: 旧表記・禁止句=0件、URL失効=0件、**アイコン未登録=8件を検出**(§5.1)。

### 4.3 canonical ID台帳(P2・改修へ)
表示名の文字列キーをやめ、`canonical_id`(例: kukuru, almond-eye)+`display_name`+`aliases[]`の台帳に。`persona_send`の解決順=完全一致→alias→警告。デブライネ4表記のような名前ゆれ事故の恒久対策。移行中は現行の別名キー方式(追加のみ・削除しない)を維持。

## 5. アバター(アイコン)信頼性

### 5.1 ★アイコン未登録8名(2026-07-17 チェッカー初回実行で発覚・Chamiへ画像を依頼中)
`persona_avatars.json`に**キーが存在しない**=この名前で投稿すると**顔なし**になる。台帳では「v1適用済」でもアイコンは別管理のため、manifestが揃っていても顔が無い状態が起きる。

| 未登録キャラ | 部門 |

| 早坂芽衣 | copy-director |

- 対応: Chamiに画像を依頼→`persona_avatars.json`へ登録(**URLと元msg_idを両方控える**=§5末尾)。
- 登録後は起動時チェッカーが自動で0件になる(=完了の判定が機械化されている)。

### 5.2 信頼性の運用
- **起動時点検(P0-2・本日BOOT反映済)**: 人事boot時に整合チェッカー(URL失効チェック込み)→失効>0なら `refresh_avatars.py --all`。全キャラ一斉失効(2026-07-16実発生・34件)を無人検知する。
- **恒久化はR2移行(改修)**: 研究室からは「アイコンはR2恒久化済(migrate_avatars_to_r2.py)」と連絡があったが、**実態は全URLがDiscord CDNのままで一斉失効していた**=移行が未完か、後から戻っている。**改修に実態確認を依頼する(P2-2)**。認識ズレのまま進むと「恒久化済みのはずなのに顔が消える」が再発する。
- **新画像の登録手順**: Chami添付→登録時に**URLと元msg_idを両方控える**。添付元メッセージが消えると再署名APIでも復活できない(デブライネの顔が再署名しても変わらなかった件の教訓)。

## 6. 手順書の運用ルール(BOOT陳腐化の再発防止)

- **BOOT.md=手順の正本**。手順の変更は必ずBOOTへ書く。**引き継ぎ書=状態と文脈のみ**(手順を書きたくなったらBOOTへ書いてリンクする)。
- セッション開始時にBOOTと引き継ぎ書の**矛盾を見つけたら作業前に直す**(heartbeat/waiter取り違えでトークンを浪費した実例より)。
- 横断手順(3段印・退避先等)が変わった連絡を受けたら、自部門BOOTへの反映を同ターンで行う(「あとで」にしない)。
- **成果物は作成即コミット(2026-07-18追記)**: 未コミットで共有ツリーに置くと他セッションの破壊操作で消える(本設計書自身が一度消失)。自部門領分はパス限定add→commitを1呼吸で。

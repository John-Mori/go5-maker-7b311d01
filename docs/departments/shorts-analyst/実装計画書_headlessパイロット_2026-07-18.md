# 実装計画書: headless都度起動パイロット(分析部門・2026-07-18)

> 承認: Chami 2026-07-18「最も目的を合理的かつ効率的、堅実に達成する方法を選んで実装へ行ってくれ。任せる」(msg_id=1527918083971481660)。
> 選定理由: 調査書_エージェントOSSと恒久改善_2026-07-18.md §2。常駐廃止×headless都度起動は空起床(PR-1)・偽生存(PR-3)・定期トリガー(PR-4)を構造的に消すが、トークン収支が未実測。よって**最閑散部門(shorts-analyst=実測2日で実依頼3件)で1週間パイロット→実測→全体展開判断**が最も合理的・堅実。
> 実装担当: **system-engineer**(コード所有権)。分析部門は本仕様書+BOOT_headless.md+実測分析を担当。設計疑義があれば研究室へescalate。

## 1. 構成(新規常駐ゼロ)

```
Discord着信 → inbox_poller(既存常駐・改修) → 部門箱へ書き込み
  → [headless対象部門なら] 箱を原子的に退避 → claude -p を都度起動(タイムアウト付き)
  → 処理完了(exit 0)/失敗(≠0 or timeout)なら退避ファイルを箱へ戻す(リトライ・上限3回→main箱+アラート)
定期分析 → Windowsタスクスケジューラ(週1・StartWhenAvailable) → 同じ起動ラッパー
```

## 2. system-engineerへの実装依頼(受け入れ条件つき)

### 2-1. inbox_poller.py改修
- 設定 `HEADLESS_DEPTS = ["shorts-analyst"]`(リスト定数 or local/設定ファイル。ロールバック=リストを空にするだけ)。
- headless対象部門は**脈ファイル判定を適用しない**(waiterが居ないため。常に部門箱へ配達+送信印)。sweepの回収対象からも除外。
- 配達後: `local/inbox/<dept>.jsonl` → `local/_work/headless/<dept>_<UTCts>.jsonl` へmv(原子的)→起動ラッパーをsubprocessで呼ぶ(poller自身はブロックしない)。

### 2-2. 起動ラッパー(新規・scripts/llm/headless_run.py 等)
- 多重起動ガード: 部門別lockファイル(PID+開始時刻)。実行中なら起動せずスキップ(退避ファイルは残る=次回配達時に合流処理)。
- `claude -p "<BOOT_headless.mdの内容+退避ファイルパス>" --output-format json` を **タイムアウト付き**(既定15分・強制kill)で実行。認証=CLAUDE_CODE_OAUTH_TOKEN(local/cli_auth_token.txt。revive_lab.ps1と同方式)。
- 終了処理: exit 0→退避ファイルは処理側が消している前提を検証(残っていればmain箱へ)。exit≠0/timeout→退避ファイル内の各レコードにattempt+1を付けて部門箱へ追記(**喪失より重複側**)。attempt>3→main箱へ移送+incident部屋へ通知。
- **実測ログ**: `local/logs/headless_runs.jsonl` へ1行=1起動 {ts, dept, msg_ids, exit, duration_s, usage(claude -pのjson出力から), trigger: message|weekly}。
- 無限待ち禁止との整合: 待つLLMプロセスが存在せず、実行時間はハードタイムアウトで有限=INC-091[2026-07-14]の思想を機構で満たす。

### 2-3. タスクスケジューラ登録(週次分析)
- `go5_analyst_weekly`: 週1回(金 20:00 JST・StartWhenAvailable=見逃し時ログオン後に1回)→起動ラッパーをtrigger=weeklyで呼ぶ(BOOT_headless.mdの週次節を実行)。

### 2-4. 受け入れ条件(QA検証)
1. テスト着信1件で: 配達→退避→claude -p起動→Discord返信(HTTP 204)→processed記帳→退避ファイル消滅、が無人で完走する。
2. 強制失敗試験(タイムアウト0.1分等)で: レコードがattempt+1付きで箱へ戻り、4回目にmain箱+通知へ行く。**喪失ゼロをmsg_id突合で確認。**
3. headless_runs.jsonlにusageトークンが記録される。
4. ロールバック試験: HEADLESS_DEPTSを空にすると従来のwaiter運用に即復帰する。

## 3. 切替手順(カットオーバー)

1. 実装完了→QA受け入れ条件1〜4をパス。
2. **その後に**現行の分析部門セッションがwaiter武装を停止(それまで現行運用を継続=二重応答防止。HEADLESS_DEPTS追加とwaiter停止は同じタイミングで)。
3. 1週間のパイロット実測へ。

## 4. 実測計画(分析部門担当・1週間)

- 指標: ①起動回数・②起動あたりトークン(usage実測)・③着信→返信レイテンシ(msg ts→persona_send時刻)・④喪失件数(processed台帳とチャンネル実メッセージのmsg_id突合・**0件必須**)・⑤品質(Chami体感+報告様式準拠)。
- ベースライン: 現行常駐の空起床 実測15〜30回/日(本セッション実績)+応答ターン。
- 判定: 喪失0件かつレイテンシ中央値≤3分なら「堅実性クリア」。トークンは実測値を全体展開判断の材料として報告(品質優先6段=トークン効率は5位。数値だけで自動判断しない)。
- 報告: 1週間後に分析部門chへ実測レポート→全体展開(他部門への拡大)の裁定をChamiへ。

## 5. 本計画の限界

- PC電源断・ログオフ中は従来同様に停止(全層が家PC稼働前提=既知の制約のまま)。
- パイロット中、分析部門の「会話の続き」文脈は失われる(都度起動のため)。文脈は台帳(hypotheses/STATUS/知見)が正=台帳を読む起動プロンプトで補う設計。
- Kanban D1移植(調査書§3-2)・MCP化(§3-3)等はパイロット結果を見てから順次(一斉にやらない=堅実優先)。

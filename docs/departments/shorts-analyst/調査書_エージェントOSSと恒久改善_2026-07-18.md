# 調査書: 世界のAIエージェントOSSに学ぶ「分析運用」の恒久改善 (2026-07-18)

> 発端: Chami依頼(2026-07-18 chatペイン)「現在の各分析運用に抱える問題を恒久的に解決できる方法を、世界中のGitHubから探して改善案をまとめて。実装はまだしない」。
> 方法: 7領域×並列サーチ(sonnet・トークン節約指示に従い検索は軽量モデル)→上位候補の実在・鮮度・ライセンス検証+辛口批評(Fable)。計8エージェント・候補41件・スポット検証7件。**本書は調査と提案のみ・実装ゼロ。**

## 0. 結論(30秒版)

1. **恒久解決の本命は、OSSの導入ではなくアーキテクチャの裁定だった。**「常駐セッション廃止×headless都度起動(`claude -p`+Windowsタスクスケジューラ+着信駆動)」に転換すると、空起床(PR-1)・セッション死/偽生存(PR-3)・定期トリガー不確実(PR-4)の3問題が**構造的に消滅**する(待つLLMプロセス自体が無くなる=無限待ち禁止とも本質整合)。
2. OSSは「丸ごと導入」より「**設計パターンの移植**」が費用対効果の主戦場。特にNousResearch/hermes-agentのKanban設計(MIT確認済)は、うちのD1 dept_tasks/dept_events「設計だけあって生産者不在」問題の実装解そのもの。
3. 即導入価値がある既製品は3つだけ: **cloudflare/mcp**(公式・D1/Workers読み取りのMCP化)、**basic-memory**(md台帳のMCP記憶化)、**promptfoo**(安全網の回帰テスト)。いずれも常駐追加なし・無課金。
4. 大型フレームワーク(LangGraph/CrewAI/AutoGen/claude-flow/Windmill/Langfuse自ホスト等)は全て**過剰装備**と判定。家PC1台・Discord固定・承認制の制約に合わない。

## 2. 最重要の発見: 裁定の順序依存

調査カタログ全体が「部門セッション常駐維持」を暗黙の前提に候補を選んでいたが、批評工程で判明したのは**その前提自体が最大の裁定事項**だということ。

**常駐廃止×headless都度起動 構成(調査で確認した公式サポートの組み合わせ)**:
- Discord着信 → 既存Pythonポーラーが `claude -p "<部門プロンプト+着信>"` を**その時だけ**起動 → 処理して終了(exit code+実行時間上限で生死判定が自明になる)
- 定期分析 → Windowsタスクスケジューラ(OS標準・OSS導入ゼロ)が同じheadless起動を叩く
- 長期文脈の喪失(トレードオフ) → 記憶基盤(§3-4)とhandoff/recapパターンで補う

| 効果 | 理由 |
|---|---|
| PR-1消滅 | 待機中のLLMプロセスが存在しない=空起床という概念が消える |
| PR-3縮退 | 「長命セッションのフリーズ検知」が「有限プロセスのexit code監視」に置き換わる |
| PR-4解決 | セッション開閉と無関係にOS標準cronが発火(見逃しは起動時キャッチアップ) |

★ただし**トークン収支は未実測**(都度起動のコンテキスト再構築コストが空起床削減を相殺しうる)。検証可能性>トークン効率の優先順位に従い、採用判断の前に1部門での実測が必須。
★現行の「会話の続きができる常駐窓」の使い勝手は失われる(Chamiの体感に直結)。全部門一斉でなく、**閑散部門から段階移行**が現実的。

## 3. 採用候補トップ5(検証済み・各初手つき)

### 3-1. 常駐廃止×headless都度起動(アーキテクチャ裁定・最優先)
- 効く: PR-1, PR-3, PR-4。§2のとおり。
- 初手(承認後): 最小の1部門でwaiter常駐を止め「ポーラーがclaude -pを都度起動」する試行設計書1枚(トークン収支の実測計画込み)を作る。担当=system-engineer+研究室。

### 3-2. Hermes KanbanパターンのD1移植 [ライセンス=MITと実地確認済]
- 出典: NousResearch/hermes-agent(216k star・2026-07-07 v0.18.2)のKanban機能。SQLite5テーブル(tasks/task_links/task_runs/task_comments/task_events)。
- 中身: ①タスク行の**原子的claim+TTL**(期限切れ/PID消失で自動的にreadyへ差し戻し) ②heartbeat断の二重監視 ③実行前クラッシュでも「ゼロ継続時間run」を合成し試行履歴を欠落させない ④人間の介入は非同期コメント/unblock(無限待ちを作らない)。
- 効く: PR-2(mv退避・sweepの人手規約→機構的at-least-once)、PR-3残余、PR-6(dept_tasksの実装解)、PR-8(append-only監査証跡)。**既存D1利用でインフラ増ゼロ。**§3-1と組むと「ディスパッチャがready行を見てclaude -pを起動」の完結構成になり、キューOSS(AMQ/huey)は不要になる。
- 初手: kanban.md+kanban_db.pyを精読し、D1版スキーマ+状態遷移表+ディスパッチャ仕様(冪等消費のdedup列込み)の設計書1枚(コードは書かない)。

### 3-3. MCP二本立て(PR-5) — 公式リモート+自作薄ラップ
- 調査で確定: GAS/Bluesky/D1/Sheetsを横断カバーする「決定版MCP」は世界に存在しない(個人メンテの小規模repoが乱立)。
- 方針: **cloudflare/mcp**(公式・Apache-2.0・647star・リモートMCP=OAuth1回で常駐ゼロ)でD1/Workersの読み取りを標準化。GAS・Blueskyは既存Python関数(JSONP剥がし等)を**FastMCP(公式Python SDK)の@mcp.toolで薄くラップ**(stdio=セッション起動時spawn・常駐増ゼロ)。
- 安全設計: 読み取り専用/書き込み系のツール分離。write系は既定オフ(投稿確認モーダル等の既存承認フローを温存)。D1側もOAuth権限を読み取り中心に絞る。
- 初手: 手作りcurl/wrangler/JSONP剥がし箇所の棚卸し表+読み書き権限一覧(write既定オフ)を作る。

### 3-4. basic-memory(md台帳のMCP記憶化)+軽量鮮度ルール(PR-7)
- basicmachines-co/basic-memory(3.5k star・v0.22.1・AGPL-3.0・Windows対応・SQLite既定でDocker不要)。**現行のmdファイル台帳をファイル形式を変えずに**MCP経由で読み書き・全文検索できる。
- 鮮度decayは組み込みでない→「N日未更新の仮説にwarning」を日次スキャン(§3-1のタスクスケジューラに相乗り)で自作。
- 注意: AGPLのため内部利用に留め、公開repoへ同梱・改変配布しない。
- 初手: 知見md1フォルダの読み取り専用取り込みPoC範囲+複数セッション同時書き込みルール案を作る。

### 3-5. 検証二層: promptfoo回帰+hooks自前トレース(PR-8)
- **promptfoo**(MIT・npxローカル完結・サーバ不要。2026-03のOpenAI買収後もライセンス継続を公式声明で確認済): 部門ルーティング判定・復帰プロンプトの効き目などをYAML回帰テスト化=「発火しない安全網」の実弾テストを自動化。
- トレースは**Claude Code hooks(PreToolUse/PostToolUse)のJSONL追記**でゼロインフラ。語彙だけOTel GenAI Semantic Conventionsに寄せ、将来の乗せ替え余地を残す。
- 初手: 回帰テストケース3件分のYAML試案+hooksで記録する項目一覧を作る。

## 4. 移植価値のある設計パターン(repo非依存・つまみ食い可)

- **Maildir原子性**(tmp書込→fsync→atomic rename): 受信箱に不完全ファイルが決して見えない。現行mv退避の機構化(実例=avivsinai/agent-message-queue。ただしwake機能はWSL必須のためWindows直採用は不可・パターン参照のみ)。
- **visibility timeout+claim/reclaim**: 取得したメッセージを一定時間他から不可視にし、ack無しなら自動復帰。PR-2とPR-3を同じ機構で解く。
- **transactional outbox**: 受信記録とキュー書き込みを同一トランザクションに。INC-100(ドレインの窓)の根治発想。
- **冪等消費**: msg_id upsertで重複実行を無害化=「喪失より重複側に倒す」を安全に採用する前提条件。
- **Dead Man's Switch**(Push型生存監視・Period+Grace Time): 現行脈ファイル運用の正式パターン名。既製(uptime-kuma)は偽生存を検知できないため導入は保留、概念のみ借用。
- **task-lifecycle語彙**(pending/ready/running/blocked/done/failed/canceled): dept_tasksスキーマの状態名として流用(A2A/Hermes共通)。
- **decay+強化スコアリング**(agentmemory実例・Apache-2.0): 台帳の「最終更新日+参照回数」メタデータだけで軽量模倣可能。
- **Discord Gateway+REST履歴補完**(批評が指摘した盲点): ポーラーが落ちても再起動時に「最終処理済みmessage ID以降」をREST APIで再取得すれば取りこぼしゼロ。**キューOSSを足す前に受信側で根治できる**現行資産の延長策。

## 5. 不採用・過剰と判定したもの(理由つき)

| 候補 | 判定理由 |
|---|---|
| Windmill / Cronicle / Inngest | cronのために新しいNode/Docker常駐を増やす構成。OS標準タスクスケジューラ+claude -pで常駐ゼロの同等物が得られる |
| LangGraph / CrewAI / AutoGen(AG2) / A2A | 実行モデルがClaude Code独立セッション構成と不整合。checkpoint+interrupt・human-in-the-loop等のパターンのみ借用 |
| ruflo(旧claude-flow) | 基盤丸ごと入れ替えに近く、自律swarm拡大の思想が承認制と逆方向 |
| mem0 / MemOS / Letta / graphiti | ベクトルDB・グラフDB・Postgres常設が事実上必須。PR-7はmd台帳の運用問題でありDB基盤の問題ではない |
| Langfuse(自ホスト) / OTel本格導入 | 4コンポーネント常設 or コレクタ自作が必要。hooksローカルJSONLで足りる規模に過剰。クラウド無料枠は機微データ外部送信の新論点を生む |
| uptime-kuma / WinSW | 単体では優秀だが偽生存を検知できず、§3-1採用なら監視対象自体が消える。アーキテクチャ裁定が先 |
| gas_mcp / 汎用curl系MCP / honker / litequeue | star一桁〜数百・experimental・現行完成品(deploy_gas.mjs)を上回らない |
| Claude Code Channels(Discordプラグイン) | research preview・配達保証の明記なし・既存ボットと二重化。将来性は要ウォッチ(公式のpush型受信) |
| Claude Code Agent Teams | 1リード従属・セッション再開でteammate消失=11部門常駐と根本不整合 |

## 6. 調査が自認する盲点(批評工程の指摘・正直に)

1. サーチ工程は常駐前提で候補を選んでおり、§2の裁定が先というメタ判断は批評工程で初めて出た(=カタログのfit判定は裁定次第で変わる)。
2. headless化のトークン収支は未実測(§2★)。
3. キュー/台帳をSQLite/D1へ移す場合、既存バックアップ体系(retention3日+月次)への組み込みが未検討。
4. Claude Code純正のセッション内cron(/loop・CronCreate)は「セッション起動中のみ発火」のため、常駐前提でしか使えない点に注意(調査初報の「fit A」は条件付きに訂正済)。

# 報告様式(shorts-analyst)

> 出典: 設計書_分析部門セッション改善_2026-07-18.md §3.7。適用=分析部門chへの全報告。

## 型(この順・20行以内)

1. **結論(2文以内・先頭)** — 質問に答える。数字の羅列で始めない。
2. **観測** — 数字と件数。**n必須**(nが小さい時は断定しない)。videoIdは**acc1-/acc2-接頭辞**でチャンネルを混ぜない。
3. **仮説** — 観測から言えること・言えないこと。
4. **次のアクション** — 誰の何の行動につながるか(Chamiの実決定=投稿時刻・作品選定・文言のどれに効くか)。

## 規律

- **確度ラベル**: 数字・事実には [実測] / [机上] を付す。実測は取得日時と経路(data-paths.md参照)を台帳側に残す。
- **出典・引用リストはDiscord表面に書かない**(2026-07-17規約)。根拠は hypotheses.md/知見.md へ記帳し、表面では「根拠は台帳SA-H001」とだけ言う。
- 送信は**短文でも `persona_send.py --body-file`**・HTTP 204確認まで「送った」と言わない。
- `--etitle`(embed)は3900字で黙って切れるため長文報告に使わない。
- 秘匿値(SHORT_SHARED_SECRET/ADMIN_SECRET/APIキー)をレポート・台帳・Discordに書き出さない。
- 喋る前にpersona_manifest確認。■箇条書きレポート体(素のClaude)へ退行しない。深い分析ほど要注意。
- 詳細が20行に収まらない時は「詳細は台帳へ・表面は要約」。

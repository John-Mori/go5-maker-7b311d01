# 検討引き継ぎ: 部門セッションの「イベント駆動ウェイク」化

> 起票: 分析-数字(shorts-analyst) / 2026-07-15 / Chami指示による調査・検討
> 宛先: system-engineer(実装) — ただしINC-091との設計トレードオフを含むため、判断が要ると見たら research-room へ escalate
> 分類: 横断(トークン効率・常駐アーキテクチャ)。実装は分析部門では行わない。

## 1. Chamiの要望(原文の要旨)
「何もなくても十分(=10分)ごとに起動する仕組みなら、ひたすらトークンの無駄。
**メッセージを受信したことを検知した時だけ起動する**仕組みを作れないか検討してほしい。
実装は分析部門ではなく system-engineer へ。難しければ research-room に共有して判断を仰いで。」

## 2. 現状の仕組み(調査結果)

| コンポーネント | 実体 | Claudeトークン消費 |
|---|---|---|
| `inbox_poller.py`(`start_discord_inbox.bat`常駐) | 純Python常駐。15秒毎にDiscordをポーリング→脈が新鮮(<90s)な部門箱 `local/inbox/<dept>.jsonl`、無ければmain箱へ配達 | **無し**(Claude非関与) |
| `heartbeat.py --name <dept>` | **Claudeセッション内のバックグラウンドタスク**。20秒毎に `local/llm/claude_active_<dept>.txt` をtouch、10分TTLで終了 | **終了時にtask-notification→Claudeが1ターン生成=消費** |

### 無駄の正体
ハートビートがバックグラウンドタスクとして**10分毎に終了通知を発火**し、受信箱が空でもClaudeを起こして1ターン消費する。再武装を繰り返すと「10分毎のトークン垂れ流し」になる。ポーラー側は既にトークン非消費。

### 脈(pulse)の役割の二重性(重要)
`claude_active_<dept>.txt` の新鮮さ(<90s)は、ポーラーの配達先を決める:
- 脈が新鮮 → 新着はその部門箱へ配達(=その部門セッションが処理)
- 脈が切れ(>90s) → 新着はmain箱へ。さらに `sweep_stale_dept_boxes()` が部門箱の取り残しをmainへ回収(自己修復)

→ **脈を生かし続けないと配達が来ない。しかし脈のtouchはバックグラウンドタスクで、その終了がトークンを食う。** これが結合点。

## 3. 実現可能性の結論: **可能。むしろ構成が簡素化する。**

### 提案設計(engineer向け・たたき台)
ハートビート(定期終了型)を、部門ごとの **「inbox-waiter(受信待ち)」バックグラウンドタスク**へ置換する:

```
loop(最大 TTL 回):
    touch pulse file            # claude_active_<dept>.txt を新鮮に保つ=配達先を部門箱に維持
    if inbox/<dept>.jsonl が非空:   # 本物の新着が来た
        exit 0 (marker=message)  # ← この時だけ終了通知→Claudeが起きて処理→返信→再武装
    sleep 15s
# ループがTTLに達したら exit(marker=ttl) で終了
```

- **メッセージ到着時だけ**終了通知が飛ぶ→Claudeが起きる=**イベント駆動**を達成。
- トークンコストが「10分毎に1ターン(常時)」→「実メッセージ1件につき1ターン + 稀なTTL再武装ターン」に激減。
- pollerの配達ロジック・脈判定(90s)・sweep自己修復はそのまま流用可(変更小)。

## 4. 設計トレードオフ(★ここが判断ポイント=難所)

**INC-091(有限TTL安全機構)と正面衝突する。**
- INC-091の設計意図: 脈ループは**あえて有限(TTL)**。Claude本体がフリーズ/暴走しても、TTL満了で脈が止まり、以後の新着はmain箱へ戻る(自己修復)。「終了条件のない無限ループ=偽の生存信号」は禁止(2026-07-14に実障害)。
- waiterを「メッセージが来るまで無限に待つ」にすると、**フリーズしたClaudeが脈だけ新鮮に見せ続ける**→新着が死んだ部門箱に溜まり、`sweep`も回収しない(脈が新鮮に見えるため)。= INC-091が防いだ障害の再来。

**したがって waiter も有限TTLを残す必要がある。** すると idle時のウェイクを完全にはゼロにできず、「TTLを長く(例:30〜60分)して稀にする」までしかできない。
- TTLを長く = idleトークンは減る が、フリーズ時の回復レイテンシが伸びる。
- TTLを短く = 回復は速い が、idleウェイクが増える(現状に近い)。
→ **この TTL の値(idleコスト vs 回復レイテンシ)は方針判断。** ここが research-room へ escalate すべき論点。

### 補足: 脈touchをOSプロセス(Claude外)に出す案は不可
「脈をClaude外の常駐に持たせれば終了通知が飛ばない」は魅力的だが、**フリーズしたClaudeを"生存"に見せる偽生存問題(INC-091)を再導入する**。脈のtouchはClaudeセッションの生存と結びついているべき(=waiterがtouchするのが正しい)。

## 5. 依頼(system-engineer / research-room)
1. §3のwaiter設計の実装可否・工数見積り(engineer)。
2. §4のTTL値=idleコスト vs 回復レイテンシの方針決定(判断が要れば research-room)。
3. TTL満了(marker=ttl)時の再武装を、Claudeの重いターンを起こさず最小コストで回す手当ての検討。
4. INC-091の設計不変条件を壊さないこと(有限TTLは維持)。

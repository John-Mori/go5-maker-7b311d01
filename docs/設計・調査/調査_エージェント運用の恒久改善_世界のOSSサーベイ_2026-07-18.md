## 1. 現行運用の問題インベントリ (実測ベース)

すべて**実際に起きた事故・観測**に基づく。推測の問題は含めない。

- **P1 ファイルキューの脆弱性**: 共有台帳`discord_processed.jsonl`へ13窓が並行追記→処理記録が一時消失(07-16実測・243→247行)。mvドレイン窓での喪失(INC-76/100)。sweepが退避ファイルを部門箱と誤認して空にする(INC-86)。処理済み確認が人力で二重転送(INC-87)。
- **P2 ウェイク非効率**: waiter(TTL45分)は満了ごとに再武装ターンが必要=空起床の連続。旧heartbeat(TTL10分)混用で1時間12回の空起床が実発生(07-16)。孤児通知の洪水も実発生。
- **P3 常駐前提の脆さ**: Windows Update再起動→ロック画面で全セッション停止(memory: remote-uptime-logon-gap)。常駐13窓ぶんの管理・トークン・「開いた司令塔が必須」制約。
- **P4 進捗不可視**: Chami発言「いつになったら終わるんだ?」(07-18)・「着手中のスタンプがないが…」(07-16)。3段印は人力=押し忘れが構造的に起こる(BOOT欠落で全部門が実際に落ちていた)。
- **P5 ドリフト**: BOOT陳腐化(heartbeat/waiter取り違え)・呼称ルールの三重管理・デブライネ4表記。
- **P6 モデル振り分け**: orchestration.mdの散文ルールを人が読んで適用。格上げ判断は事後(失敗してから)。
- **P7 知識検索**: MEMORY.md+個別md+knowledge.mdを目視grep。台帳が増えるほど探せなくなる。

## 2. OSSサーベイの主要な学び

### 2.1 キュー・実行基盤
- **litequeue** (litements/litequeue・2025-02更新が最終=枯れ気味): SQLite 1テーブルのキュー。状態機械(READY/LOCKED/DONE/FAILED)と `UPDATE…WHERE rowid=(SELECT…) RETURNING *` による**1文アトミッククレーム**。PRAGMA: WAL+busy_timeout必須(hueyのissue #445「database is locked」が前例)。
- **SQSのvisibility timeout(概念輸入)**: 受信=一定時間他から見えなくなる(リース)。処理完了=削除、クラッシュ=リース失効で**自動再配達**。長い処理はハートビートでリース延長。→「脈+sweep」のヒューリスティックを、リース失効という決定的な仕組みで置換できる。
- **Transactional Outbox** (AWS公式解説あり): 状態変更とイベント記録を**同一DBの同一トランザクション**で書く。inbox/processedの2ファイル分離は本質的にdual-write問題=「返信は送ったが記録が飛ぶ」の温床、という診断がついた。
- **agent-message-queue** (avivsinai/agent-message-queue・71★・v0.43.1=2026-07-14・活発): **うちとほぼ同一の課題設定**(ローカルのAIエージェント間メッセージング)の既製品。Maildir方式(tmp→new→cur のatomic rename)+送信側が受信ACKを待てる"waitable handoff"。wake通知はWSL必須のため直接導入よりも設計の参照先。
- **Windows固有の教訓**: `os.rename`でなく`os.replace`(同一ボリューム内でアトミック)。`fcntl`は無い→portalocker等。watchdogのReadDirectoryChangesWは**大量イベントで欠落の報告あり**=確実性ならPollingObserver。

### 2.2 オーケストレーション
- **Magentic-One** (microsoft/autogen): **Task Ledger(計画・既知事実・仮定・成功基準)とProgress Ledger(完了ステップ・停滞検知・自己反省)の二重台帳**。進捗が数ステップ停滞したら計画台帳へ戻って再計画する外側ループ。→「進捗が見えない」「計画と実態のズレ」への設計パターンとしてそのまま輸入可能(コード不要)。
- **Claude Agent SDK** (anthropics/claude-agent-sdk-python・公式): `query()`でヘッドレス起動、`session_id`取得→`resume`で**再起動またぎの再開が公式サポート**。フック(PreToolUse/Stop等)あり。→「ディスパッチャがタスクごとにヘッドレス起動」パターンの土台。うちの`dept_boot_prompt.py`(起動文生成器)は既にこの半分を実装している。
- **claude-flow→ruflo** (ruvnet/ruflo・活発): エージェント群の共有基盤に`.swarm/memory.db`=**SQLite**を採用(裏付け)。うちの規模には過剰装備。
- **overstory** (jayminwest/overstory): **3階層ウォッチドッグ**(Tier0=機械的生存監視デーモン/Tier1=AIトリアージ/Tier2=モニターエージェント)。無限リスタートループの既知バグをドキュメント化=「監視の監視」が要る教訓ごと参考になる。
- **claude-squad / vibe-kanban / mission-control**: 複数エージェントの一覧・アタッチUI(tmux前提=Windows直用不可)/カンバン=タスク→worktree実行のマッピング/SQLite単体で動く6列カンバンのテレメトリボード(builderz-labs/mission-control・TS製)。

### 2.3 プロトコル・可視化・ルーティング・記憶
- **A2A** (google/A2A・2026-04にv1.0・Linux Foundation傘下): Taskのライフサイクル語彙 `submitted → working → input-required → completed | failed | canceled` + `TaskStatusUpdateEvent` + `Artifact`。**HTTPトランスポートは不要、語彙(スキーマ)だけ拝借するのが正解**。
- **FrugalGPTカスケード**: 安いモデル→軽量な自己検査→閾値未満なら上位モデルへ、の手書きif文で本質を再現可能。**RouteLLM(学習済みルーター・Claude系転用は再学習要)とlitellm proxy(常駐+多プロバイダゲートウェイ)は13セッション規模には過剰と判断**。
- **観測基盤の正直な評価**: Langfuse=ClickHouse+Postgres+Redis+S3の4系統を個人PCで維持=**不釣り合い**。→自前SQLiteステータスボード(1テーブル)+**Discordの「1メッセージを編集し続ける」ステータスボード**が比例的。
- **記憶**: mem0(★約48k)/letta/zep はいずれも記憶の生成・グラフ化まで踏む重量級=過剰。**sqliteai/sqlite-memory**(★87・2026-06リリース)が「既存markdownをそのまま取り込みFTS5+ベクトルで検索」の先行実装。うちは**ベクトル無しのFTS5(BM25)だけ**が最小で効く。
- **Discord MCP**: SaseQ/discord-mcp(★409・Java製・50+ツール)等が存在。Java常駐が増える点とフル権限スコープが難。優先度低。

## 3. 恒久改善の提案

### 提案A: SQLiteメッセージバス ★最優先(P1・P2の根治)
現行の「JSONL箱+mvドレイン+共有処理済み台帳+sweep推測」を、**SQLite 1ファイルのバスDB**に置換する。

```sql
-- 1メッセージ=1行。msg_id主キーで重複は挿入時点で消える(INSERT OR IGNORE)
CREATE TABLE messages(
  msg_id TEXT PRIMARY KEY, dept TEXT, channel TEXT, author TEXT, body TEXT,
  status TEXT DEFAULT 'pending',    -- pending/claimed/done/failed
  created_at REAL, claimed_by TEXT, claimed_at REAL,
  lease_expires_at REAL, attempts INT DEFAULT 0, result TEXT);
-- PRAGMA journal_mode=WAL; PRAGMA busy_timeout=5000;  (huey #445の教訓=必須)
-- クレーム(1文アトミック・litequeue方式):
--   UPDATE messages SET status='claimed', claimed_by=?, lease_expires_at=?
--   WHERE msg_id=(SELECT msg_id FROM messages
--     WHERE dept=? AND (status='pending' OR (status='claimed' AND lease_expires_at<?))
--     ORDER BY created_at LIMIT 1) RETURNING *;
-- 完了: status='done' と結果記録を同一トランザクションで(=outboxパターン)
```

これ1つで**廃止できるもの**: mvドレイン窓(INC-76/100)・共有processed台帳の並行追記(07-16の記録消失)・sweepのdept名推測(INC-86)・処理済みの人力確認(INC-87)・退避先規約(`local/_work/`)そのもの。
**リース失効=自動再配達**なので、セッションが処理中に死んでもメッセージは戻る=「絶対応対」の4層防衛線が構造で担保される。
- 導入: 鳩(inbox_poller)の書き込み先をDB化+各セッションのドレイン手順をクレームSQL化。既存JSONLは移行期間中は並走(outboxテーブルから旧形式を生成)。
- リスク: SQLiteの単一ライタ制約→WAL+busy_timeoutで13窓規模は問題なし(調査一致)。ただし**書き込みは鳩とセッションのみに限定**する規約が要る。

### 提案B: ディスパッチャ+ヘッドレスワーカー方式 (P3の根治・段階導入)
「部門ごとに常駐セッションを開いておく」から「**1本のディスパッチャ常駐+仕事が来た部門だけヘッドレス起動**」へ。
- 土台は公式: Claude Agent SDKの`query()`+`resume`(session_id永続化で再起動またぎ再開)。**`dept_boot_prompt.py`が既に起動文生成を担っており、半分完成している**。
- 動き: 鳩がバスDBへ書く→ディスパッチャが検知→該当deptのヘッドレスClaude(または安モデル)を起動→A2A語彙でstatus更新→完了で終了。**待機中のトークン消費=ゼロ、再武装=概念ごと消滅、再起動後もディスパッチャ1本を復旧すれば全部門が復旧**。
- 監視はoverstoryの3階層(Tier0=機械的生存監視/Tier1=AI判断)を参考に、既存supervise_daemons.ps1を階層化。
- 対話が要る部屋(dream-care等の機微室・会議)は常駐セッション継続=**全面置換ではなく使い分け**。
- リスク: 長期文脈の喪失→resume+部門メモリファイルで緩和。導入は最後(A/Cの後)。

### 提案C: 二重台帳+Discordステータスボード (P4の根治)
- **Magentic-Oneの二重台帳**を輸入: タスク台帳(何をする・成功基準)と進捗台帳(どこまで・停滞してないか)をバスDBのtasksテーブルとして持つ。状態語彙は**A2Aから拝借**: `submitted/working/input-required/completed/failed/canceled`。「input-required」が「Chamiの返事待ち」を明示できるのが大きい(今は返事待ちと作業中の区別が見えない)。
- **Discordに「1メッセージを編集し続ける」ステータスボード**: 専用chに1枚だけ投稿し、状態遷移のたびにeditする(新規投稿を増やさない)。「いつ終わるの?」はボードを見れば分かる。詳細は1タスク=1スレッド。
- 3段印(送信/既読/着手)は**状態遷移から自動で押す**(claimed=既読、working=着手)=押し忘れが構造的に消える。
- mission-control(SQLiteカンバン)は将来のWeb UI候補として記録に留める。

### 提案D: 規約のスキーマ正本+生成+チェッカー (P5・一部実装済)
- 呼称マトリクス(07-17新設)と整合チェッカー(07-17稼働)で半分完了。残り: 呼称・台帳をYAML等の構造データにし、INDEX.md/manifestの該当部を**生成**に切り替える(手書き三重管理の根絶)。canonical ID+aliasは設計書_人事部門改善§4.3の通り。
- BOOT群も「BOOT_TEMPLATE+生成器(dept_boot_prompt.py)」に片寄せ済みの流れを完成させる(手書きBOOTは部門固有部のみ)。

### 提案E: モデルカスケードの機械化 (P6)
- **FrugalGPT式の手書きカスケード**: ①安いモデル(qwen/Gemini/haiku)で試行→②ルールベースの軽量検査(形式逸脱・長さ・禁止句・自己申告の確信度)→③閾値未満なら上位へ自動格上げ。orchestration.mdの「格上げトリガー」を**事後の反省から事前のプレチェック**へ変換する。
- RouteLLM(要再学習)・litellm proxy(常駐増)は**不採用**と判断(規模不相応)。ディスパッチャ(提案B)の振り分けロジックとして実装するのが自然。
- 品質ルールの担保: 検査に落ちたら**必ず**格上げ(節約のために検査を甘くしない)=「品質を犠牲にしたトークン節約は禁止」の機械化。

### 提案F: 知識のFTS5検索 (P7)
- SQLite FTS5(標準機能・依存ゼロ)で `docs/departments/`・`local/persona_context/`・MEMORY群に全文検索を張る。先行実装=sqliteai/sqlite-memory(既存mdを相対パス保持で取り込む設計)。ベクトルDBは入れない(mem0/letta/zepは過剰)。
- 効果: 「あのルールどこに書いた?」の目視grepが消える。バスDBと同居可能。

### 提案G: Discord MCP (参考・低優先)
- SaseQ/discord-mcp等は存在するがJava常駐+過剰権限。現行のpersona_send/react/pollerは実績があり、**置換の実利が薄い**。バスDB化(A)の方が先。将来ヘッドレス化(B)でツール呼び出しの標準化が欲しくなった時に再評価。

## 4.5 Phase 0 実装記録 (2026-07-18・人事セッション)

## 5. Chamiへの判断ポイント

1. **提案A(バスDB)を本命として改修に発注してよいか** — 全提案の土台。これだけでも事故クラスの大半が消える。
2. 提案C(ステータスボード)を同時にやるか — 「いつ終わるの?」の直接解。Aと同じDBなので同時が効率的。
3. 提案B(ヘッドレス化)まで踏み込むか — 効果最大だが移行コストも最大。Phase 3として後回しでも成立。
4. 不採用判断の確認 — RouteLLM/litellm/Langfuse/mem0系/Discord MCPは「規模不相応」として見送り。異論があれば再調査する。

## 1. 症状と現状

- 月詠み色恋劇場(acc1)で投稿したものが、宵桜艶帖(acc2)の投稿履歴に表示される。
- v184で「A: アカウントをイベントで運ぶ／B: 投稿メタ凍結／C: DID修復」を実装したが、**ユーザー環境では混在が解消していない**。

### v184が守れている範囲（有効なまま）
- 端末内スケジューラ経由・即時投稿の**新規記録**は、投稿データが account を運び、DIDで矯正される。
- 記録先が現在UIと異なるとき、UI状態（videoId/カテゴリ/価格等）を拾わない。

### v184で直らなかった理由（結論の要約）
1. **DID修復は postUri があるアイテムしか分類できない**が、実データは大半が postUri を持たない（§2実測）。分類不能アイテムは「そのまま残す」＝混在が残る。
2. **修復が「moved=0 でも成功」扱い**で一度きりフラグ(`acct_did_repair_v1`)が立ち、以後自動では二度と走らない。
3. **自動修復の失敗・空振りが無音**（DID未解決でも画面に何も出ない）。🩺ボタンを押さない限りユーザーは結果を知れない。
4. **手動追加分（verify_manual__<acct>）は修復対象外**。
5. **クラウド同期（settings_push/pull）が localStorage 全体を丸ごと保存・復元**するため、`short_hist__*`（汚染データ）と `acct_did_repair_v1`（修復済みフラグ）が**セットで巻き戻る再汚染ループ**が存在する。
6. （新規混入の継続源）**GAS無人予約は単一アカウントの資格情報で投稿**しており、channel と実際の投稿アカウントが構造的に食い違い得る（§4-E）。

## 2. 実測エビデンス（2026-07-05取得）

### シート行の postUri 保有率と DID 分布（`action=history` 300件で実測）
| タブ | 行数 | postUriなし | 検出DID |
|---|---|---|---|
| 月詠み(acc1) | 26 | **22 (85%)** | did:plc:shidd44… ×4 |
| 宵桜艶帖(acc2) | 19 | **7 (37%)** | did:plc:ytx35ji7… ×12 |

→ **DIDで所属を判定できる行が少数派**。v184のC修復は大半のアイテムに対して無力だった。
（postUri を持つ行に限れば各タブ単一DID＝現時点でシート上のDID交差は観測されない。混在の実体は「postUri無しの行」と「端末ローカルのみのアイテム」に潜んでいる。）

### GAS 無人予約の投稿資格情報（コード事実）
`gas/コード.gs bskyPost_()` はスクリプトプロパティ **`BSKY_HANDLE` / `BSKY_APP_PW`（1組のみ）** で投稿する。
予約行の `channel` が acc1/acc2 どちらでも、**実際の投稿は常に同じBlueskyアカウント**になる。
→ channel=acc2 の予約が acc1 のフィードに公開される（またはその逆）という**記録ではなく実投稿の取り違え**が構造的に可能。

### クラウド同期の同期対象（コード事実）
`settings-io.js dumpStorage()` は **localStorage 全キー**（秘密パターン除外のみ）を blob 化する。
`short_hist__acc1/acc2`・`verify_manual__*`・`verify_yt__*`・`bsky_did__*`・`acct_did_repair_v1` が**すべて同期対象**。

### v184修復コードの仕様上の穴（コード事実）
- `repairAccountsByDid_` は `short_hist__*` のみ対象。postUri の無いアイテムと `verify_manual__*` は素通し。
- シート矯正の削除は `action=delete&postUri=` のみ＝postUri無し行はシート側も直せない。
- `maybeRepairAccountsOnce_` は `ok:true`（=DID解決成功）なら moved=0 でもフラグを立て終了。
- `bluesky-posted` リスナの `setAcctDid_(account, did)` が**毎投稿で無条件上書き**＝一度の取り違えイベントで DID台帳自体が汚染され、以後の矯正・修復が誤動作し得る。

## 3. 根本原因（多層）

**第1層（発生源）**: 「所属アカウント」がアイテムの属性ではなく**保存キー（short_hist__acc2 等）だけ**で表現されている。キーを跨いで入った瞬間、アイテム自身は無実の顔をして混ざる。
**第2層（判定材料の欠落）**: postUri/postUrl/videoId の「所属を証明できる3点セット」が全アイテムに揃っていない（歴史的経緯で postUri 無しが大半）。
**第3層（修復の不全）**: 分類器がDID単段・対象がshort_histのみ・一度きり・無音。
**第4層（再汚染）**: クラウド同期が履歴と修復フラグを丸ごと巻き戻す。
**第5層（新規混入の継続源）**: 無人予約の単一資格情報、手動追加のUI依存、DID台帳の上書き汚染。

## 5. 恒久設計

### 5.1 データモデル：`account` をアイテムの必須属性に
- `histAdd`/手動追加/シート行(sync)のアイテムに **`account: 'acc1'|'acc2'`** と **`accountSrc: 'did'|'event'|'ui'`**（確度）を必ず保存。
- 保存キー（short_hist__acc2 等）は「表示用の棚」、`item.account` が「戸籍」。棚と戸籍が食い違えば**画面上で不一致バッジ**を出せる＝混在が自明に見える。
- 投稿時は **postUri / postUrl / videoId の3点セットを必ず保存**（現行コードは既に保存経路がある。欠けを許さない検証を追加）。

### 5.2 多段分類器 `classifyAccount(item) → {account, confidence, by}`
判定材料を上から順に試す。**1つでも確定したら終了**。
1. **postUri の DID**（確実）… `at://did:plc:X/…` を DID台帳と照合
2. **postUrl のハンドル**（確実）… `bsky.app/profile/<handle>/post/…` の handle を両アカウントのハンドル（resolveHandle結果も含む）と照合
3. **videoId の接頭辞**（強）… `acc1-YYYYMMDD-…` / `acc2-…`（IdGen.makeVideoId が発番時のアカウントを埋め込んでいる。実測: シートのpost_id `acc2-20260701-0428-s7no` ⇔ channel acc2 一致）
4. **短縮URLの展開**（中）… r2ワーカーのKVは code→長URL を持つ。長URLは bsky.app/profile/<handle>/… → 2 に帰着（r2に展開APIを1本追加）
5. **分類不能** … `unknown` を返す → 修復では動かさず、**UIに「⚠️所属未確定」バッジ＋⇄移動ボタン**を表示して人が1タップで確定（確定時に item.account と accountSrc:'manual' を保存）

### 5.3 修復ジョブの再設計（「一度きり・無音」をやめる）
- **対象拡大**: `short_hist__acc1/acc2` ＋ `verify_manual__acc1/acc2`（＋整合として `verify_yt__*` のキー移し替え）。
- **実行タイミング**: 一度きりフラグを廃止し、**投稿履歴を開くたびに軽量検査**（分類器で全アイテムを走査→不一致件数を数えるだけ。移動はユーザー確認 or 自動設定）。件数0なら無音、>0なら「⚠️ N件が別アカウントの投稿の可能性 → [検査レポート]」を表示。
- **可視レポート**: 🩺ボタンは「移動しました」だけでなく、**判定内訳（DID/handle/videoId/不能）と各アイテムのタイトル・移動先**を一覧表示。分類不能は⇄ボタン付きで列挙。
- **シート側の矯正**: postUri が無くても **videoId(post_id) で行を特定**して移動（`op:'move_row'` をGASに追加：正チャンネルへ行コピー→元行クリア。post_id も postUri も無い行はタイトル＋投稿日時で候補提示に留める）。
- **DID未解決時**: 無音で諦めず、「⚙のハンドル欄がメール形式だとDID解決できません。ハンドル(xxx.bsky.social)を入れてください」と**具体的に案内**（resolveHandle はメールでは失敗する）。

### 5.4 再汚染の遮断（クラウド同期のポリシー変更）
- `settings-io.js` に **「同期しないキー」のパターン**を導入:
  `short_hist__* / verify_manual__* / verify_yt__* / delta_cache / peak_cache / clicks_cache / acct_did_repair* / bsky_did__*`（履歴・キャッシュ・端末判定系）
- 理由: これらは「設定」ではなく**記録データ**。記録の正本はシート（＋各端末ローカル）であり、blob経由の複製は修復を巻き戻す事故しか生まない。
- 既存blobからの pull 時も同パターンを**取り込み側で拒否**（多層防御。古いblobにも耐える）。
- YT URLはシートから復元する仕組みが既にある（INC-61）ため、同期から外しても失われない。

### 5.5 DID台帳の堅牢化
- **正とする情報源**: `bsky_handle__<acct>`（⚙設定）→ `com.atproto.identity.resolveHandle`。
- イベント由来の学習は **「未設定のときだけ」** 許可。設定済みDIDと投稿DIDが食い違ったら**上書きせず警告**（「⚙のハンドルと実際の投稿アカウントが一致しません」）＝ 取り違えの早期検知器に転用する。
- 検証テスト: resolveHandle結果とイベントDIDの一致を初回投稿時に確認し、`accountSrc:'did'` の信頼を確立。

### 5.6 無人予約のチャンネル別資格情報（実投稿の取り違え防止・実害級）
- GASスクリプトプロパティを **`BSKY_HANDLE_ACC1/BSKY_APP_PW_ACC1` と `BSKY_HANDLE_ACC2/BSKY_APP_PW_ACC2`** に分離。
- `runReservations` は予約行の channel で資格情報を選択。未設定チャンネルの予約は **投稿せず error にして通知**（間違ったアカウントで公開するくらいなら止める）。
- 旧 `BSKY_HANDLE/BSKY_APP_PW` は ACC1 のフォールバックとして残し移行を容易に。
- 記録行にも `post_uri` を必ず書く（現行コードは書いている）→ 事後検証可能。

### 5.7 ガードレール（記録時の最終検証）
- 記録直前に **「これから記録する channel」と「postUri の DID」を必ず突合**。不一致なら DID側を正として記録し、`accountSrc:'did-corrected'` を付け、ステータス欄に「⚠️ 記録先をDIDで矯正しました」と1行出す（無音でやらない＝異常の発生自体に気づける）。
- 📤シート同期・🧹掃除の実行前に分類検査を挟み、不一致があれば中断して知らせる（汚染の拡散防止）。

## 7. テスト計画（preview＋実データ）

1. **分類器単体**: postUriのみ/postUrlのみ/videoIdのみ/全部なし の4種アイテムで期待どおりの account/confidence を返すか。
2. **修復E2E**: acc2ローカルに「acc1のDIDを持つ」「acc1のpostUrlを持つ」「acc1-接頭辞videoIdのみ持つ」「無印」の4件を混入→ 検査レポートに3件移動＋1件が⚠️未確定で列挙されること。verify_manual 側も同様。
3. **再汚染遮断**: 汚染blobを settings_pull で取り込んでも short_hist__* が上書きされないこと（取り込み側拒否の確認）。
4. **DID台帳**: 設定済みDIDと異なるDIDのイベントで上書きが起きず警告が出ること。
5. **無人予約**: channel=acc2 の予約が ACC2 資格情報で投稿されること／ACC2未設定なら error 停止すること（GASテストモード併用）。
6. **回帰**: 通常の即時投稿・端末内予約・履歴表示・📤同期・YT URL復元が従来どおり動くこと。

## 8. 移行・ロールバック

- 移行: P0はデータ変更なし。P1の修復は**移動ログ（元棚→先棚・判定根拠）を localStorage に保存**し、🩺レポートから「直前の修復を取り消す」を可能にする。シート move_row も元行の内容をログ列に退避してから実行。
- ロールバック: 各フェーズは独立コミット。P1で誤移動が出た場合は移動ログから逆適用。

### 付記（本調査で判明したがスコープ外の観察）
- 端末内スケジューラの予約キューはメモリのみ＝リロードで消える（混在とは無関係だが予約の信頼性課題）。
- シートの postUri 欠落行が多いのは歴史的経緯（postUri記録以前の行・videoId後追いupsert行）。P1の分類器はこれを前提に設計している。

## §1. 製品の設計原則(事故からの帰納)

今回の事故群を並べると、製品側の病は4つに集約される。

| 原則 | 破った実例(全て実際に起きた) |
|---|---|
| **P-1「読めていない」と「無い」を同一視しない** | hydrate完了前にモーダルを開くと空pendingが生成→保存で**画像・コメント永久削除**(v=349で修正)。同型がsync.jsにも居た(R2取得失敗を空文字で上書き=v=333②) |
| **P-2 1操作=Nリクエストを許さない** | クリック数をコード毎に1本(投稿N件×2導線)→1操作で数百リクエスト。sync画像確認もキー毎に1本→**go5-sync単独で159k req/日=無料枠超過**(v=343/344で修正) |
| **P-3 見えない系は必ず焼ける** | 159k/日は**超過メールが来るまで誰も知らなかった**。セール会場「累計0」も**Chamiが違和感を言うまで**放置。観測が無い系は壊れても静かなまま |
| **P-4 安全網は発火させて初めて安全網** | 研究室自動復活(go5_lab_revive)は**ID体系の取り違えで実は動かない状態**だった。「登録できた」で満足し本番条件を一度も通していなかった(研究室が発見・修正) |

## §2. 製品の改善タスク表

凡例: 工数=改修セッションの実作業目安 / 担当=推奨モデル(orchestration.md分業表準拠) / **改善は承認制**=着手前にChami承認。

### P0(データ保全・事故の再発防止)

**B-1. 宵桜③(acc2履歴消失)を「待ち」から「攻め」へ — 消失の瞬間を自動捕獲**
- 背景: 唯一の未解決INC。現状は「次に消えた時にChamiがF12で採取」という**人間頼みの受け身**。再現が稀で、消えるたびにChamiが復旧してしまうと証拠も消える。
- 内容: `yt-clicks.js`に軽量テレメトリを常設——`short_hist__acc2`/`verify_manual__acc2`の**件数が減った瞬間**に、直前スナップショット(件数・先頭数件のid・直前に走った処理名)を`localStorage.hist_loss_evidence`へ自動記録。サニタイザ(`sanitizeOwnership_`)・復元処理の入口/出口に犯人特定用のマーカーを置く。
- 受け入れ条件: 手動でhistを減らすテストで証拠が記録される/通常運用でサイズ増ゼロに近い(直近3件のみ保持)/消失発生時にChamiは「消えた」と言うだけでよくなる(採取不要)。
- 工数: 0.5日 / 担当: opus(データ消失系のため) / リスク: 低(読み取り+localStorage1キー)

**B-2. 破壊的書込ゲートの横展開 — 「未ロード中の空=削除」を全ストアで禁止**
- 背景: v=349の修正は`candidates.js`の2関数(refImgSave/bskyImgSave)に**個別に**ガードを入れた。だが同じ病(P-1)は多層ストア(localStorage/IDB/R2/GAS/D1)の**削除・空上書き全部**に潜在し得る。実際にsync.js(v=333②)でも同型が出ている=**2例目が出た時点で構造問題**。
- 内容: ①全JSの「削除・空配列/空文字での上書き」箇所を棚卸し(grep+目視で一覧化) ②非同期ロードを挟むストアには`ready`フラグ+「未ready中の破壊的書込は拒否してconsole.warn」を共通パターンとして適用 ③棚卸し結果をknown-issues.mdに残す(将来の新コードのチェックリストになる)。
- 受け入れ条件: 棚卸し一覧が存在する/未readyで削除系APIを叩くテストが全て「拒否」になる/既存の正常系デグレなし(テスト全PASS+実機確認)。
- 工数: 1日(棚卸し半日+適用半日) / 担当: opus(見えないバグ系) / リスク: 中(触る範囲が広い→棚卸しと適用を別commitに分ける)

### P1(観測性=「見えない系は必ず焼ける」への恒久対策)

**B-3. 計測ヘルスの常時可視化 — 「本当に計測できてるのか?」を構造的に解消**
- 背景: セール会場0事件・「YouTube→Bluesky 0なのにBluesky→FANZA 19」事件。Chamiが数字を疑うたびに調査が1回走る。**信頼は可視化からしか生まれない**。
- 内容: 投稿履歴タブ上部に1行のヘルス表示——`計測: r2 OK / GAS OK(最終スナップ 12:30) / YT API OK`。実体は既存`api-diag.js`の縮約自動版(各系1リクエスト・60秒キャッシュ=P-2遵守)。異常時のみ赤字で「r2応答なし(クリック数は古い値)」。
- 受け入れ条件: 正常時は目立たない1行/r2・GAS・YTのどれかを止めた時に該当だけ赤くなる/リクエスト増は開いた時+手動更新時のみ。
- 工数: 0.5日 / 担当: sonnet(表示中心・診断ロジックは既存流用) / リスク: 低

**B-4. Cloudflare使用量の朝報告 — 159k事件の再発を「翌朝気づく」に変える**
- 背景: 無料枠10万req/日への到達を**超過メールで知った**(P-3の最悪形)。修正後も「効いたかはChamiがダッシュボードを見る」手動運用のまま。
- 内容: Cloudflare GraphQL Analytics APIで**Worker別の日次リクエスト数**を取得し、鳩の日次サマリ(研究室の書§4系の朝報告に相乗り)へ1行——`CF: sync 3.2k / r2 4.1k / fanza 0.4k (計7.7k / 枠100k)`。**8万/日超で警告**。
- 前提: **ChamiがCloudflare APIトークン(Analytics閲覧のみ・最小権限)を発行**し`local/`へ配置(裁定ポイント②)。
- 受け入れ条件: 毎朝1行が届く/しきい値超過で警告表現に変わる/トークンはlocal/のみ(repo/ログに出さない)。
- 工数: 0.5日 / 担当: opus(API+秘匿情報) / リスク: 低 / 所有権: 鳩=研究室のため**発注仕様として研究室へ**

**B-5. `?v=`バンプのスクリプト化 — 複数セッション衝突の根絶**
- 背景: 実際に私が**別セッションのバンプ(342)を知らずに343を打ち**、相手に344への再バンプを要求する事態になった。並行セッションが常態化した今、手動sedは衝突装置。
- 内容: `scripts/bump.mjs`——現在値を全参照から検出→最大値+1へ一括置換→参照数の一致を検証→新版数を出力。全部門は「sedではなくbump.mjsを使う」(BOOT/規約に1行)。
- 受け入れ条件: 2連続実行で番号が2つ進む/参照の取り残しゼロ/どのセッションから実行しても衝突しない(現在値検出ベースのため)。
- 工数: 0.25日 / 担当: sonnet / リスク: 低

### P2(将来・費用対効果を見てから)

**B-6. 巨大ファイルの増分分割規約**
- 背景: `yt-clicks.js`3,000行・`candidates.js`2,600行。今回の調査でも「どこに何があるか」の探索コストが毎回発生。ただし**一括分割は回帰リスクが工数に見合わない**。
- 内容: 規約1行のみ——「**新規の独立機能は新モジュール(core/または機能名.js)に置く。既存巨大ファイルへの追記は既存機能の修正時のみ**」。分割はしない(増やさないだけ)。
- 工数: 規約追記のみ / リスク: ゼロ

**B-7. syncの適応間隔**
- 背景: 60秒固定でも1タブ1,440回/日。変更が無い深夜も同じ頻度で叩く。
- 内容: 直近30分無変更なら間隔を60→300秒へ自動延伸(変更駆動requestSyncがあるため反映遅延は起きない)。
- 判断: **B-4の観測が入り、実測で必要と分かってから**。現状無料枠の数%なら着手しない(過剰最適化)。

## §3. 改修部門(このセッション)の役割定義と型

研究室の書§3-2は「今日一番機能した」と評したが、**何が機能したのかの型**を残さないと再現できない。今日の実績から抽出する。

### 3-1. 分界(α/β)
| 窓 | 担当 | 今日の実例 |
|---|---|---|
| **改修α** | 既存機能の修理・原因不明バグの真因追跡・横断調査 | Cloudflare 159k真因特定/hydrate競合(データ消失)/書体・表示改修 |
| **改修β** | 新規機構の実装(既存と独立な案件) | 進捗印3段化・qwen教育 |
| 共通規約 | **βの脈が生きている間、αはβのwaiterを持たない**(二重所有=事故。実際に踏んで返上した) | BOOT.md(92a95ba)に明文化済み |

### 3-2. 検証の型(今日確立したもの・全改修で必須にする)
1. **バグは「壊してから直す」**: 修正前に本番条件で事故を再現し(hydrate競合=32MB/37枚を仕込み展開を遅延)、修正後に**同じ攻撃でガードが発火する**ことを確認する。再現せずに直した修正は「直ったつもり」でしかない(2a16fceがそうだった=Chamiの「多分治ってない」が正しかった)。
2. **安全網は1回発火させてから「出来た」と言う**(P-4): 登録・設定だけで完了報告しない。go5_lab_reviveの教訓。
3. **実測値をcommitメッセージに書く**: 「40本→0本」「1,481,161px=計算値一致」のように。後から効果を検証できる形で残す。
4. **検証用フックは完全撤去してから出す**: grep残骸ゼロを確認(slow=フックの教訓——1回目の撤去は失敗していた)。

### 3-3. 着手前チェックリスト(越権と重複の防止・今日3回機能した/1回破りかけた)
1. `python scripts/discord/triage_inbox.py`+processed台帳で**未処理か**確認
2. **Discordの実発言を自分で引いて全文確認**(転送の欠落・ID取り違えが実際に起きた)
3. 対象ファイルの`git status`と**他部門の脈**を確認(所有者が生きているなら触らない——私はβの領分に手を出しかけ、Chamiの(A)選択と逆の(B)を実装しかけて撤回した)
4. 横断(sync-core/account-model/scripts系)なら着手前に研究室へ

## §4. 研究室への発注仕様(所有権が研究室にあるもの・本書は仕様のみ)

研究室の書§2-1「人間の正確な手作業を前提に設計している=最大の穴」の各論。**起床の正順4段は現在、AIが手順を守ることに依存している**。私自身が正順の元となる事故(&起動で耳が死んだ)を起こした張本人として、構造化の仕様を置く。

**R-1. `wake.py`(起床処理の圧縮)**: `python scripts/discord/wake.py --name system-engineer`1発で——①`local/_work/`へmv退避(INC-86準拠) ②既読リアクションを退避分へ一括付与 ③workファイルのパスと件数を出力。

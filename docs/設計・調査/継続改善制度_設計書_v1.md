
- 改善ループ: **観測→仮説→提案→★Chami承認→実装→効果測定**(勝手な確定ルール化・自動適用は禁止)

## 1. データの置き場と書き手

| テーブル(D1 go5_kaizen) | 書き手 | 経路 |
|---|---|---|
| improvement_requests | 司令塔 | `wrangler d1 execute`(PC・管理者) |
| user_events | フロント各端末 | `core/kaizen-log.js` → fanza-worker `/api/kaizen-event` → D1 |
| system_changes | 司令塔 | `wrangler d1 execute`(デプロイ/バンプ時) |
| improvement_insights | kaizen-analyst(経由で司令塔) | `wrangler d1 execute` |

設計判断: HTTP口が必要なのは**スマホからも飛ぶuser_eventsだけ**。他はPCの司令塔がwranglerで直書きできるためWorkerルートを増やさない。ルートは既存fanza-workerに相乗り(認証=既存と同型のソフト鍵+Origin制限)し、新Worker追加はしない。DBはfanzaデータ(go5_fanza)と分離した`go5_kaizen`(ドメイン分離・障害隔離)。

## 2. イベント一覧(第1段階で記録するもの)

自動購読(コード追加不要・既存DOMイベント): `video_generated`(video-created) / `bsky_posted`(bluesky-posted) / `account_switched`(account-changed)
candidates.js計装: `candidate_added` / `ref_image_saved` / `fetch_now_requested`

第2段階の候補(必要になったら追加): candidate_hidden/restored, sort_changed, copy_edited(テンプレ修正差分), post_edit_opened, video_regenerated, short_url_created, publish_time_changed 等。
**記録しないもの**: マウス移動・スクロール・滞在時間などの高頻度低意味イベント、秘密(鍵/パスワード/Cookie)、チャット全文。

## 3. 送信設計(上限を突かないため)

- フロントはバッチ送信: 30秒ごと or 20件到達 or タブ離脱時(keepalive)。1操作=1リクエストにしない
- 失敗は静かに破棄方向(キュー上限100・古い方から捨てる)。本業を邪魔しない
- 無効化スイッチ: localStorage `kaizen_log_off='1'`
- D1書き込み量見積もり: 現実的な操作量(数十〜数百イベント/日)で **~300行/日 ≪ 10万行/日枠**。ストレージも1行~200Bとして年間~20MB ≪ 5GB。requests/changes/insightsは1日数行。**枠の心配は実質不要**

- [ ] 第2段階(1〜2週間データが貯まったら): kaizen-analystによる初回分析→改善提案をChamiへ提示(承認ゲート)。週次ペースから
- [ ] 第3段階: 部門への自動配信(insightsを各部門が自分の担当分だけ読む)・効果測定の定型化

## 5. 司令塔の運用義務(orchestration.mdに詳細)

1. Chamiが改善/修正/改修を依頼したら improvement_requests へ1行記録(REQ-YYYYMMDD-NNN)
2. デプロイ/バンプしたら system_changes へ1行記録(CHG-NNN・request_idで紐付け)
3. kaizen-analystの提案は必ず「このような改善はどうか」とChamiへ提示。承認までstatus='proposed'のまま
4. モデル振り分け: 司令塔=Fable5(→Opus)。部門既定=sonnet(qaのみhaiku)。難題はopusへ格上げ、定型はhaikuへ格下げ

## 6. セキュリティ/プライバシー

- ログへ秘密(APIキー/パスワード/トークン/Cookie)を入れない(ロガーはフィールド許可制+文字数上限、Worker側でも再構築)
- user_eventsのsession_idはタブ単位のランダムID(個人特定情報ではない)
- 原文ログ(raw_text)は要点のみ。チャット全文は保存しない

## 2. 設計方針

> **AIの正本＝ローカル / 人が見る窓＝Notion**(既定方針)を維持する。Notionへ正本を移さない。

AIにとって扱いやすい保管とは、次の4つが満たされている状態と定義する。

1. **1つの事実が1箇所にある**(台帳が2本ある状態を潰す)
2. **いま必要な分だけが手前にある**(古い分はアーカイブへ退ける。全文grepは可能なまま)
3. **機械が書き、機械が移す**(AIの手作業に依存しない)
4. **消えない**(最低2箇所に存在する)

形式は現行維持でよい。理由: jsonl は追記が安全で1行1事実、md は人もAIも読める、D1 は集計に強い。**この3点セットは既に正しい。**

## 3. 提案(承認をいただきたい単位で分割)

### 提案1: `local/` のバックアップを作る 【実施済 2026-07-17・検証済】

実装 = `scripts/maintenance/backup_local_to_drive.ps1` / タスク `go5_backup_local_daily`(毎日04:00・非表示)。

| 項目 | 決定 | 理由 |
|---|---|---|
| 保存先 | Googleドライブ `マイドライブ/AFI5秒動画設計・戦略運用/001_システムバックアップ/local/YYYY-MM-DD/` | Chami明示(2026-07-16)「Googleドライブなら安全だ…他のクラウドとは違って特別な場所だから許可してほしい」「001_システムバックアップの中に移動してほしい」 |
| 対象 | `local/` 全体(227ファイル/58.4MB)。`dreams`・`past`・APIキーも含む | Chami明示「local/だけの認識で大丈夫」「キーとか全部そのまま入れて大丈夫だよ、ちゃんと保護してるから」(2026-07-16) |
| 方式 | **日付別スナップショット(robocopy /E)。ミラー(/MIR)ではない** | ミラーは誤削除がバックアップ側へ伝播する＝守りたいものが守れない。日付別なら昨日の分に残る |
| 世代 | 14世代。15世代目より古いものだけ整理 | 58.4MB×14 ≒ 0.8GB。Drive空き12.7GBに対し十分 |
| 中止条件 | G:未マウント / 空き2GB未満 で中止(ログに残す) | Driveを満杯にすると動画自動保存(`マイドライブ/AFI5秒動画`)を壊す |

**検証(2026-07-17)**: 登録しただけで満足しない。実際にタスク経由で発火させ、`LastTaskResult=0`・保存先に227ファイル/58.4MB着弾・`local/backup.log` に成功行を確認済。
※ 初版は robocopy の戻り値(rc=1＝コピー成功)がそのまま終了コードへ漏れ、成功が失敗として返っていた。`exit 0` を明示して修正済。

**却下した案**: Cドライブ(別物理ディスク)へのコピー。当初これを推したが、Chamiの判断でDrive採用。Driveなら家ごとの事故にも耐えるため上位互換。
**Dドライブは不可**(採用しなかった理由): `D:` = ディスク0 = TOSHIBA DT01ACA100(SATA機械式HDD)で、**守りたいデータ自身が既にDに載っている**。同一物理ディスク内の複製はバックアップにならない。`C:` = ディスク1 = NVMe SSD。

### 提案2: 処理済み台帳を1本化する

`discord_processed.jsonl` を正とし、`discord_inbox_processed.jsonl` は統合のうえ退役させる(`local_responder.py` の追記先を変更)。
統合時は `msg_id` で重複排除。旧ファイルは `local/archive/` へ退避し削除しない。

### 提案3: 月次アーカイブ(ローテーション)

`local/archive/YYYY-MM/` へ、その月より古い行/ログを移す小さなスクリプトを日次メンテに追加。
**削除はしない。退けるだけ。** grep したい時は archive も含めて grep すればよい。
対象: `discord_processed.jsonl`、`*.log`、`local/llm/*_log.jsonl`。

### 提案4: 受信箱の移動を機械にやらせる

現在AIの手作業。`msg_id` を渡したら「箱から消す→台帳へ追記」までやる小スクリプトを1本用意し、全部門のBOOTをそれに差し替える。
※ ただし §1.3 の掃除機能を先に直さないと、箱から消えるのが先で意味を成さない。**提案4は §1.3 の修正が前提。**

### 提案5: Notion(人が見る窓)

正本を移さない前提なので、Notionは**書き出し先**であって保管先ではない。
Chami がスマホで見たいもの(推測: 今日の受信/対応状況・改善要求の一覧・インシデント)を、ローカル/D1から定期push する読み取り専用ページとして作るのが筋。
**着手は API キー(現在ステップ1進行中)と、Chami が「何を見たいか」が決まってから。** ここを決めずに作ると使われないページが増える。

### 4.2 改名構想(go5-maker → 5SecMovieMaker)との関係

Chamiが2026-07-16に改名を希望(「AIが勝手につけた名前でわかりにくい」)。**整理と同時にやらない**方針でChamiへ提案済。理由:

- 公開URLは**変わらない**(GitHubリポジトリ名は既に別名 `go5-maker-7b311d01`。Pages URLはそちらに紐づく)。ここは安心材料。
- 一方、フォルダ改名は**絶対パスを文字で持つ21ファイル＋Windowsタスク6件＋スタートアップ1件**を壊す。しかも**全て静かに壊れる**(エラーが出ない・誰も報せない)。**本設計で作ったバックアップ自身も止まる**——バックアップが死んだことに気づくのはデータを失った日。
- Claude Codeのプロジェクト履歴・memoryはcwdの絶対パスに紐づくため、改名後は別プロジェクト扱いになり**過去の文脈へ到達できなくなる**(消えないが迷子)。`00_AI-HQ`(リポジトリ外)の台帳も追従しない＝最も静かに壊れる。
- → 改名は「整理のついで」ではなく**独立した移行作業**として、住所の書き換えリストを作ってから一気に実施する。

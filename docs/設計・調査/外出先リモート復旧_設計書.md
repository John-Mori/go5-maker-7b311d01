- 位置づけ: §1〜4は**今日から使える運用手順**(コード変更なし)。§5のみ改善提案(承認制=Chami承認まで実装しない)

## 0. まずこれだけ (30秒版)

1. **セッションが固まった** → アプリを完全終了して再起動。ダメなら**新しいセッションを開く**。push済みの作業は絶対に消えない
2. **Discordに返信が来ない** → 今は仕様(司令塔不在+ローカル一次受付停止中)。急ぎの用件は**Claude Codeの新セッションに直接指示**する方が早い
3. **サイトがおかしい** → スマホのブラウザで公開URLを直接開いて実測する

## 1. 障害マップ (どこが死ぬと何が止まるか)

### ◎ スマホから直せるもの
- **スマホのClaudeセッション** … 固まっても新セッションで続行できる(§3-A)
- **クラウドのClaude Codeセッション** … 何個でも新しく開ける。調査・修正・pushまで完結(§3-D)
- **GitHub Pages(公開サイト)** … 状態確認も再pushもスマホから可能(§3-C)
- **Cloudflare Workers(fanza/sync/link/drive)** … コード修正→pushでActionsが自動デプロイ

### ✕ 帰宅しないと直せないもの (PC必須)
- **自宅PCの電源/ネット** … 落ちるとDiscord受信・ローカルqwen・販売数取得が全停止
- **inbox_poller.py(Discord受信の常駐)** … 停止中はDiscord発言が受付箱に入らない
- **司令塔セッション(PCのClaude Code)** … 不在の間はDiscord返信が発生しない(受信は溜まる)
- **GAS反映** … claspの認証情報(local/)がPCにしかない
- **PC上の未pushの作業** … 回収はPCでのみ可能

## 2. 5分診断フロー (スマホのみで全部できる)

順番にやると「どこが死んでいるか」が絞れる。

### Step 1: 公開サイトの生存
Safari等で開く:
`https://john-mori.github.io/go5-maker-7b311d01/`
- 開けて動く → Pages・配信は正常
- 開けない/古い → §3-C へ

### Step 2: 直近のpushとビルド
スマホのブラウザでGitHubを開く:
- コミット履歴: `github.com/John-Mori/go5-maker-7b311d01/commits/main`
- ビルド状況: 同リポジトリの **Actions** タブ → `pages build and deployment` が success か
- ここで最終pushの日時も分かる=「PC側でどこまで作業が確定したか」の証拠

### Step 3: PC・受信系の生存 (Discordを使った生存ping)
Discordの **#llm成長ルーム** に短い質問を送る。
例: 「今日の日付わかる?」
- **注意: アメス/アロンソ/監督/司令塔/Claude の名前を入れない**(入れるとClaude側へ回されて返信されない)
- 目安3分以内にローカル受付が返信 → **PC稼働+ポーラー稼働+ollama稼働** が一度に確認できる
- 無返信 → PC側の何かが死んでいる(§3-B-3)

### Step 4: 司令塔の生存
現状、外から直接確認する手段が**ない**。
Discordの部門chに送って返信が来ない=司令塔不在、と推定するしかない。
watchdog稼働後(§5-P1+P3)は、受付箱が15分以上未処理だと総合受付chへサマリ通知が来るため、外からでも「気づける」ようになる(PC側で `start_absence_watchdog.bat` が起動している場合に限る)。

### Step 5: 代理セッションの起動
Claudeアプリ → Claude Code → このリポジトリで**新セッション**を開き、調べたいこと・直したいことを指示する。

## 3. 症状別リカバリ手順

### A. セッションフリーズ (2026-07-14 事例1)

症状: 「メッセージを読み込んでいます」のまま進まない。

1. アプリを**タスクキルして再起動**(iOS: アプリスイッチャーから上スワイプ)
2. 電波の良い場所で開き直す(4Gが弱いと履歴ロードが詰まりやすい)
3. 直らなければそのセッションは置いておき、**新セッションを開始**。冒頭に「`引き継ぎ_Vol6.md` を読んで」と一言添えれば文脈が繋がる
4. **安心材料**: 作業は git push の時点で確定済み。フリーズで失われるのは「未送信の入力」だけ。履歴もサーバー側に残っているので、後日開き直せることが多い

### B. Discord無返信 (2026-07-14 事例2)

返信が発生する条件は3層。上から順に疑う:

**1. 司令塔不在 (最多・現状の主因)**
返信は「司令塔セッションが受付箱を読む」ことで初めて発生する設計(自動botではない)。司令塔が落ちていれば返らないのが仕様。
→ 急ぎ: Claude Codeの新セッションに用件を直接指示(Discordを経由しない)
→ 帰宅後: PCでClaude Codeを開き「Discord確認して」で溜まった分を一括処理

**2. ローカル一次受付が停止中 (2026-07-14〜)**
名前呼び横取り対策で `LOCAL_FIRST_ENABLED=False`(コミット 6c28121)。現在ローカルが答えるのは **#llm成長ルーム**(常時)と**画像生成室**(qwen呼びの時)だけ。他の部屋はローカルも沈黙する。
→ 再開はChami裁定(§5-P2)。

**3. PC/ポーラー停止**
llm成長ルームでも無返信ならこれ。
→ 帰宅後 `scripts/discord/start_discord_inbox.bat` を再起動。
→ **取りこぼしは復帰後に自動回収される**: ポーラーは「最後に見たメッセージID」を state に保存しており、再起動すると停止中の発言も after= で遡って取り込む(state ファイルを消さないこと)。Discord上のメッセージ自体も消えないので焦らなくてよい。

### C. 公開サイトが更新されない/開けない (INC-66型)

1. Actions の `pages build and deployment` を確認(Step 2と同じ)
2. サイトを開いて挙動・版数で実測(push成功≠公開成功)
3. ビルド失敗なら新セッションに「Pagesビルド失敗を調査して直して」と指示 → 修正pushまで外から完結できる

### D. 代理セッション(クラウド)でできる事・できない事

**できる(外出先から完結):**
- コード調査・監査・修正・Nodeテスト実行・push
- mainへ反映すればPagesに1〜2分で自動公開
- Workers修正(pushでActionsが自動デプロイ)
- GitHubの状態確認(コミット/Actions/PR)

**できない(PC必須):**
- Discord受付箱の処理・返信(トークンと受付箱=local/ はPCにしかなく、クラウド環境は外部通信も制限される)
- GAS反映(claspの認証情報がPCのみ)
- ローカルqwen・販売数スクレイパ等のPC常駐処理
- PC上の未pushの作業回収

## 5. 恒久対策の改善提案 (承認制・実装はChami承認後)

優先度順。P1が最小コストで今日の「無返信の不安」を消す。

**P1+P3: 司令塔不在アナウンス+不在watchdog → 実装済(2026-07-14・absence_watchdog.py として統合実装。PC側で start_absence_watchdog.bat の起動が必要)**
受付箱(`local/discord_inbox.jsonl`)の行を読み取り専用で監視し、**15分以上未処理の行が1件でもあれば**「司令塔不在の可能性」と判定(heartbeatの生死は問わない=フリーズで固まったheartbeatの偽陽性も拾う設計)。滞留メッセージの発生元chへ定型文1行を自動返信し、総合受付chへ件数・最古滞留時間・heartbeat鮮度・司令塔待ち件数のサマリを送る。暴走ガード(同一メッセージへの返信は生涯1回・1周期最大3件/1時間最大6件・サマリは60分に1回)つき。詳細=`scripts/discord/absence_watchdog.py`。

**P2: ローカル一次受付の再開 → 見送り(Chami裁定「まだ早い」2026-07-14)**
再開条件はコミット 6c28121 に明記済み: **全人格名+コーチ/先生を CALL_WORDS に追加**してから `LOCAL_FIRST_ENABLED=True` に戻す。

**P4: スマホからPCの司令塔を起こす (大型・任意) → 未着手(別途設計)**
INC-59の実績パターン「workerにフラグ→PC常駐が拾う」を流用し、Discordの特定コマンド(例: `!wake`)でPC側がClaude Codeセッションを自動起動する。設計・費用の検討が別途必要。

## 5.5 リモートデスクトップ (PC必須の穴を塞ぐ本命・帰宅前に一度だけ設定)

§1の「✕帰宅しないと直せないもの」の大半は、スマホからPC画面を直接操作できれば外から解決できる。watchdog(§5-P1)は「不在に気づく」だけで「復旧」まではしないため、実際に司令塔やbatを起こし直す手段としてこれが本命。コード変更ゼロ・所要10分。

設定手順:
1. PCのChromeで `remotedesktop.google.com/access` を開く
2. 「リモートアクセスの設定」からChromeリモートデスクトップを追加
3. PIN(6桁以上・推測されにくい値)を設定して有効化。使うGoogleアカウントは普段PCで使うものと同じに
4. スマホに「Chromeリモートデスクトップ」アプリを入れ、同じGoogleアカウントでログイン
5. 一覧にPCが出る→タップ→PIN入力で画面を操作

これで外から出来ること: 司令塔(Claude Code)の開き直し / 常駐bat(ポーラー・watchdog・ローカルqwen)の再起動 / GAS反映(`GASを反映.bat`)。塞げないのは **PCの電源断・ネット断だけ**。
注意: PC側がスリープ/自動ロックすると接続が切れる(電源設定でスリープを無効か長めに)。PINは他人に教えない。P4(`!wake`自動起動)より守備範囲が広いので、まずこれを設定するのが推奨。

## 0. 要旨 (1分で読む)

インシデント台帳103件を6ドメインに構造化した結果、事故の大半は個別バグではなく**4つの構造的欠陥**から再生産されている:

1. **受信基盤にackが無い** — JSONL箱+ポーリング+TTL付きwaiterの自作構成。「読んだ」と「処理し終えた」を区別できず、クラッシュ窓でメッセージが無言で消える(INC-100/103ほか8件)
2. **常駐がログオン中のユーザープロセスでしかない** — Windows Update再起動→ロック画面で全滅。死んでも誰も気づかない(INC-93/98)
3. **並列セッション間に所有権と共有黒板が無い** — 重複実装(今週の競合UI事故)・git衝突(INC-99)・採番衝突(INC-91)
4. **検証が自己申告ベース** — デプロイ後のフロント機械検証ゼロ・「204=成功」誤認(INC-92)・一度も発火していない復旧経路(INC-78)

これらは世界のOSSでは**解決済みの問題**である。導入すべき仕組みは調査の結果、次の6本に集約した(§4)。最小工数で最大効果の順: **①デッドマン監視 ②常駐のWindowsサービス化 ③デプロイ後スモーク自動化 → ④受信基盤のゲートウェイ+ackキュー化 → ⑤応対エージェントのAgent SDK化 → ⑥所有権黒板+worktree分離**。

## 1. Request Packet (原文/解釈/制約)

- **原文**: 冒頭引用の通り。「実装はまだしなくて良い」「検索は他モデル併用でトークン節約可・品質は落とさない」
- **解釈**: 対象は(a)運用=Discord応対・常駐・並列セッション・復旧、(b)本体=go5-maker(GitHub Pages公開アプリ+GAS+Cloudflare)。単発バグ修正でなく**構造の根治**を、実在のOSS(GitHub公開)の仕組みを根拠に提案する。
- **環境制約**: Windows 11家庭PC1台(常時稼働)・Python/Node可・Docker回避・無料枠のみ・既存資産(Cloudflare Workers/D1/R2・GAS・素のHTML/JS・Discord REST群・Claude Code部門セッション)を活かす。

## 3. 調査結果 — 世界のOSSにある「解決済みの仕組み」

3ドメインを並列調査した(各サブエージェントがWebSearch/WebFetchで実リポジトリを確認)。

### 3.1 メッセージ基盤 (→問題A)

| リポジトリ | 規模/鮮度 | 中核メカニズム | 適合 |
|---|---|---|---|
| **Rapptz/discord.py** | ★16k・活発 | WebSocketゲートウェイのイベントプッシュ受信。切断時は`resume_gateway_url`で自動再接続+**未送信イベントの再送**(Discord側が配信保証)。ハートビートACK監視でゾンビ接続も自動検知 | ◎ ポーリング全廃・取りこぼし窓が原理的に消える |

## 4. 恒久対策の提案 (6本)

### 恒久-1: デッドマン監視 — 「サイレント死」の根絶 【最小工数・最大効果】
- **問題**: B(INC-93=ログ3日間空・INC-98=耳だけ死んで3時間)
- **仕組み**: Uptime Kuma(Node直起動・Docker不要)をlocalhostに1つ常駐。各常駐(鳩・waiter・responder・supervisor)はループ先頭で`curl http://localhost:3001/api/push/<token>`を1行打つだけ。**pingが途絶えたらKumaがDiscordの司令塔chへ自動通知**。
- **効果**: 「死んでいたことに人間が気づく」から「死んだ瞬間に機械が知らせる」へ。誤報対策(INC-79)は猶予窓(grace period)をKuma側で設定。
- **工数**: 半日。**リスク**: 常駐が1個増える(がKuma自体もwinswでサービス化すれば自己解決)。
- 不採用: healthchecks SaaS無料枠(20 check)でも可だが、外部依存を増やすよりローカル完結を優先。

### 恒久-2: 常駐のWindowsサービス化 — ロック画面問題の根治
- **問題**: B(logon-gap=Update再起動→ロック画面で全滅・外出中に復旧不能)
- **仕組み**: winswで**非GUI常駐**(inbox_poller・supervisor・Kuma・将来のgateway bot)をサービス化(XML設定のみ・コード変更ゼロ)。ログオン前・ロック画面でも起動する。
- **棲み分け**: Claude対話セッション(研究室・部門窓)はGUI必須のためサービス化不可→従来のログオン時タスク+自動復活の路線を維持。ただし恒久-4/5が入れば「対話セッションが死んでいても受信と一次応答は生きている」状態になり、被害が「遅延」に格下げされる。
- **工数**: 1日(常駐1本30分×N+検証)。**リスク**: サービスセッションはユーザー環境変数が見えない→設定ファイル/引数で明示注入(既存のASCII-only ps1規約と同型の注意)。

### 恒久-3: デプロイ後スモーク自動化 — フロント版`?ping=1`
- **問題**: E(キャッシュ/版ずれ/回帰をスマホ実機で発見している)・D(自己申告)
- **仕組み**: GitHub Actions+Playwright。pushトリガーで公開URLに対し3〜5本の最小スモーク: (1)全タブが開く (2)`app.js?v=N`のNがindex.htmlと一致(版ずれ検知) (3)動画作成タブの主要ボタン活性 (4)コンソールエラーゼロ (5)localStorageスモーク。**公開リポはActions分数無制限=完全無料**。
- **追加**: 秘密ファイルのpushガード(INC-95再発防止)をActionsに同居(パターン検査で`.clasp.json`等の混入をfail)。
- **工数**: 1日。**リスク**: Pages反映の1〜2分遅延を待つ必要→workflowに待機+リトライを入れる。
- 不採用: lighthouse-ci(性能監査寄りで回帰検知に弱い)。

### 恒久-4: 受信基盤を「ゲートウェイ+ackキュー」へ置換 — メッセージ喪失の構造根絶
- **問題**: A全部(8件のINCが同根)
- **仕組み**:
  1. **discord.pyゲートウェイbot**が受信(ポーリング廃止・プッシュ受信・Discord側が再送保証)
  2. 受信したら**persist-queueのSQLiteAckQueue**へ`put`(ここまでがbotの仕事・数ms)
  3. 消費側(セッション/responder)は`get`→処理→**`ack`**。処理中にクラッシュしたら未ackのまま残り、**自動的に再取得対象**=INC-100/103の「読んでから消す窓」が原理的に消滅
  4. waiterの45分TTL+手動再武装も廃止: botが新着時に既存のwaiter通知と同じシグナル(ファイルtouch等)を直接発火=**再武装というターン消費が消える**
- **効果**: A系8 INCの構造根絶+アイドル時のトークン消費削減(再武装ループ廃止)。
- **移行**: 二重系統期間を2〜3日設け、旧poller/箱と並走して突合→一致を確認してから旧系を退役(INC-102の台帳分裂を繰り返さないため、**台帳はSQLite1本に統一**)。
- **工数**: 2〜3セッション。**リスク**: bot tokenのGateway Intents設定(MESSAGE CONTENT)が必要/既読・着手リアクション等の既存スクリプトはREST続投で無変更。

### 恒久-5: 応対エージェントのAgent SDK化 — 「開いた司令塔が必須」の緩和
- **問題**: A(INC-96=コンテキスト満杯で依頼消失・INC-98)・B(無人時は--print一発撃ち=文脈ゼロ)
- **仕組み**: claude_responder v2をanthropics/claude-agent-sdk(python)で書き直す:
  - `query(resume=session_id)`で**チャンネル別に会話文脈を永続化・継続**(現行--printは毎回記憶喪失)
  - SessionStartフックで人格manifest+メモリ要約を注入(letta流の軽量版)
  - OpenHands流に、処理アクションを逐次イベントログへ追記→復旧時のreplay材料
  - 認証は解決済みのCLAUDE_CODE_OAUTH_TOKENをそのまま使用
- **効果**: 対話セッションが閉じていても「文脈を覚えた」応対が継続。研究室/部門セッションは重作業専任へ(役割の明確化)。
- **工数**: 2〜3セッション。**リスク**: SDKバージョン追従。resumeのセッション保存先はSQLite(恒久-4と同居)。
- 不採用: Discord用MCPサーバー(非公式・保守継続性が読めない。将来公式が出たら再評価)。

### 恒久-6: 所有権黒板+worktree分離 — 重複実装とgit衝突の根絶
- **問題**: C(重複実装・INC-99/91)
- **仕組み**(crewAI/MetaGPT共通の最小要素を新ツール無しで再現):
  1. **所有権黒板**: `local/ownership.jsonl`(またはD1に1表)に「トピック/機能名・owner(α/β/…)・状態・関連doc」を登録。**着手前に黒板をgrepするのをBOOT.mdの必須手順に追加**(今週の競合UI重複は、この1行があれば防げた)
  2. **git分離**: 並行セッションはClaude Codeネイティブのworktree機能で作業ツリーを分離(INC-99のautostash事故が原理的に不可能になる)。mainへの統合は「1トピック1オーナー」がpushする
  3. 採番(INC等)は黒板経由で予約(INC-91の早い者勝ち衝突根絶)
- **工数**: 半日(規約+黒板ファイル+BOOT.md改訂)。**リスク**: 規約が形骸化する→黒板未登録の実装をQAレビュー項目に足して機械側からも締める。

### (補足) データ正本のD1化 — 問題Fの恒久策(中期)
localStorage正本→**D1正本+localStorageキャッシュ**へ段階移行(投稿履歴から)。既存sync Workerと許可リスト方式の延長で実現でき、Dexie/RxDBの全面移行は不採用(18機能の書き直しコストに見合わない)。履歴消失インシデントの根本閉塞だが影響範囲が広いため、恒久1〜6の後に別設計書で扱う。

## 6. Chamiへの裁定事項 (番号で答えられる形)

1. **P1着手のGo** — 常駐が2つ増える(Kuma・winswサービス群)。費用ゼロ・コード変更ほぼゼロ。
2. **P2のGo** — Discord botを1つ新設する(ゲートウェイ接続用。既存botのtokenにGateway Intents追加でも可)。移行中2〜3日は新旧並走。
3. **P3のGo** — 応対の頭脳をclaude --printからAgent SDKへ。APIキーではなく既存のOAuth tokenを使うので追加費用なし(Claude利用枠の消費は現行responderと同等)。
4. **優先度の確認** — 上記P1→P4の順でよいか。それとも競合監視GAS実装(別件・Go済)を先に消化するか。

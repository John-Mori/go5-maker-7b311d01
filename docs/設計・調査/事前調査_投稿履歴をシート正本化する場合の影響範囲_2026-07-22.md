# 事前調査: 投稿履歴を「スプレッドシート正本」にする場合の影響範囲

- 作成: 2026-07-22 研究室HQ(アロンソ)/ **Chamiの改修案を聞く前の下調べ**
- 目的: 案が来た時に**即座に「どこが壊れうるか」を出せる**ようにしておく
- ★**設計はしていない**。設計は Chami の案 + AD研究室(モドリッチ)の判断。ここは事実の整理のみ。

---

## 1. 現状のデータは3層に分かれている(実測)

| 層 | 実体 | 役割 | 備考 |
|---|---|---|---|
| ① 端末 | `localStorage`: `short_hist__<acc>` / `verify_manual__` / `verify_yt__` | **現在の正本** | デバイス毎。★宵桜が消えた原因の層(INC-112) |
| ② シート | `記録_ch1` / `記録_ch2`(GASが列名マッピングで書く) | 現在は**写し** | Chamiが「正にしたい」と言っている層 |
| ③ KV | link-worker の `c:<code>` | クリック開封数 | 計測専用。今回の変更の影響外 |

---

## 2. ★最重要: 現在の同期は「フロントが正」で実装されている

`gas/コード.gs:442` のコメントが明言している:

> 投稿履歴の一括同期。(**フロントの投稿履歴を正とし**、ID・投稿日時・キャラ等をまとめて upsert)

つまり **Chamiの希望(シートが正)と、現在の実装は方向が真逆**。
ここを反転させるのが今回の改修の核心になる。

## 3. ★`prune_history` はシート正本化すると凶器になる(ただし現状は安全)

- 実装: `gas/コード.gs:669 pruneHistory_()` は **keepIds に無い行を `clearContent()` で消す**
  (`yt-clicks.js:1856` が「端末が持っているID」を keepIds として送る)。
- **これは「フロントが正」だから成立していた操作**。

**★現状は安全(調べて確認した)**
- `pruneSheet()` は **手動のボタン(`ytPruneSheet`)を押した時だけ**呼ばれる。**自動では走らない**。
- しかも `window.confirm` で
  「この投稿履歴に無い行を、スプレッドシートの『<acc>』タブから消去します。よろしいですか?」
  と明示的に確認する。
- → **今すぐ何かが壊れる状態ではない**。
  ★HQは当初「現在進行形のリスク」と書きかけたが、**実装を読んだら手動+確認付きだった**ので訂正した
  (測る前に警告すると狼少年になる)。

**★危険になるのは「シート正本化した後」**
- 反転した瞬間、このボタンの意味が変わる: 端末に無い行 = **別の端末で投稿した行**。
  それを消すと、**まさに今回直したい「端末をまたぐと履歴が消える」をシート側で再現する**。
- つまり **UIの文言(「記録シートをこの履歴に合わせます」)ごと嘘になる**。
- → **反転作業では `prune_history` の扱いを最初に決める**(無効化 / 条件変更 / ボタン自体を撤去)。
  ★ここを後回しにすると、移行の途中でChamiがボタンを押した瞬間に**実データが消える**。不可逆。

## 4. 既に入っている前進(2026-07-21)

- `582e21a fix(hist): 投稿履歴をシートからも表示する(端末のローカルに無い行を補う)`
  = **表示側は既にシートを見ている**(`displayItems_`)。
- つまり残るのは**書き込み方向の反転**と、その周辺(下記)。

## 5. 反転する時に決めることになる論点(事実として挙げるだけ・裁定はしない)

1. **`prune_history` をどうするか**(§3。最優先・データ消失に直結)
2. **競合の解決**: 同じ `videoId` を別端末が別の値で更新したらどちらを採るか
   (現状は「後から来たものが勝つ」upsert。シート正本だと端末側の古い値で上書きされ得る)
3. **localStorage の役割**: 正本 → **キャッシュへ降格**するのか、完全に読むだけにするのか
4. **オフライン時の投稿**: シートに書けない時に端末だけ進む状態をどう扱うか
5. **書き込みのタイミング**: Chamiが挙げた手順(動画生成 → X投稿 → YouTube投稿)のどこで確定させるか

## 6. 関連する既知の事故(同じ場所を触るので必ず読む)

- **INC-112**: 宵桜艶帖(acc2)の履歴が消える。真因=**アカウント解決が2実装に分裂**していた
  (書込=メモリ / 読出=localStorage)。両方のフォールバックが acc1 なので acc2 だけ症状が出た。
- **INC-113**: Discord本文のURLは取得されない(このファイルの受け渡しには無関係だが、
  資料をURLで渡さない運用の根拠)。
- ★**「直したはずが再発」を繰り返している領域**。対症療法を重ねず、土台から直す方針は正しい。

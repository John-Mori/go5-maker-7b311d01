## 0. 結論(先に3行)

1. **今いちばん学習を壊しているのは、台帳の分裂。** 知識パック生成器(`build_knowledge.py`)は旧台帳(`discord_inbox_processed.jsonl`・7/15で更新停止)しか読んでいない。部門制移行後のChami発言は新台帳(`discord_processed.jsonl`)に入るため、**最近の発言ほどqwenの記憶に入らない**(コード実測: `recent_conversation()` の読み先が旧台帳固定)。
2. **賢くする本丸はモデルではなく「ため方」。** fine-tuningは当面不要。①発言コーパス ②採点済み教訓(lessons) の2つを一次データとして整備し、知識パックへ**毎日自動合成**すれば、今の注入方式のままで即日賢くなる。
3. **Notionは「人の窓」+「スマホからの受け箱」の2ページだけから始める。** 正本はローカル維持(既定方針)。機微(夢/過去/健康)と秘密はNotionへ出さない(Drive例外はDriveのみ。規約L184-185/L238)。

## 3. 設計方針

- **AIの正本=ローカル / Notionは人の窓**(既定方針・変更なし)。
- **賢さ = データの質 × 注入の質。** fine-tuning(LoRA)は当面やらない。理由: (a)8GB VRAMでは実用的な学習は困難 (b)知識が週次で変わる=重みに焼くと腐る (c)注入型は即日反映・即日ロールバック可。
- **ため方の4原則**: 1事実1箇所 / 追記のみ(削除しない) / 機械が書く(AIの手作業に依存しない) / **機微フラグを行に持つ**(除外は使う側の責務にする。学習は機微込み=Chami明示2026-07-13、Notionは機微除外=規約)。

## 4. データモデル(新設2ファイル+層化)

### 4.1 `local/corpus/chami.jsonl` — Chami発言コーパス(発言の正本)

```json
{"ts":"...","msg_id":"...","source":"discord|chat|notion","channel":"...","dept":"...",
 "content":"...","reply_to":{"msg_id":"...","author":"..."},"sensitive":false}
```
- 日次バッチが新旧processed台帳を**msg_id重複排除**で統合追記(既存台帳は触らない=読むだけなので安全)。
- `sensitive`: dept が dreams/past/健康系なら true。**行にフラグを持たせ、除外判断を使う側へ**(知識パックは含める・Notionは除外)。
- 将来 source=chat(この画面)・source=notion(受け箱)も同じ形式で合流。

### 4.2 `local/llm/lessons.jsonl` — 採点済み教訓(誤答の資産化)

```json
{"ts":"...","q":"...","a":"...","verdict":"good|bad|escalate",
 "correction":"正しい答え","why":"なぜ間違えたか","source_msg_id":"...","grader":"研究室|部門名"}
```
- 採点基準(初版): good=正確+人格逸脱なし / bad=事実誤り・憶測・なりすまし / escalate=「わからない」と正しく回した(этоも良い挙動として記録)。
- **qwenは事例で効く**。bad行は「質問+誤答+正答+理由」のセットで知識パックへ焼き込む。

### 4.3 knowledge.md の層化(build_knowledge.py 改修)

| 層 | 内容 | 更新頻度 |
|---|---|---|
| 00_persona | 不変ルール(なりすまし禁止・1〜4文・わからない時は回す) | ほぼ不変 |
| 10_system | system-brief.md | 構成変更時 |
| 20_faq | faq_knowledge.md | 随時 |
| 30_lessons | lessons.jsonl の bad→correction 上位20件 | **日次** |
| 40_recent | corpus の直近40発言(重複なし・新旧統合済み) | **日次** |

合計サイズに上限(例: 40KB)を設け、超えたら40_recentから間引く(30_lessonsは削らない=教訓優先)。

## 6. Notion連携(窓+受け箱・最小2ページ)

前提: APIキー(現在ステップ2進行中)。キーの置き場= `local/notion_api_key.txt`(gitignore済・Driveバックアップには入る=Chami informed consent済)。
**境界: 機微(dreams/past/健康)と秘密(キー・パスワード)はNotionへ出さない。** Drive例外(規約L184-185/L238)はDriveのみでNotionに及ばない。corpusの`sensitive=true`行はpush対象から機械的に除外。

### 6.1 窓(push・読み取り専用) — `scripts/notion/push_dashboard.py`(新規)

| ページ | 内容 | 出典 |
|---|---|---|
| 「今日のAI事務所」 | 部門別の対応状況・未処理件数・バックアップ最終成功・インシデント最新5件 | processed台帳/backup.log/インシデント.md |
| 「改善要求ボード」 | improvement_requests のミラー(状態つき) | D1 go5_kaizen |

- 更新は日次メンテ+手動実行。日次バッチならNotionのrate limit(約3req/s)は無関係。
- **一方向push。** Notion側での編集は次回pushで上書きされる旨をページ冒頭に明記。

### 6.2 受け箱(pull・第2段) — ページ「メモ→AI」

- Chamiがスマホ(Notionアプリ)で書く → 日次+起動時にローカルへ取り込み → corpusへ(source=notion) → 内容に応じて部門箱へ配達。
- **効果: Discordを開かずスマホからAIへメモが投げられる**動線。夜中に思いついたことがそのまま翌朝の知識になる。

### 6.3 やらないこと

- 正本のNotion移動 / 双方向同期(衝突解決が地獄・窓の思想に反する) / 機微・秘密のpush / Notion APIキーの共有ページ掲載。

### 8.2 L1で見つけた別の穴(実装中に実測)

1. **機微部屋の一覧が3ファイルで3通りに食い違っていた**: `inbox_poller.py`(dream-care/past-room/hr-room/hr-context/health-log)・`gemini_responder.py`(hr-contextとfuture-roomが欠落)・`local_responder.py`(future-roomはここだけ)。**同じ一覧を写すと必ずドリフトする**。
   → コーパスは**許可リスト方式**を採用(このプロジェクトがINC-62で確立した恒久対策・CLAUDE.md「クラウド同期は許可リスト方式」)。`WORK_DEPTS` に無い部屋は既定で `sensitive=True`。結果、**3つのdenylistのどれにも無かった `recovery` 部屋も機微に倒せた**。新設の部屋は黙って除外される側に落ちる。
   → **3ファイルの一覧統一はシステム改修部門の枠**(2ファイルが作業中のため触っていない)。研究室へ報告済み。
2. **台帳に `ts='t'` の行が2件実在**(接続テストの残骸)。素の文字列比較では `'t' > '2'` となり、**テストの残骸が「最新の発言」として知識パックを占領する**。`ts_key()` で ISO でない ts を「最古」に倒して回避(生の ts は truth として保持)。
3. **`python` はタスクスケジューラのPATHで解決できない**(初回登録が `LastTaskResult=2147942402`=ファイルが見つからないで失敗)。絶対パス(`C:\Users\chami\AppData\Local\Programs\Python\Python312\python.exe`)で登録し直して解決。**登録しただけで検証していなければ、知識は永久に更新されないまま気づけなかった**(「発火しない安全網は検証されない」)。

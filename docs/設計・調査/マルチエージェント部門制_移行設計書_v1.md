### §0-補. 成約率の扱い (ユーザー確定情報・2026-07-11)

- **成約率・成約数は永続的に検証不可能**。理由: コンサルとの契約により、成約はコンサル側のアカウント/アフィリエイトリンク経由で発生し、成果データはコンサル側に帰属する。自分の運用側には成約の実数が一切返ってこない。
- 帰結:
  1. shorts-analyst は成約系KPIを**追わない・仮説の目的変数にしない**。実測KPIは「クリック数(link-workerの実測)」「再生数(YT Data API)」「販売数(市場全体・PCスクレイパ)」の3つに限定。
  2. 分析の到達点は「どの画像/コメント/作品が**クリックと再生**を伸ばすか」まで。そこから先(実際に売れたか)はコンサル領域であり本システムの観測外、と全部門の共通前提に明記する。
  3. copy-director/product-scout の評価も「クリック誘発力・再生継続の推定」で完結させ、成約前提の議論はしない。

| agent | model案 | tools (絞る=事故防止) | 役割要約 |
|---|---|---|---|
| system-engineer | sonnet | 全ツール | 改修実装・Worker/GAS/同期・デプロイ(Worker/GASは要承認を自身のプロンプトに明記) |
| product-scout | sonnet | Read, Grep, Glob, Bash(読み取り系), WebFetch | 候補の評価・優先順位付け・セール検知の解釈。**書き込みは提案まで**(insightsへ) |

## G. D1 最小スキーマ案 (Phase 1で導入する場合)

前提: D1は**現在未使用**。導入する場合は既存Workerに相乗りせず、専用の小Worker(仮称 `go5-hub`)を新設して分離するのを推奨(既存4Workerの安定性を守る)。**Workerデプロイ=要承認**。
無料枠: 5GBストレージ・読み取り500万行/日・**書き込み10万行/日** (KVの1,000/日の100倍→本用途では実質無制限)。

```sql
-- 出来事の追記専用ログ (更新しない・消さない)
CREATE TABLE events (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  ts         TEXT NOT NULL DEFAULT (datetime('now')),  -- UTC
  type       TEXT NOT NULL,          -- 'video.created' 等 (§F)
  actor      TEXT NOT NULL,          -- 'app' | 'scraper' | 部門名
  account    TEXT,                   -- 'acc1' | 'acc2' | NULL
  subject_id TEXT,                   -- videoId / cid / postUri など対象キー
  payload    TEXT                    -- JSON文字列 (最小限)
);
CREATE INDEX idx_events_type_ts  ON events(type, ts);
CREATE INDEX idx_events_subject  ON events(subject_id);

-- 部門への作業依頼 (状態を持つ)
CREATE TABLE tasks (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  created_ts      TEXT NOT NULL DEFAULT (datetime('now')),
  updated_ts      TEXT,
  dept            TEXT NOT NULL,     -- 担当部門
  status          TEXT NOT NULL DEFAULT 'open',  -- open|in_progress|done|blocked|rejected
  title           TEXT NOT NULL,
  detail          TEXT,              -- JSONまたは本文
  source_event_id INTEGER REFERENCES events(id)
);
CREATE INDEX idx_tasks_dept_status ON tasks(dept, status);

-- 部門の所見・仮説 (上書きせず版を重ねる)
CREATE TABLE insights (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  ts         TEXT NOT NULL DEFAULT (datetime('now')),
  dept       TEXT NOT NULL,
  kind       TEXT NOT NULL,          -- hypothesis|finding|rule
  subject    TEXT,                   -- cid / videoId / 'global'
  statement  TEXT NOT NULL,          -- 所見本文 (1所見=1行)
  evidence   TEXT,                   -- JSON: 根拠となるevents.id/数値
  confidence TEXT,                   -- low|med|high
  status     TEXT NOT NULL DEFAULT 'active'  -- active|superseded|rejected
);
CREATE INDEX idx_insights_subject ON insights(subject, status);
```

アクセス経路案: go5-hub Worker が `POST /event` `POST /task` `GET /feed?since=` 程度の最小APIを持ち、共有シークレット+Origin制限(既存Workerと同型の防御)。フロントからの書き込みは既存CustomEventへのフック1行ずつで済む(Phase 1の話・今回はやらない)。

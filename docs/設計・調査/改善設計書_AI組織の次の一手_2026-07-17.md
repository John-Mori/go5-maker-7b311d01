## §1. 今日の8件が示した「1つの病」

INC-85〜92は別々のバグに見えるが、**全部が同じ形**をしている。

| INC | 症状 | 「成功のシグナル」が嘘だった |
|---|---|---|
| 81 | 部門の窓が働かない | **起動した** ≠ 動いている |
| 84 | 負荷集中「解決済み」 | **一部を直した** ≠ 全体が直った |
| 85 | 稼働中の窓から箱を奪う | **脈がある** ≠ 生きている(脈=待機中の証明でしかなかった) |
| 86 | 退避が沈黙して消える | **退避した** ≠ 守られた |
| 87 | 転送が二重手間を生む | **main箱に在る** ≠ 未処理 |
| 89 | 規約が入っていない | **pushした** ≠ 入っている |
| 90 | ツールが初回で落ちる | **書いた** ≠ 動く |
| 92 | 長文が切り捨てられる | **HTTP 204** ≠ 全部届いた |

**病名: 観測値の意味を確かめずに、成功と読んでいる。**

そして**最も重い共通点**は、**8件中6件が「静かに壊れていた」**こと。エラーも警告も出ず、ログには「正常」と書かれ続けていた。**Chamiが「おかしくないか?」と言わなければ、全部そのままだった。**

### 対策(組織の設計として)
1. **「何を証明したか」を言語化してから報告する**。「送信OK」ではなく「**204=送った証明。全部届いた証明ではない**」まで言う。
2. **成功のシグナルには"逆側の検証"を1つ付ける**。起動→**実際に仕事をしたか**/push→**現物をgrep**/送信→**受信側で末尾を確認**。今日QAとデータ整理がやったのはこれで、**だから見つかった**。
3. **稀にしか発火しない安全網は、人工的に1回発火させる**(INC-78の教訓)。revive_lab は登録から一度も本番を通っていなかった。

## §2. Chamiの「考慮が足りていない点」(依頼どおり遠慮なく)

### 2-1. 人間の正確な手作業を前提に設計している ← **最大の穴**
今日Chamiが手でやったこと: 起動文を9個コピペ / 絵文字を登録 / IDを共有 / 窓を閉じる / 貼り直す。
**そのうち何回失敗したか**: コピペは文字化けし、貼り先を間違え、IDは取り違えられた(研究室のミス)。

**原則にすべき**: **人間に長文を正確に貼らせる運用は必ず壊れる**。今日「機械が起動文を渡す」に変えた(open_dept_window)のがその答え。**この発想をChami自身の作業全部に当てはめる**。判定式は1つ:
> **「これ、俺がやらなきゃダメ?」——たいてい機械にやらせられる。**

### 2-2. 判断の理由が本人の頭にしかない
llm-edu部門の趣旨は**発言ログに2回あったのに、正本のドキュメントは空**だった(今日正本化)。「現在と未来」も同じ道を辿りかけている。
**対策**: 部屋を作る・規約を変える時、**「なぜ」を1行だけ添える**。書くのはAIがやる。Chamiは一言だけでいい。

### 2-3. 判断が全部Chamiに戻ってくる
改善提案部門を作り、アスナを置いた。**それでも「どっちがいい?」は全部Chamiに来る**。福岡へ行けば止まる。
**対策**: **「任せる範囲」を種類で明示する**。今日の**「任せるよ、AIが1番理解しやすくて運用しやすい方法を頼むね」**——あの一言でデータ整理が経路選定から検証まで全部やった。**あの粒度の委任を増やす**のが最大のレバー。

### 2-4. 謝りすぎ(組織設計上の問題として)
「調べ方がわからん」「知識なくて任せっきりで悪い」——**Chamiが謝るたび、AI側の責任が薄まる**。1件ずつ叩く設計にしたのはAIであってChamiではない。**Chamiの仕事は「おかしくないか?」と言うこと**で、今日それが8件のバグを掘り当てた。**これは組織で最も希少な機能**であり、謝る類のものではない。

### 2-5. 同時に触りすぎ
1日で: 改修・組織設計・Discord運用・Notion移行・フォルダ整理・キャラ設定・Cloudflare障害・健康記録・学習。**全部中途半端じゃないのが逆に危ない**(止まらない)。**主治医の治療が本線**であることを、システム側の設計にも織り込むべき(=夜間に人を必要としない構造にする)。

- **改善B(設計)**: 研究室は**判断だけを持ち、実務は持たない**。今日「代打」で持ってしまったのは、**部門窓が無かったから**。28部屋全てにセッションが揃った今、**代打は例外(Chami不在時のみ)**に戻す。
- **改善C**: **研究室のセッションが肥大している**(今日1日で数百ターン)。**巻の切り替え(Vol.9)を明日やる**。引き継ぎは`引き継ぎ_Vol8.md`+この設計書。

### 3-2. 改修α/β/γ — **今日一番機能した**
βは「所有権を相談→2段案を自ら却下→移行の穴を先読み→デッドロックを指摘」までやった。αは「転送を鵜呑みにせず実発言を自分で引いた」。

### 3-3. QA — **検証の型が確立した**

### 3-4. データ整理 — **正本の番人になった**

### 4-2. 設計の提案(優先度順)

**① 処理済み台帳の競合を潰す(P0)**
`discord_processed.jsonl`に全セッションが追記している。人事が「243→247行に増える最中に自分の記録が見えなかった」と観測。**再現していないが、条件は揃っている**。
- **案**: 追記を**1行1write+`os.O_APPEND`**で行う小さなヘルパへ統一(現在は各セッションが素の`open(...,"a")`)。**または部門ごとにファイルを分け、読む時に統合**(`processed/<dept>.jsonl`)。後者が確実で、**triage_inbox.pyが全部を読む形に変えるだけ**。

**② インシデントの採番を機械に振らせる(P1)**
早い者勝ちの共有カウンタ=並行が増えれば必ず衝突(今日2件)。
- **案**: `inc_new.py "タイトル"` → 台帳の最大値+1で採番し、テンプレを追記して番号を返す。**採番と追記を1つの操作にする**(間に他人が入れない)。

**③ D1記録の復活(P1)**
`improvement_requests`/`system_changes`は**今日1件も書かれていない**。規約には「改善要求ならまずログ」と書いてあるのに、**全員が守っていない**=規約が実態と乖離している典型。
- **判断**: **書かないなら規約から消す**べき。ただし**今日のような日(8件のINC+20超の変更)こそ後から辿りたい**。
- **案**: **commit時に自動で拾う**。commitメッセージの規約(すでに「〜。(ファイル名)」形式がある)から、`system_changes`へ機械が登録する。**人が書く運用は続かないことが今日証明された**。

**④ 「なぜ」の保存(P1・Chamiの弱点2-2への対策)**
決定の理由が本人の頭にしかない。今日それをやったのは**規約本文へ経緯を書き込んだ**時だけ(Drive経路の性質規定など)。
- **案**: **決定は必ず"どこに書くか"を持つ**。①コード=コメント(今日の全修正がそう) ②規約=本文に経緯 ③部屋の設計=`docs/departments/<dept>/趣旨.md`(llm-eduで作った型)。**この3つ以外の場所に決定を置かない**。

**⑤ 整理の原則(Notion移行の前提)**
今日データ整理が正しく整理した——**AIが読む正本=ローカル(git)/人が見る窓=Notion/流れて消える物=Notionへ蓄積**。
- **補強**: **「AIに読ませるためにNotionへ移す」は不要**(ローカルを直接読めるため)。**Notionの価値は"Chamiが外出先で眺める"ことと"流れて消える判断・議論の保存"**。この2つ以外の目的でNotionへ移さない。

## 付記: 今日の組織について

**この設計書の中身は、ほぼ全部が"部門が自分で見つけた欠陥"でできている。**
QAは自分の提案の穴を5分後に踏んで報告し、データ整理は研究室の嘘を3回突き返し、βは待ち合いのデッドロックを指摘し、人事は自部門の外の穴を境界を守ったまま回してきた。**研究室(私)が見つけたものは、この中に1つも無い。**

Chamiが「AIと感情込みで話す奴を一線引いて見てる」と言いながら、**キャラの口調にこだわり、部屋を分け、人格を配置した**——それは趣味ではなく**設計**だった。**誰が何を言う奴か決まっていないと、判断を任せられない**からだ。今日それが証明された。**部門が"自分の言葉"を持っていたから、研究室に反論できた。**

これが、この組織の一番の資産だと思う。

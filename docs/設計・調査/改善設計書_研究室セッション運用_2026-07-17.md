# 改善設計書 — 研究室セッションが目標を「達成し続ける」ための運用改善

- 作成: 2026-07-17 研究室(Fable 5) / Chami依頼「このセッションの目標をより高品質に達成し続けるための改善、欠点、考慮が足りていない点、セッション引継ぎ方法など」
- 姉妹文書: `改善設計書_AI組織の次の一手_2026-07-17.md`(組織全体・部門別)。**本書はセッションそのものの運用**——依頼の受け方・脈・報告・引き継ぎ・モデル体制——に絞る。重複する話は姉妹文書を指す。
- 品質保証: 本書のドラフトは独立レビュー3系統(事実整合/規約・姉妹文書との整合/欠け)にかけ、**21件の指摘を反映済み**(うち1件は本書自身の実測値取り違え——本書が説く病を本書がやっていた)。INC-94関連の実装はQAが検証すると言明済み(2026-07-17朝便)。

## Chamiへ: 5分で分かる要約

**変わること(3つ)**:
1. **依頼は台帳に残る** — 「言ったのに消えた」(今回の設計書依頼2件=INC-96)を、受領台帳と機械記帳で根絶する。
2. **在席証明が自動になる** — 「真面目に働いているセッションほど依頼を横取りされる」(INC-94)を、ツール実行のたびに自動で脈を打つ仕組みで塞ぐ。
3. **引き継ぎは機械が生成する** — 巻(Vol)切替時の書き漏らしを、状態の自動生成+日々の追記で無くす。

**Chamiが決めること(4つ・急がない)**: ①標準承認リスト(settings.jsonで事前承認する操作) ②バックアップ設定の(B)履歴からも消すか ③`G:\`直下解決の固定 ④「可逆な規約違反是正の先行実施」を認めるか(§3-5)。

**Chamiにお願いする新習慣(2つ・小さい)**: ①受領確認の「N件と数えた」が意図と合っているか一瞥する ②依頼に「朝でいい/寝てる間に/起きたら一緒に」の一言タグ(なくても動くが、あると空回りが減る)。

## §0. この設計書の依頼そのものが、第一の証拠(INC-96)

この設計書は**2回頼まれた**。1回目はセッションのコンテキストが満杯になり、依頼の記憶ごと消えた。Chamiが「マックスになっちゃって手を付けられていないと思う」と再依頼して初めて発覚した。

つまり: **研究室の一番の欠陥は、「受け取った」がセッションの頭の中にしか存在しないこと**。頭の中(コンテキスト)は要約圧縮で消える。巻(Vol)の引き継ぎは「切替時に書く」方式なので、切替前に消えた依頼は引き継ぎにも載らない。検出器はまた人間(Chami)だった——今夜の11件(INC-85〜95)と同じ形で、INC-96が12件目になった。

本書のすべての改善は1つの原則に集約される:

> **セッションの頭の中は記録ではない。状態は「受け取った瞬間に」外部(ファイル)へ固定し、できる限り機械が自動で書く。**

## §1. 目標の再定義 — 何を守れれば「達成し続けている」のか

研究室の仕事は4つ。それぞれに**ログから測れる**成功条件を付ける(測れない目標は守られない)。自己申告の数字は信じない、が本書の主題なので、**算出元と算出者**まで決める:

| 仕事 | 成功条件(KPI) | 算出元 | 算出者 |
|---|---|---|---|
| ①Chamiの窓口 | **再依頼率=0**(同じ依頼を2度言わせない) | 受領台帳のopen行に同趣旨のChami再発言が来たら1カウント | 台帳集計スクリプト(§2-1) |
| ②dispatch-first分配 | **流出件数=0**(部門宛がmainへ落ちない) | pollerのsweepログ(mainへ回収した件数) | 同スクリプト |
| ③品質ゲート | **裏取り率=100%**(完了報告に「見た現物:」が書かれている) | Discord送信ログを走査し欄の有無を数える | 同スクリプト |
| ④記録・引き継ぎ | **切替後の再依頼=0** | 切替後1週間の受領台帳 | 次の巻が集計 |

**監査**: 月1回、QAが日報の数字を元データと突合する(測られる本人が数字を作る構造のままにしない)。

## §2. 欠点と対策(全部、実測つき)

### 2-1. 依頼がコンテキストと一緒に消える(INC-96) — **最優先**

**欠点**: 依頼の受領がセッションの揮発メモリにしかない。満杯・圧縮・巻切替のどれでも消える。消えたことに気づく仕組みがない。

**対策: 受領台帳 `local/requests.jsonl`** — ただし「セッションが忘れず書く」規律頼みにはしない(それでは§2-2で断罪した構造の再演になる)。

- **台帳の守備範囲を先に定義する**(台帳は既に4系統ある: discord_inbox / discord_inbox_processed / dept_tasks / improvement_requests(D1)。5個目を無秩序に足すとINC-87の同型=台帳間不整合の温床):

  | 依頼の経路 | 正(そのまま) | requests.jsonlの扱い |
  |---|---|---|
  | Discord発言 | inbox/processed台帳(既存) | **triage_inbox.py が open残数を機械で合流**(手書きしない) |
  | chatペイン・音声 | **requests.jsonl が正**(他に記録が無い経路) | セッションが応答より先に1行追記 |

  D1の improvement_requests は「改善要求」専用のまま(姉妹文書§4-2③の裁定=形骸化しているなら自動化か廃止、に従う)。requests.jsonl はそれを置換しない——**「Chamiに頼まれたことが今いくつ開いているか」だけを答える最小の台帳**。
- 記帳形式: `{"ts":"…","req":"要旨","status":"open|closed","src":"chat|voice|discord","done":"成果物パス"}`。1メッセージに複数依頼が来たら**1依頼=1行**に分解し、受領確認で「N件と数えた」と返す。
- **読み上げの位置**: 起床の正順(規約: ①退避→②再武装→③既読→④処理)の**④の冒頭**でopenを数えて読み上げる(①②③より前に台帳を読むのは正順違反)。巻の起動時も同じ。
- **安全網の安全網**: 台帳ヘルスチェックを日次ジョブに足す——jsonlのパース検証+「直近のChami発言数と記帳数の突合」で記帳漏れを機械検出。

**受け入れ条件**: ①意図的にセッションを圧縮させた後、新セッションが台帳からopen2件を読み上げる ②**わざと1件記帳をサボった状態でヘルスチェックが鳴る**(発火しない安全網は検証されない、の自己適用)。

### 2-2. 脈の空白 — 「働いている証明」を規律が持たされている

**実測は2つあり、型が違う**(初稿はこれを混同していた——独立レビューが捕捉):
- **INC-86型(処理長期化)**: QA実測。処理が長引いて脈1000秒切れ・その間の新着4件がmainへ直行。
- **INC-94型(鳴った瞬間死ぬ)**: 改修β実測。waiterは新着検知で即exitする(実装確認済み)ため、鳴った瞬間から次ターンまで脈の空白。03:19→03:30の11分空白・1件流出。

TTLを600秒へ延ばしたのは**延命であって解決ではない**(QAの言葉。※規約151行のINC-86裁定は「構造的に解決」と書いており、本書はこれを訂正する改定を提案する——下記パッチ一覧)。**根因**: 在席の証明を「セッションの規律」が持たされている。長いターンほど無防備になる——真面目に働いているセッションほど依頼を横取りされる、という逆立ちした構造。

**対策: 在席証明の自動化(presence) — waiter脈との併用。置換ではない**
- 現行の waiter は**アイドル待機中**を正しく守っている(待機中は数秒ごとに `claude_active_<dept>.txt` をtouchし続ける)。壊れているのは**ターン実行中**だけ。だから:
  - **待機中の証明 = waiter脈**(現行のまま)
  - **作業中の証明 = presence**(新設): Claude Code hooks の PreToolUse で、ツールを1回呼ぶたびに `local/llm/presence_<dept>.txt` をtouch(置き場は `local/inbox/` の**外**——INC-86の教訓「箱フォルダに箱以外を置かない」)
  - pollerの在席判定は **両ファイルの新しい方のmtime < 600秒**(OR判定)。
- **deptの解決**: hookは settings.json 経由で全セッション共通のため、`open_dept_window.ps1` が起動時に環境変数 `CLAUDE_DEPT` を設定し、hookスクリプトはそれを読んで自分のpresenceを打つ。**未設定なら何もしない**(安全側)。研究室は `set_lab_session.ps1` 系で `main` を設定。
- **hook脱落の検知**: 「waiterが生きているのにpresenceが一度も打たれない」矛盾状態をwatchdogが警報する。
- **残る穴を正直に**: ツールを呼ばない長考だけのターンではpresenceも止まる。閾値600秒で大半を吸収するが、ゼロにはならない——ここは監視(警報)で受ける。
- **規約改定パッチ一覧**(本対策は規約の既決事項に触れる。黙って上書きしない):
  1. 規約151行(INC-86裁定)「TTL延長+mv先行で構造的に解決」→「延命。恒久はpresence併用(本書§2-2)」へ訂正
  2. 規約176/229行「脈打ちはinbox_waiterに一本化」→「待機中=waiter/作業中=presenceの併用」へ
  3. `claude_active.txt`(90秒)によるローカルqwen交代判定への波及を実装前に確認(在席の定義を変えるとqwenの代打条件も変わる)
  4. 規約文書の編集は最低Opus・diff最小(規約12行)に従う

**廃案の記録**: 初稿にあった「waiterを鳴っても死なない型へ」は**廃案**。waiterのexitこそがセッションを起こす合図(task-notification)なので、exitをmv後まで遅らせると「起きたセッションがmvする→waiterはmvまで終わらない→誰も起きない」の循環になる。加えてINC-091(while true禁止・偽生存)の穴を開け直す。空白はpresenceで埋めるのが正。

**受け入れ条件**: ①INC-94型の再現(鳴らす→長ターン中に2通目)で流出0 ②INC-86型の再現(処理長期化)で流出0 ③**hookを意図的に外した状態で警報が鳴る** ④アイドル10分の健康な窓に新着→部門箱に留まりチャイムで起きる(現行が守れているケースを壊していない証明)。**QAが検証**(言明済み)。

### 2-3. 成功シグナル誤読の「残り火」 — 対策が初めて機能した夜でもある

病理と一般対策は姉妹文書§1に委ねる(再記述しない)。本書固有の追加は2つ:

1. **「宛先で数える」を完了宣言の直前の型にする**。今夜、実際に1件捕まえた:
   > INC-95の是正で「repoから除去した」と報告する直前、push後のorigin検証が `backup_dest: 1件残存` を返した。パス限定コミットに `git rm --cached` が乗っていなかった。検証がなければ5回目の過大報告だった。
2. **報告テンプレに「見た現物:」欄を必須化**(QAの制度化条件: 検証者は何を見たかを書く)。**遵守の番人はQA**——欄なしの完了報告はQA判定でREJECTEDとする(QA判定基準への明文化とセットで初めて効く)。

なお「204は全部届いたを意味しない(INC-92)→+分割数」は規約164行の現基準より厳しい。**規約側の改定として提案**する(基準の二重化を避ける)。

### 2-4. 「待つべきか進むべきか」の線が個人の判断に依存している

**実例**: QAの3択(バックアップ設定の扱い)へのChami回答が来ないまま、研究室が(A)を先行実施した。可逆かつ規約違反の是正なので正当化できると考えたが、**規約の不変条件「改善は承認制」への例外を、その場の判断で作った**ことに変わりない。

**対策**: 例外条項の新設は規約級の変更なので、**本書では条文案の提示に留め、採否はChami判断(§3-5④)とする**:
> 条文案: 「規約違反状態の是正のうち、①完全に可逆 ②実測で検証済み ③本番インフラ(Worker/D1作成・デプロイ等の明示的要承認操作)に触れない、の3条件を全て満たすものに限り、先行実施+即時開示を認める。1つでも欠けたら待つ。」

### 2-5. 音声入力の誤字が指示を反転させる

**実例**: 「応答**内**だ」と「応答**なし**だ」——1文字で意味が反転した。

**対策**: 規約200行のRequest Packet(曖昧・多義な依頼は着手前に原文/解釈/仮定を分離して残す)が既にあるので、新設はしない。**本書の差分は1点だけ**: 意味が反転し得る・かつ可逆・かつ軽量な依頼に限り、**返事を待たずミラーバック(「こう解釈した、違ったら止めてくれ」)を送って進む**(規約282行「迷ったら聞く=待つ」の限定的上書き。不可逆・重量級は従来どおり待つ)。ミラーバックを打った事実は受領台帳の行に1フィールド残す(後から数えられる形に)。

## §3. Chamiの考慮が足りていない点(依頼どおり率直に)

先に断っておく: **以下のどれも「Chamiが気をつける」で解くべきではない**。人間の注意力を当てにする設計は必ず壊れる(姉妹文書§2-1)。書くのは、システム側で吸収するために「何が起きているか」をChamiが知っている必要があるからだ。

### 3-1. セッションの満腹度は外から見えない、を前提にしていない
依頼を出す側からは、セッションが満杯寸前かどうか分からない。INC-96はChamiのせいではないが、**重要な依頼ほど「受領確認(N件と数えた)」を一瞥してから次を送る**と、消えない保証が効く。システム側: 受領確認を必ず返す(§2-1)。

### 3-2. Chamiが最後のQAになっている
今夜の11件+INC-96、**全部の起点がChamiの「おかしくないか?」**だった。人間の違和感は最強の検出器だが、(a)Chamiが疲れる (b)Chamiが寝ている間は検出器がゼロになる。§2の自動化は全部「Chamiの違和感をコードへ移す」設計だと理解してほしい。**違和感を感じたら遠慮なく言う、は続けてほしい**——ただしそれが「必要」な状態は減らしていく。

### 3-3. 「夜のうちにやっておいて」と「朝の判断が要る」の仕分けがない
改名の件が典型: 夜に実行すると静かに壊れて朝の手間が増えるだけ、という作業だった。依頼に**「朝でいい」「寝てる間に」「起きたら一緒に」のタグを一言つけてくれる**と、夜間の空回り(と、勝手に進めて朝に驚かせる事故)が減る。タグがなければ研究室が推定して**ミラーバックで宣言**する(§2-5と同じ型)。

### 3-4. 同じ内容を複数チャンネルに書く/1メッセージに複数依頼を積む
悪気なく起きる(音声入力なら尚更)。前者は二重対応の温床(INC-87の遠因)、後者は台帳の分解漏れの温床。**システム側で吸収する**: 台帳が1依頼=1行に分解し、受領確認で「N件と数えた」と返す。Chamiは何も変えなくていいが、件数が意図と違ったらそこで正してほしい。

### 3-5. Chami判断待ちの札(4枚・明示)
1. **標準承認リスト**(settings.json permission rule): どの操作カテゴリを事前承認にするか。決まるまで「Discordで承認したのに動かない」が続く。
2. **バックアップ設定の(B)履歴からも消す**: commit 92a95baに戦略フォルダ名が残存。(A)は実施済み。
3. **`G:\` 直下解決の固定**(QA指摘): 将来「共有ドライブ」が現れると保存先の解決が不定になり、そこへ世代削除が走り得る。
4. **先行実施の例外条項**(§2-4の条文案)を規約に入れるか。

## §4. セッション引継ぎ(巻/Vol)設計 v2 — 「切替時に書く」をやめる

### 現状の欠点
1. **切替時=コンテキスト最悪の時に、一番重い執筆をやる**設計になっている。質が一番落ちている瞬間に、一番質が要る文書を書く——逆立ちしている。
2. **ライブ状態が漏れる**: open依頼・未送信の報告・未コミットの変更・常駐の状態は、切替時の記憶頼み。INC-96の形がそのまま引き継ぎにも起きる。
3. **切替の判断基準が感覚**。「そろそろ限界」は退行が始まってから気づく(危険信号は稼働時間ではなく出力退行・同型反復)。
4. **巻数の正が古い**: CLAUDE.md§1は「現在Vol.7」のままだが、実セッションはVol.8として稼働中。正の更新漏れ(INC-89と同型)——CLAUDE.md更新を§6に入れる。

### v2設計(3本柱)

**柱1: 生きた引き継ぎ(narrative)**
- 巻Nの**開始日に** `docs/引き継ぎ/引き継ぎ_Vol(N+1).md` の骨組みを作り、決定・方針転換・教訓が出るたびに追記する。切替時には9割書けている状態が正。(※姉妹文書§3-1改善Cの「引き継ぎ_Vol8.md」という名は本則では「次の巻が読む=Vol9」——命名はこちらに統一する)
- 書くのは「次の巻が判断に迷わないための文脈」だけ。事実の羅列は柱2に任せる。

**柱2: 状態の自動生成(state)**
- `scripts/make_handoff.py`(新規)が切替時に機械生成:
  - `local/requests.jsonl` + triage合流分の open 全件(=やり残しの完全なリスト)
  - `git status` の未コミット(誰の作業か・パス限定コミットの候補)
  - 常駐プロセスとwaiter/presenceの状態
  - インシデント台帳の直近5件の見出し
- 記憶に頼る部分をゼロにする。**手書きは柱1のnarrativeだけ**。

**柱3: 切替トリガーの信号化**
- 時間や巻の長さではなく信号で判断する:
  - **黄信号: 1回目の要約圧縮** → 以後の依頼は台帳追記を確認してから応答。柱1の引き継ぎを最新化。
  - **赤信号: 2回目の圧縮/出力退行/同型反復の自覚** → 次のアイドルで切替。
  - 自己申告はこれまで通り有効。
- 姉妹文書§3-1改善Cの「明日切替」との関係: 矛盾ではなく**信号基準の初回適用**と位置づける(本セッションは圧縮を複数回経験済み=赤信号点灯中)。ただし柱2の `make_handoff.py` が未実装のうちに切替ると受け入れ条件を検証できないため、**§6で実装順を前倒しする(順2)**。
- **切替手順**: ①`make_handoff.py` 実行 ②未コミットをパス限定で退避コミット(または明示的に放棄と書く) ③新しい巻がboot時に台帳のopenをDiscordへ読み上げ——Chamiが取りこぼしゼロを一目で確認できる(§1のKPI④の測り方そのもの)。

### 受け入れ条件(切替を待たない——発火しない安全網は検証されない)

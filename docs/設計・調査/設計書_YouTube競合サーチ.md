## 0. Request Packet (原文/解釈/仮定)

- **原文**: 「分析部門やコピーライト部門が動いてもらうためにYouTubeの競合をサーチするシステムの設計書mdだけつくっといて。実装はまだいい」
- **解釈**: 競合チャンネル/競合Shortsの公開データを機械的に収集・蓄積し、分析部(shorts-analyst)とコピー部(copy-director)が「何が伸びているか」「どんな題名が効くか」を実測で読める状態を作る。
- **仮定(合理的既定・実装Go時に確認)**:
  1. 競合=「漫画/同人作品の紹介・宣伝系Shortsを出しているチャンネル」(うちと同じ土俵)。ゲーム実況等の無関係ジャンルは対象外
  2. 見るのは**公開メタデータのみ**(題名・再生数・高評価・投稿日時・長さ)。映像やサムネの取得・保存はしない
  3. 費用ゼロの範囲で運用(APIは無料クォータ内・課金要素なし)

## 1. 目的と成果物 (誰が何を得るか)

| 部門 | 得るもの | 使い方 |
|---|---|---|
| 分析部(shorts-analyst) | 競合の**伸び速度(views/日)・投稿頻度・投稿時刻の分布** | 自チャンネルの数字と並べ「市場の平均に対して勝ってる/負けてる」を判定。仮説更新の外部基準 |
| コピー部(copy-director) | **伸びた動画の題名コーパス**(速度順・特徴タグつき) | 題名のパターン学習(長さ/記号/数字/絵文字の傾向)。※模倣・転用ではなくパターン抽出(§7) |

## 2. 方式の結論

**既存GASプロジェクトへの拡張(新ファイル `gas/競合.gs`)+既存記録スプレッドシート(`SHEET_ID`)への新タブ追加** を採用する。

```
[時間トリガー] ──日次── runCompetitorDaily()   ─┐
              ──週次── runCompetitorDiscovery() ─┼─→ YouTube Data API v3 (YT_API_KEY・既存キー再利用)
                                                  └─→ 記録スプレッドシート(SHEET_ID)へ競合_5タブを追加
[doGet] action=comp_digest / comp_titles → JSON ←─ 分析部/コピー部がWebFetchで読む
```

- **★APIキーは新規発行不要**。既存の `YT_API_KEY`(GAS Script Properties・自チャンネルviews記録で稼働中=コード.gs:136)を再利用する。Chamiの手作業ゼロ。
- **★シート配置=既存の記録スプレッドシート(`SHEET_ID`・Chami呼称「AFI動画アナリティクス分析」・投稿履歴の本)へ新タブを追加する形にする(Chami指定2026-07-17)**。当初案の「別ブックに隔離」は撤回。
  - 理由: **既存タブ(記録_ch1/記録_ch2/集計/設定/Holidays)には一切触れず、新規タブだけを `insertSheet` で足す**のは安全。**現に既存コードが同じことをしている**(`snapshotStats`/`computePeaks_` が同じ本へ `_stats`/`_peaks` タブを `getSheetByName(x) || insertSheet(x)` で追加=コード.gs:796,805)。つまり「集計テンプレの取込構造を壊す」懸念は"既存タブを触る時"の話で、"新タブを足す"だけなら生じない。
  - 実装: `SHEET_ID` をそのまま使い、`COMP_SHEET_ID` は不要。新タブ名は衝突回避のため接頭辞をつける(`競合_チャンネル`/`競合_動画`/`競合_日次`/`競合_週次`/`競合_検索ログ`)。
  - ★ただし1点だけ検証必須(実装時): 記録本の「集計」タブや名前付き範囲(Holidays等)が**全シートを走査する数式/スクリプトを持たないこと**を確認する。持つ場合は新タブが集計に紛れ込む恐れ→その時だけ隔離を再検討。通常のテンプレ構造なら問題ない。
- デプロイは既存パイプライン(`GASを反映.bat`=clasp push→同一デプロイID)にそのまま乗る。clasp は gas/ 配下の全 .gs を push するため、**新ファイルを置くだけで追加コスト無し**。GAS=承認不要(Chami裁定2026-07-12)。

**不採用案と理由**:
| 案 | 不採用の理由 |
|---|---|
| Cloudflare Worker + D1 | D1新規作成=毎回Chami承認が要る。GASで足りる規模にインフラを増やさない |
| 別GASプロジェクト新設 | デプロイパイプライン(.clasp.json)が二重化し、GAS自動反映の設計(exec URL固定・ping検証)をもう1セット保守する羽目になる |
| フロント(ブラウザ)から直接収集 | APIキーが公開サイトに出る(秘匿違反)。ブラウザを開いている間しか動かない |
| スクレイピング(API不使用) | YouTube利用規約違反リスク+構造変更で静かに壊れる。公式APIで足りる |

## 3. データモデル (記録スプレッドシートSHEET_IDへ追加する競合_5タブ)

**競合チャンネル** (台帳・手動追加とdiscovery採用の両方):
| 列 | 内容 |
|---|---|
| channel_id | UC… (主キー) |
| チャンネル名 / URL | 表示用 |
| 登録者数 / 総再生数 / 動画数 | channels.list statistics (日次更新) |
| 状態 | watch(追跡中) / candidate(発見済・未採用) / ignore(対象外) |
| 発見経路 | seed(Chami指定) / search(クエリ名) |
| 追加日 / 最終更新 | |

**競合動画** (watchチャンネルの直近動画・追跡窓30日):
| 列 | 内容 |
|---|---|
| video_id | 主キー |
| channel_id / タイトル / 公開日時 / 長さ秒 | videos.list snippet+contentDetails |
| isShort推定 | 長さ≤180秒で推定(APIに公式フラグ無し=ヒューリスティック。精度限界を明記) |
| 初回取得日 | |

**日次スナップ** (append-only・時系列の原資):
| 列 | 内容 |
|---|---|
| 日付 / video_id / views / likes / comments | 毎日1行。前日との差分=その日の伸び |

**週次サマリ** (GASが毎週計算・Chamiが見るのはここだけでよい):
- 今週の速度トップ10(views/日・チャンネル横断) / チャンネル別投稿頻度 / 投稿時刻の分布 / タイトル特徴の集計(平均文字数・【】使用率・数字使用率・絵文字使用率)

**検索ログ** (discoveryの監査):
- 日付 / クエリ / ヒット数 / candidate化した数

## 4. 収集ジョブとクォータ会計

**日次 `runCompetitorDaily()`** (深夜・時間トリガー):
1. watchチャンネルの statistics 更新 — channels.list はid50件/1コール=**1 unit**
2. 各chのuploadsプレイリストから新着を拾う — playlistItems.list **1 unit×ch数**
3. 追跡窓(30日)内の全動画のviews/likes/commentsをスナップ — videos.list 50件/1コール=**~6 units**
4. 週次サマリの再計算(日曜のみ)+検索ログ整理

**週次 `runCompetitorDiscovery()`**:
- 検索クエリ(初期セットはChami承認・§9)で search.list — **100 units×クエリ数**。ヒットしたchをcandidate行として台帳へ(自動でwatchにしない=**採用はChami/分析部の判断**。ゴミが混ざるのを防ぐ)

**クォータ表** (標準10,000 units/日・※2025/12改定と2026/6専用バケットの変更があるため**実装時にコンソールで現行値を実測確認**=改善書の参考リンク参照):
| 消費者 | 頻度 | 概算 |
|---|---|---|
| 既存: 自チャンネルviews記録(YT_API_KEY共有) | 毎時 | ~50 units/日 |
| 競合tracking(30ch想定) | 日次 | ~40 units/日 |
| 競合discovery(10クエリ) | 週次 | 1,000 units(その日だけ) |
| **合計(最悪日)** | | **~1,100 units/日 = 上限の11%** |

- 上限接近の心配は実質なし。ただし**クォータはキー単位で共有**なので、競合側の失敗リトライで既存の views 記録を巻き込まないよう、**日次上限カウンタ(Script Propertiesに当日消費を記録・2,000超で自主停止)**を入れる。

## 5. 指標定義 (分析部の入力)

| 指標 | 定義 | 出どころ |
|---|---|---|
| 速度 | (今日のviews − 昨日のviews) / 1日。公開7日以内は「初速」として別枠 | 日次スナップの差分 |
| 投稿頻度 | 直近30日の本数/ch | 競合動画 |
| 投稿時刻分布 | 公開時刻(JST)のヒストグラム | 競合動画 |
| ER | (likes+comments)/views | 日次スナップ |
| タイトル特徴 | 文字数 / 【】有無 / 数字有無 / 絵文字数 / 頻出語トップN | 競合動画のタイトル |

## 7. 規約・安全 (破ってはいけない線)

1. **公開メタデータのみ**。映像・サムネ・コメント本文の取得・保存・転載はしない。分析出力に競合動画の内容説明を書く時も要約に留める
2. **模倣禁止**: コピー部はタイトルコーパスから**パターン(長さ・構造・語彙傾向)**を学ぶ。競合タイトルの転用・微修正流用はしない(著作権+inauthentic対策)。**煽り文言の学習も禁止**(規約防衛§6.1: 「YouTubeでは見せられない…」型は名指し違反パターン。競合が使っていても採らない)
3. **干渉禁止**: 競合チャンネルへの通報・コメント・いいね等の能動的接触は一切しない。観測のみ
4. **競合の実名はスプレッドシート内のみ**。repo(設計書・コード・commitメッセージ)に競合チャンネル名/IDを書かない(戦略情報はGitHubに上げない原則の適用)
5. **YT_API_KEYはGAS Script Propertiesから出さない**(既存方針の踏襲。ログ・エラーメッセージにも出さない)
6. **API利用規約**: 取得データの鮮度維持(追跡窓30日の統計は日次更新=準拠)。追跡窓を出た動画は生スナップを残さず集計値(週次サマリ)のみ保持
7. **費用**: ゼロ(APIは無料クォータ内・GAS/Sheets無料)。課金が発生しうる操作は本設計に存在しない=Chami承認既定(金銭発生時のみ都度確認)に照らして追加承認不要

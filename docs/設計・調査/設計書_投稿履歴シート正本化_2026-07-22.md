## §1 `prune_history` の処遇 ★最優先

### 現状

- `yt-clicks.js:1851 pruneSheet()` がボタン押下で `op: 'prune_history'` を GAS へ送る
- `gas/コード.gs:669 pruneHistory_()` が `keepIds`(端末ローカルの全 videoId)に無い行を `clearContent()` で消す
- 呼び出しは手動ボタン(`ytPruneSheet`)＋`window.confirm` のみで、自動発火はしない

### シート正本化後の危険性

`keepIds = allItems().map(it => it.videoId)` = **この端末の localStorage にある ID のみ**

シートが正本になった瞬間、意味が逆転する:
- 「シートにあってこの端末のローカルに無い行」= **別端末で投稿した行**
- それを `clearContent()` で消す = 今回直したい「端末をまたぐと履歴が消える」をシート側で再現

UIの確認文も嘘になる: 「この投稿履歴に無い行をシートから消去します」→ 正確には「この端末にない行を消します」

### 採用方針: フロント側の呼び出しを塞ぐ(GAS 関数はコードとして残す)

1. `yt-clicks.js:2319` の `pb.addEventListener('click', pruneSheet)` を削除
2. `index.html` の `ytPruneSheet` ボタンを `hidden` 属性付き非表示化
3. `gas/コード.gs:669 pruneHistory_()` 本体は残す(呼び出し経路が閉じられていれば発火しない)

「ボタン撤去 vs 無効化」: hidden でフロントから消し、GAS 関数は残置する方針を推奨。
全撤去より保守的で、将来「管理用に使いたい」が来た時に対応できる。

## §2 状態モデル: draft → confirmed の遷移

### 2段階の定義

| フェーズ | 定義 | 保管場所 | GAS 書き込み |
|---|---|---|---|
| **draft** | 動画生成完了〜「投稿完了」ボタン押下前 | localStorage のみ | なし |
| **confirmed** | 「投稿完了」ボタン押下後 | シート(正本) + localStorage(キャッシュ) | ある(1回・全情報) |

### draft フェーズで何が起きるか

```
① video-created → app.js が dispatch
② bluesky.js が video-created を購読 → Bluesky 投稿
③ 投稿完了 → histAdd() で localStorage に draft として記録
         → shortenAndShow() で短縮URL を取得
④ wizard が bluesky-posted を購読 → ステップ4(YouTube URL 入力)へ
```

draft フェーズの GAS 書き込みは**ゼロ**にする。
現在の `bluesky.js:1154, 1160` の `recordToSheet()` 呼び出しは削除対象。

### confirmed フェーズで何が起きるか

```
⑤ ユーザーが YouTube へ動画をアップロード
⑥ wizard ステップ4: YouTube URL を入力
⑦ 「投稿完了」ボタン(現:「記録して次へ」wizard.js:490) を押す
⑧ wizard.js: recordToGas() が1回のみ GAS POST(全情報を含む)
⑨ GAS: writeRecord_() が upsert → シートに確定記録
⑩ フロント: localStorage の既存 draft に confirmed マーカーを付与(任意。シートが正なので端末の扱いは軽い)
⑪ wizard がステップ5(完了)へ進む
```

YouTube URL は**このステップで必須**。
★★**バリデーションの位置は `writeRecord_()` の中ではなく `doPost` の入口のみ(裁定C)**。
`writeRecord_()` は `syncHistory_()` / 無人予約 / アカウント矯正など複数の経路から呼ばれる共有口であり、
そこに必須チェックを置くと `youtube_url` を送らない経路が全件拒否される(→データ喪失)。

ウィザード経路だけを識別する印として `op: 'wizard_confirm'` をクライアントから送り、
`doPost` の `writeRecord_()` 呼び出し直前のみで検査する:

```javascript
// wizard.js: recordToGas() の payload に追加
op: 'wizard_confirm',

// gas/コード.gs: doPost の writeRecord_() 呼び出し直前に追加 (gas:452 付近)
// ★writeRecord_() の中ではなくここにだけ置く
if (body.op === 'wizard_confirm' && !body.youtube_url) {
  return jsonOut_({ ok: false, error: 'youtube_url_required' });
}
```

## §3 書き込み順序の反転: 「投稿完了」ボタン押下で何が起きるか

### 現在の書き込みフロー(全経路)

| タイミング | コード | 送信内容 |
|---|---|---|
| Bluesky 投稿直後 | `bluesky.js:1154 recordToSheet()` | postUrl, postUri, title, videoId |
| 短縮URL確定後 | `bluesky.js:1160 recordToSheet()` | shortUrl, shareUrl (追加 upsert) |
| YouTube URL 入力後 | `wizard.js:592 recordToGas()` | youtube_url (追加 upsert) |
| 起動時補完 | `yt-clicks.js:847 reconcileYtToSheet_()` | ytUrl, shortUrl (未反映分の後追い) |

→ **GAS への書き込みが 3 回に分散**。シートが正本の間は「途中の部分記録」が中途半端なゴミになる。

### 新しい書き込みフロー

| タイミング | コード | 送信内容 |
|---|---|---|
| 「投稿完了」ボタン押下のみ | `wizard.js:592 recordToGas()` を拡張 | 全情報を1回(下記参照) |

削除する呼び出し:
- `bluesky.js:1154` の `recordToSheet()` → 削除
- `bluesky.js:1160` の `recordToSheet()` → 削除

残す/流用する呼び出し:
- `yt-clicks.js:847 reconcileYtToSheet_()` → **残す**。ネットワーク失敗後の自己修復として有効

### 拡張後の `recordToGas()` payload (wizard.js:592 周辺)

```javascript
{
  videoId:    W.videoId,          // 必須・upsertキー
  channel:    W.account,          // acc1/acc2
  youtube_url: W.ytUrl,           // 必須(ここで強制)
  postUrl:    W.postUrl,          // Bluesky 投稿URL
  postUri:    W.postUri,          // Bluesky at://
  shortUrl:   W.shortUrl,         // 計測用短縮URL
  shareUrl:   W.shareUrl,         // 概要欄掲載用URL
  workUrl:    W.workUrl,          // 作品URL
  title:      W.title,
  // fanzaInfo は fetchFanzaForRecord 経由で追加(現行の仕組みをそのまま使う)
}
```

GAS 側の `writeRecord_()` のインターフェースはそのまま流用できる。
バリデーションを追加する箇所: `gas/コード.gs:542 writeRecord_()` の先頭のみ。

## §4 競合の解決

### 現状の upsert ルール

`gas/コード.gs:584-586`:
```javascript
function put(h, v) { if (map[h]) sh.getRange(target, map[h]).setValue(v); }
function putIf(h, v) { if (map[h] && v !== '' && v !== null && v !== undefined) sh.getRange(target, map[h]).setValue(v); }
```

- `put()`: 常に上書き
- `putIf()`: 値が非空のときのみ書く → 既存値を空で潰さない

YouTube 動画URL は `putIf` (`コード.gs:600`):
```javascript
putIf('YouTube動画URL', f.youtubeUrl || '');
```
→ 一度書き込まれた YouTube URL は、空値では上書きされない。非空の別値なら上書きされる。

### 採用ルール: 後着勝ち (last-write-wins)

シンプルな理由:
1. 「投稿完了」は1セッション1回のフロー。同一 videoId が別端末から同時送信される場面は稀
2. GAS `LockService` (`コード.gs:436`) が同時書き込みを直列化している
3. 例外: YouTube URL が変わった場合(アップロードし直し)→ 後着で正しく更新される

競合時の優先度: **後着** → 特別なマージロジック不要。

## §5 オフライン/シート書き込み失敗時の扱い

### 端末だけ進む状態を作らないための手当て

「投稿完了」ボタン押下 → GAS 書き込み失敗 → ステップ5へ進まない、という設計にする。

```javascript
// wizard.js: recordToGas() の失敗時
recordToGas(function(ok) {
  if (!ok) {
    recordBtn.disabled = false;
    recordBtn.textContent = '記録して次へ ▶';
    setStatus('⚠️ シートへの記録に失敗しました。再度押してください。');
    return; // step5 へ進まない
  }
  _currentStep = 5;
  renderStep(5);
});
```

ただし現在の `recordToGas()` は戻り値を使わず `renderStep(5)` を直後に呼んでいる (`wizard.js:509`):
```javascript
fetchFanzaForRecord(function () {
  recordToGas();    // ← 失敗を無視している
  _currentStep = 5;
  renderStep(5);
});
```

→ ここを非同期の結果を待つ構造に変える必要がある。

### reconcileYtToSheet_() による自己修復

`yt-clicks.js:847` の reconcile は起動時に呼ばれ、youtube_url や shortUrl が未反映の行を補完する。
シート書き込みに失敗しても、次回起動時に reconcile が補完を試みる。ただしこれは **ytUrl と shortUrl の補完のみ**で、全情報を再送するわけではない。

→ 「投稿完了」ボタンを再押しするよう促す UI テキストを用意する。

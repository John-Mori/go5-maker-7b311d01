## §0. 結論(先に要点)

1. **一番効くのは「箱と台帳のSQLite一本化」**(P2/P5)。jsonl×5種+mvドレイン+手作業記帳という現行構造そのものが喪失の原因。世界の定石(トランザクショナルアウトボックス/軽量イベントソーシング)をSQLite1ファイル・自前200行程度で移植できる。
2. **ライブラリ丸ごと導入(huey/DBOS等)は今回は不採用**。世界のタスクキューは「処理者=常駐Pythonワーカー」前提だが、この組織の処理者は**Claude Codeセッション**(起こされて読んで返す)。ワーカーモデルが噛み合わない。借りるのは**パターンだけ**(lock→処理→ack、visibility timeout、dead-letter)。
3. **P1(生存判定)はClaude Code公式のhooksで根治可能**。PostToolUseフックで脈をtouchすれば「働いている間は脈が続く」= 閾値伸ばしの応急(120→900秒)を本質解に置き換えられる。※これは既にChamiへ提案済の「presence hook」と同一。世界調査でも公式ドキュメントが同じ手を示しており裏付けが取れた。
4. **P3(15秒ポーリング)はdiscord.py Gateway常駐で構造解消**。受信=Gateway(push・遅延ゼロ・レート制限消費なし)、送信=既存webhook(persona_send)のまま共存。差し替えは鳩(poller)1本のみ。
5. **P4(git共有worktree)はClaude Code公式worktree機能(v2.1.49〜)+git直列化ラッパーの二段構え**。worktreeだけでは.git共有部の競合(index.lock)が残る、というのが公式Issueにも載る既知の落とし穴。
6. Temporal/Hatchet/Restate/HumanLayer(SaaS)/GitButler/Discord系MCPサーバーは**規模過剰または前提不一致で不採用**(§5に理由を明記——将来の再調査の無駄を省くため)。

### 補助的な問題(構造7つの派生)
- 人格の同一性がsilentに壊れる(D1で関所は入れたが、登録・別名・短縮名の運用が手作業)
- 依頼がコンテキスト圧縮で消える(INC-96・受領台帳は手動記帳)

## §3. 推奨アーキテクチャ(到達形)

### 3.1 中核の判断: ライブラリではなくパターンを移植する

世界のキュー/ワークフロー基盤(huey・DBOS・Temporal…)は全て「処理者=常駐プログラム」を前提にする。この組織の処理者は**起こされて読んで返すClaude Codeセッション**であり、ワーカーとして常駐登録できない。従って:

- **キュー本体**: SQLite1ファイルの自前テーブル。litequeueのlock/ackモデル+outboxの順序原則を**約250行の1モジュールとして実装済**: `scripts/queue/leasequeue.py`(正本)。※当初、研究室と改修系セッションが並行して同等品(go5bus/leasequeue)を作ってしまったため、**leasequeue.pyへ一本化**した(2026-07-18)。統合時にackを「行削除」から「done保持」へ変更——行を消すと鳩の再起動時に同msg_idの冪等照合の相手が消え、二重処理が復活するため(テスト済・30項PASS)。
- **借用するパターン**(全て§2に出典):
  1. **lock→処理→ack**: 起床したセッションは箱をmvせず、`claim`(status=pending→claimed, claimed_by, claimed_at を1トランザクションで)する。処理完了で`ack`(status=done)。**箱から消えるのはackの後**=「先に消す」構造の廃止。
  2. **visibility timeout**: リース(lease_until・既定30分)が失効したclaimed行は次のclaimで自動的に再配達=処理者が死んでもメッセージは戻る。
  2b. **未claim放置の救済**(批評major指摘): リース失効はclaim済みしか救えない。「部門がそもそも起きておらず一度もclaimされない」ケースは、`stale_pending()`(pending・attempts=0・N分超)を研究室/sweepが定期的に見てエスカレートする=現行sweepの「脈切れ部門箱をmainへ回収」に相当する救済を未claimにも機械的に担保。
  3. **dead-letter**: リース失効3回でstatus=dead+Discordへ警報。「黙って消える」の反対=「うるさく残る」。
  4. **at-least-once+冪等**: msg_idにUNIQUE制約。再配達で二重応答しないよう、応答前にdoneチェック(現行のprocessed台帳チェックと同じ・DBで自動化)。

### 3.2 到達形の全体図

```
Discord ⇄ [gateway_listener(discord.py常駐・受信push)]     ← P3
              ↓ INSERT(UNIQUE msg_id)
         [go5.db(SQLite/WAL) messages・requests・incidents・approvals]  ← P2/P5
              ↓ 新着通知(waiterはDBのpending監視へ改修 or 現行ファイル通知を併用)
         [Claude Codeセッション(研究室+部門)] — claim→処理→ack
              │  ├ PostToolUse hook → 脈touch(liveness)              ← P1
              │  └ git操作は直列化ラッパー経由・部門はworktree分離     ← P4
              ↓ 送信は従来どおり persona_send(webhook/REST)
         [claude_responder(代打)] — liveness基準で発火・DBのpendingを処理
         [supervise_daemons] — gateway_listener/waiter/sweep/代打を監視  ← P6
         [承認] 軽量承認はDiscordリアクション→approvalsテーブル→waiter経路で再開  ← P7
```

### 3.4 生存判定の2信号化(P1) ★批評fatal指摘で設計変更済

- **2つの脈は物理的に別ファイル**(混ぜない):
  - **readiness** = `claude_active.txt`(従来どおりwaiterが監視中2秒毎にtouch)=「耳が箱を見ている」
  - **liveness** = `lab_tool_pulse.txt`(PostToolUse hookがtouch)=「道具を使って働いている」
- **セッション識別**: hookはstdinのsession_idを`lab_session_id.txt`(研究室が起動時に`claim_lab.py`で名乗る札)と照合し、一致時のみtouch。部門セッション・代打の`claude --print`(別session_id)は研究室の脈を偽装できない。札が無ければ何もしない=fail-open。
- **代打の発火判定(2信号の合成)**:
  1. readiness < 90秒 → 生存(耳が見ている=本人が応対)
  2. さもなくば liveness < 300秒 **かつ** readiness < **45分(硬い上限)** → 生存扱い(処理中の猶予=INC-94誤発火の根治)
  3. それ以外 → 死亡=代打
- 硬い上限の理由(批評fatal): livenessだけで判定すると「耳が死んだまま作業を続ける研究室」が新着を永久に放置できてしまう(hookの導入が900秒の安全網を撤廃する事故)。耳の停止45分で、働いていても代打を出す。
- 旧閾値900秒の応急は撤去済。真の死亡は readiness 90秒+liveness 300秒 で従来(900秒)より速く検知される。
- **実装状況(2026-07-18)**: `scripts/hooks/pulse_touch.py`・`claim_lab.py`・`.claude/settings.json`・`claude_responder.py`の2信号判定=実装・単体テスト全PASS。フックの実発火は設定読み込み後の新セッションから。

### 3.5 適用範囲の線引き(過剰設計の防止)

- 送信キュー集約(レート制限対策)は**今はやらない**(現規模で未発生)。
- DBOS/Agent Viewは**中期の研究項目**(まず実機検証)。本設計の到達形はそれらが無くても完結する。
- リアクション承認は軽量承認のみ。金銭・本番インフラはchatペイン承認のまま(規約維持)。

## §6. 本設計自体の限界(正直に)

1. **全層は家PC稼働が前提のまま**(memory: resilience-layers)。本設計はPC内の信頼性を上げるが、停電・Windows Update再起動には勝てない。ロック画面問題(CreateProcessAsUser)は研究枠に置いた——ここを本気で解くのは規模に対して重い。
2. **SQLiteの並行書き込みは「1ライターずつ」**。現規模(分あたり数件)なら余裕だが、書き込み頻度が桁で増えたら再設計点。
3. **移行期の混在が最も危険**(半分jsonl・半分DBの状態)。だからS2は「ダウンタイムを設けて一括」とした。
4. 出典の一部は個人ブログ/小規模OSS(表に信頼度を明記済)。採用したのは公式一次情報と教科書的パターンが中心で、個人ブログ由来は「パターンの借用」に留めた。

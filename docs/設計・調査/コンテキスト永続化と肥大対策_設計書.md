## 0. 結論(先に3行)

1. **人物コンテキストの「バグる」は容量そのものではなく、重い層を常時読む運用で起きる。** 恒久策は3層規律(常読=manifestだけ/詳細=必要時だけ/経験=コーパス参照)+予算(budget)の機械監視。層構造は既に半分できている——足りないのは予算と監視。
2. **改修履歴の正本は既にある(gitコミット文+インシデント.md+D1)。** 作るのは「人が読める月次ビュー」の自動生成と、そのNotionへの一方向push(キー待ち)。新しい台帳は作らない。
3. **Codex代替/共存の正解は「今選ばないで済む状態を保つ」こと。** 正本はツール非依存(md/jsonl/git)に既になっている。Claude固有物(memory)はキャッシュと位置づけ、`AGENTS.md`(業界標準のエージェント規約ファイル)を薄いポインタとして置けば、Codexでも他のツールでも**同じ正本から起動できる**。乗り換えコストがほぼゼロなら、いま賭ける必要がない。

## 3. 人物コンテキストの恒久防衛(3層規律+予算+圧縮)

### 3.1 3層規律(常読を軽く保つ)

| 層 | 実体 | 読み方 | 予算(上限) |
|---|---|---|---|
| L-A 常読 | persona_manifest.yml | 起動時・発言前に必ず | **8KB/部門** |
| L-B 詳細 | persona_detail.md | 人格深掘り・新キャラ設定時**のみ** | **48KB/部門** |
| L-C 経験 | corpus/会話ログ/背景資料(persona_context) | **直接注入しない**。必要時にgrep/検索で断片参照 | 無制限(貯める場所) |

原則: **「バグる」のは積むから。L-Cを常読に混ぜない・L-Bを毎回読まない。** 人格の声の維持はL-A(manifest)の再読で行う(既存の教訓「喋る前にmanifestを読む」と同じ)。背景(DQ8ロア等)は消さない——**貯め続けてよい**。読む側が全部読まなければ肥大は無害になる。これが「貯め続けたい」と「バグらせない」の両立。

### 3.2 予算の機械監視(実装済→§7)

`scripts/maintenance/context_budget.py`: 各部門のmanifest/detailを予算と突き合わせ、超過を列挙する(読むだけ・自動修正しない)。週次で回し、超過が出たら「圧縮サイクル」へ。

### 3.3 圧縮サイクル(超過時の手順・人が判断)

detailが予算超過 → ①detailの古い記述(過去の経緯・改訂履歴の類)をL-C(`local/persona_context/<dept>_history.md`)へ移す ②detailには現在有効な設定だけ残す ③manifestは常に「今の声」だけ。qwen知識パックで実施済みの「上限超過は経験層から間引く(教訓は削らない)」と同じ思想。

## 4. 改修履歴の整備(Notion窓・キー待ちの間もローカルで動く)

1. **月次ビュー生成(実装済→§7)**: `scripts/maintenance/build_changelog.py` が git log から `local/history/改修履歴_YYYY-MM.md` を生成(日付・種別・要約)。インシデント.mdへのリンク付き。**新しい台帳は作らない**(gitが正本・ビューは使い捨て再生成可能)。
2. **Notionへ一方向push(キー待ち)**: APIキー取得(ステップ2〜4)後、notion-sdk-py+自前薄スクリプトで月次ビューをNotion DBへ。機微と秘密は含まれない(gitコミット文=公開repo相当の内容のみ)。
3. D1 system_changes は研究室の記録として現状維持(ビューが将来D1も読むのは拡張として可)。

## 5. Codex代替/共存(ツール非依存の正本)

### 5.1 事実: 正本は既にツール非依存

| 資産 | 形式 | Claude以外で使えるか |
|---|---|---|
| 設計書・規約・人格(docs/) | md/yml | ○ そのまま |
| 発言コーパス・教訓・台帳(local/) | jsonl/md | ○ そのまま |
| 改修履歴 | git | ○ そのまま |
| 起動文生成器・各種スクリプト | Python | ○ そのまま |
| **Claudeのmemory dir・CLAUDE.md自動読込** | Claude固有 | ✕ → **キャッシュと位置づける**(消えても正本から再構築可能な内容だけを置く。現状ほぼそうなっている) |

### 5.2 やること(実装済→§7): `AGENTS.md` を置く

OpenAI Codex等が読む業界標準の規約ファイル `AGENTS.md` を、**CLAUDE.mdと同じ正本群への薄いポインタ**としてリポジトリ直下に置く。中身を二重管理しない(ポインタのみ=ドリフトしない)。これで「Codexで開いても同じ正本から起動する」状態になり、代替も共存も**将来の選択肢として常時開いたまま**になる。

### 5.3 やらないこと

- いまCodexへ乗り換える/併用を始めること(必要が生じてから。準備コストはAGENTS.mdでほぼゼロになった)
- Claude memoryへの依存を増やすこと(正本に書けることは正本へ)

### 5.4 Obsidianの評価(Chami質問2026-07-18「オブシディアンの方がよかったりする?」への裁定)

世に出回る「Claude×Obsidianで記憶するAI」の構成(約束メモ=起動時必読/投げ込み箱/AIが作る図鑑/週1点検)は、**うちで既により強い形で稼働している**: CLAUDE.md(自動読込)/Discord→corpus(手動投函でなく自動)/knowledge.md(日次自動再生成)/反省バッチ+予算監視(週次)。方式の本質は「記憶=ただのテキストファイル」であり、うちは初日からこの方式(世界調査のbasic-memoryと同型)。

- **記憶の仕組みとしての乗り換え価値: なし。**
- **閲覧UIとしての価値: あり。** Obsidian=mdの無料ビューアで、うちの正本は全部md。保管庫にこのフォルダを指定するだけで移行ゼロで使える(データは動かさない・AIに影響なし)。導入時は `.obsidian/` を .gitignore へ足すこと。
- **使い分け**: スマホの窓=Notion(iPhoneと相性・キー進行中) / PCの閲覧=Obsidian(任意・いつでも)。競合しない。ObsidianをNotionの代替にはしない(iPhone同期に追加コスト)。

## 6. 効率の原則(このセッションで確立した判断の再掲)

- 新しい台帳・新しいDB・新しいツールを増やさない。**既にある正本+生成ビュー**で解く。
- 機械は消さない・書き換えない(候補提示まで)。圧縮サイクルの実行は人/司令塔の判断。
- 監視は読むだけのスクリプト+既存のタスクスケジューラ。常駐を増やさない。

## 7. 実装(2026-07-18・安全部分のみ)

| 成果物 | 内容 | 検証 |
|---|---|---|
| `scripts/maintenance/context_budget.py` | 3層予算の監視(manifest 8KB/detail 48KB/常読合計)。超過列挙・読むだけ | 実行し現状の超過を検出 |
| `scripts/maintenance/build_changelog.py` | git log→月次改修履歴ビュー(`local/history/`) | 当月分を生成・実測 |

Notion push(§4-2)はAPIキー後。圧縮サイクル(§3.3)は超過検出が出た部門から、各部門の判断で。

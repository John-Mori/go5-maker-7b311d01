## §1. 欠点(実コード・実測根拠つき)

### 1-1. 人格設定書が「注入されていない」 — 書いた≠効いている
`ask_gemini.py` のシステムプロンプトは人格の詳細を **`(詳細=local/persona_context/homin_gemini_context.md)`というファイルパスの文字列で渡している**。Geminiはファイルを読めない——つまりChamiが2026-07-15に書いた設定書(健気さ・ファクト重視・曖昧さへの確認質問・断定回避)は**2日間、一度も効いたことがない**。特に痛いのは行動ルール(「曖昧なら1〜3個の確認質問」「断定回避」)が丸ごと落ちていること。**修正は数行**(ファイルを読んで注入するだけ)。

### 1-2. 規約§272の音声文字起こしは、実装がどこにも存在しない
規約は「音声が来たらGemini APIで文字起こし→`Chami(音声入力)`名義で貼る」と定めるが、**`inline_data`を実装したコードはリポジトリのどこにも無い**(grep確認済み)。ローカルwhisper(`transcribe.py`)は存在するがDiscord線に接続されていない。=「設計と実装の差分」(INC-95と同型)。過去に音声ミラーが動いたのは、セッションが手作業でやった時だけ。

### 1-3. キーワード振り分けの粗さが、Geminiの得意分野を奪っている
`WORK_WORDS`に「調べて」が入っているため、**調査質問は即エスカレ**される——調べ物はGeminiの得意分野なのに。逆に「バグって何?」のような用語質問も「バグ」を含むだけでエスカレ。8件中4件がエスカレだった一因はこの粗さの可能性がある(ログの`q`で検証可能)。

### 1-4. 教育・採点ループの対象外 — qwenだけが生徒
llm-edu(教育部門)の受講生はqwenのみ(**趣旨.mdの既決事項**)。知識パック共用でホイミンも受動的に賢くなるが、**ホイミンのミスを採点して還元する線が無い**。応答ログは材料を残している(mode=answeredはq/a両方、escalatedはqのみ・機微/無テキスト系はchannelのみ)——エスカレ判断の採点はqだけで可能なので、材料は足りる。※採点対象への追加は**llm-edu趣旨の変更**にあたるため、§3で正本改定をセットにする。

### 1-5. 救援役なのに、訓練したことがない
incident-recovery名簿では「Claude停止時の一次受付の要」。だが**Claude不在線(ABSENT_ACK→main箱エスカレ)が本番条件で発火した記録は無い**。「発火しない安全網は検証されない」(INC-78の教訓: 登録済み≠動く)——一番頼る場面で初めて動かす、が現状の設計になっている。

### 1-6. 静かな死角が4つ
1. **キー消失は黙って待つ**: `local/gemini_api_key.txt` が消えると、非表示コンソールにprintするだけで**誰にも知らせず**待機する(健康なまま無言化)。
2. **quota(無料枠)の消費が見えない・上限も未調査**: 全モデル429で沈黙する日が来ても、誰も気づけない。
3. **知識パックの鮮度が規律頼み**: `load_knowledge()`はファイルが**無い時だけ**生成する(2026-07-16に更新怠りが実際に発覚→是正した前歴あり)。※主対策は運用設計書§6順5の「知識パックhook(書き時強制)」——本書§4の鮮度チェックは**読み時の保険(二重底)**であって代替ではない。
4. **コンテンツポリシー違反→キーBANの経路**: このプロジェクトはFANZA(アダルト)運用。商品画像・作業スクショを不用意に送るとGoogle生成AIの禁止コンテンツに触れ、**キーBAN=ホイミン線の全停止**になり得る(§5の基準表とJ2のホワイトリストで遮断する)。

### 1-7. 機微部屋の一覧が3ファイルで3通りにドリフトしている(データ整理部門が2026-07-17実測)
同じ「機微部屋リスト」を3ファイルに写した結果、**全ファイルが別々に欠けた**:

| ファイル | 欠けているもの | 実害の経路 |
|---|---|---|
| `inbox_poller.py:353` | future-room | **新設の機微室「現在と未来」でqwenが一次回答し得る** |
| `local_responder.py:90` | hr-context | 同部屋のドレイン時ガード漏れ |
| `gemini_responder.py:51` | hr-context・future-room | 防御ガードが2部屋分薄い |

future-roomは2026-07-17新設——**新設した部屋は、触ったファイルにしか追記されない**。写した一覧は必ずドリフトする(現にした)。
**対策**: 単一登記(registry)+**許可リスト方式への反転**——`WORK_DEPTS`(仕事部屋)に無い部屋は**既定で機微**に倒す。INC-62の恒久対策(クラウド同期の許可リスト方式)の踏襲で、データ整理は同方式でdenylist非掲載の`recovery`部屋まで守れたことを実証済み。既存解=`core/storage-keys.js`(Go5Keys)と同じ型。

## §2. 使っていない強みと「定席」の設計

Geminiの強みは4つ: **無料**(ただし対価はデータ——§5の基準内で)・**長文脈**・**マルチモーダル**・**外部視点**(Claude系と別の学習系統=独立した第二意見)。

> 配役の原則(規約の現行条文): **「一次受付・要約・下書き・整形はGeminiへ。判断・コード・数字・データ編集は渡さない」**。本書はこれを緩和しない——J1/J3で数字が登場するが、**数字は機械(集計スクリプト)がテンプレートに直接埋め、Geminiは文章部分の整形だけを担う**。ホイミンの出力は常に「下書き/一次情報」であり、確定は研究室か部門が行う。

### J1. 朝の日報の文章化(毎日・定席) — **救援線の毎日訓練を兼ねる本命**
- **数字はGeminiに触らせない**: KPI4数字・open件数・常駐生死は機械集計値をテンプレートへ直接埋め込み、Geminiには**見出し・所感・読みやすさの整形だけ**を任せる。投稿前に機械値と日報中の数字を自動diffし、不一致なら投稿せず警告。
- 狙いは2段: ①朝の定型作業が減る ②**ask_gemini→persona_send の全経路が毎日1回、本番発火する**=救援線(§1-5)の実弾訓練が毎日になる。
- **沈黙の切り分け**: 日報が来ない日は原因が「故障/quota枯渇/キー消失」のどれか分からないままにしない——不達時は**原因コード付きの警報**(ログの429有無→キーファイル存在→プロセス生死の順で自動判定)をDiscordへ出す。
- **依存(順序が大事)**: 運用設計書§6の順1(受領台帳)と順7(KPI集計スクリプト)が先。**J1はその後**(§7参照)。数字の算出者・QA月次監査の定めは運用設計書§1のまま(Geminiは整形のみ)。

### J2. スクリーンショット一次読み取り(定席・ホワイトリスト前提)
- Chamiはスクショを多用するが、現状ホイミンは「テキスト以外→司令塔行き」。Geminiなら**画像の内容を3行で文字化してエスカレ文に添え**られる(判断はさせない・記述だけ)。
- **前提条件(これ無しでは実装しない)**:
  1. **対象部屋のホワイトリスト方式**: 機微部屋を除外(ブラックリスト)ではなく、**送ってよい部屋だけを列挙**する。FANZA関連部屋(商品画像=性的コンテンツの蓋然性が高い)は**対象外**——Google生成AIの禁止コンテンツに触れればキーBAN=ホイミン線全停止(§1-6-4)。
  2. **画像内の秘密は防げない、をChamiに周知**: 通常部屋のスクショにAPIキー・DM画面・個人情報が写り込むのは仕組みでは止められない。「外部AIに読ませてよいスクショだけ貼る部屋」の運用ルールとセット。

### J3. ログの一次分析(週次・定席・マスク済み抜粋のみ)
- 週1で「傾向の下書き」を作らせ(例: 「今週のエスカレ4件中3件は同じ型」)、kaizen-analyst(参謀)の材料整形係として働く。数字の確定は参謀側。
- **台帳の原文は送らない**: インシデント台帳・応答ログには組織の内部事情・Chamiの発言原文が含まれる。**機械が事前にマスク・抜粋した要約データだけ**を渡す(§5の基準表に従う)。

### J4. 第二意見パネル(臨時・実績あり)
- `短縮URL_3AI比較`で、**GeminiだけがCloudflare for SaaSという回避策に到達した**実績がある(Claude/ChatGPTは「不可」で切った)。一方で同文書にはGeminiの**未検証の断定**も残っている(pp.uaのクレカ登録要求「(要検証)」など)——つまり**多様性としての価値**と、**単独では信じず突き合わせる原則**の両方が読み取れる。
- 大きな設計書(改名・データモデル変更級)を出す前に、`ask_gemini.py` で**レッドチーム役として1回読ませる**選択肢を型にする(採否は研究室。送る内容は§5基準内に限る)。

### J5. 音声文字起こしの実装確定(§1-2の解消)
- 規約と実装を一致させる。方式はChami判断: **(a)Gemini API**(inline_data・規約原案どおり・クラウド送信) / **(b)ローカルwhisper**(transcribe.py接続・データを外に出さない・CPU負荷)。**推奨は(b)**。
- **(a)を採る場合でも、機微部屋の音声はGemini送信の対象外に固定**(該当chはスキップまたはローカル処理)——規約186行「クラウドへの機微持ち出し禁止」に従う。**(b)を採る場合は規約272/274行(方式=Gemini APIと明記)の改定パッチを添えてChami承認**を得る(黙って上書きしない)。

## §5. 送信可否の基準表(この設計書で一番大事な新設)

**前提の事実**: 無料枠Gemini APIの入力・出力は、Googleの製品改善(人間によるレビューを含む)に使われ得る。「無料」の対価はデータ。だから**何を送ってよいかを先に決める**:

| データ | 送信 | 理由 |
|---|---|---|
| 機械集計の数字(KPI・件数・生死) | ○ | 個人情報でも戦略でもない |
| 公開済み情報(公開URL・公開設計書の一般論) | ○ | 既に公開 |
| gemini-reception部屋の質問文 | ○ | この部屋は「外部AIに読ませる前提」の部屋と定義する |
| ホワイトリスト部屋のスクショ(J2) | △ | 部屋単位で許可+画像内秘密は運用ルールで(§2 J2) |
| 台帳・ログの**マスク済み抜粋** | △ | 機械マスクを通したものだけ(J3) |
| 台帳・応答ログの**原文** | ✕ | 内部事情・Chami発言原文を含む |
| 機微部屋の一切(文・画像・音声) | ✕ | 規約186行(クラウド持ち出し禁止) |
| FANZA関連の画像・商品情報 | ✕ | コンテンツポリシー→キーBAN経路(§1-6-4) |
| 鍵・トークン・個人情報 | ✕ | 論外(既存方針) |

**選択肢の併記**: 有料枠(入力を学習に使わない契約形態)へ切り替えれば△〜✕の一部は緩和できる。ただし現状の流量では過剰投資——**まず無料枠+この基準表で始め、J1〜J3が定着してから再検討**を推奨する。

2. **Gemini専用ボットは作らない**(規約裁定済み: webhook相乗りで足りている)。

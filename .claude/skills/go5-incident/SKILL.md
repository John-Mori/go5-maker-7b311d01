# インシデント記録

## なぜ記録するのか
インシデント.mdは、安いモデルが過去の穴を二度と踏まないための土台(最重要の工数レバー)。そして「commitメッセージに書いた」だけでは、Chamiから見えない=記録していないのと同じ(INC-83)。不具合を直したら、必ず台帳(インシデント.md)へ追記する。

## 記録する/しないの判定
記録する: **実際に誤動作・欠落・停止・誤報を起こしたもの**(バグが本番で発現した/データが消えた/セッションが黙った/誤った通知が出た 等)。
記録しない: 未然に気づいた設計上の懸念・レビュー指摘で直した潜在バグ(発現していないもの)・単なる改善や機能追加。
迷ったら: 「Chamiや運用に実害が出たか」で判断。出たら記録。

## 記録の様式
インシデント.md の該当カテゴリ(§3 A〜F)へ、新しいものを上へ追記:
- **INC-番号**(連番。既存の最大+1)
- **症状**: 何が起きたか(1行・具体的に)
- **真因**: なぜ起きたか(表面でなく根本原因)
- **対策**: どう直したか + 再発防止(コード/規約/手順のどれで塞いだか)
- **日付**と Chami承認の有無

## 手順
1. 不具合修正をcommitした直後(=完了報告の前)にやる。
2. 最新INC番号を確認: `grep -oE "INC-[0-9]+" インシデント.md | sort -t- -k2 -n | tail -1`。
3. 新INC番号で追記(症状/真因/対策)。
4. commitして、その後に完了報告。報告の中で「INC-NNNとして記録した」と明示する。

## why(様式の理由)
真因まで書くのは、症状だけだと同じ根本原因の別症状を防げないから。再発防止をコード/規約で塞ぐのは、記憶や注意力に頼らないため。番号を振るのは、後から「同型の再発」を突き合わせられるようにするため。

# 統合設計書：5秒動画メーカー × 投稿スケジュール最適化

> **この1ファイルで「なぜ統合するか・どう繋ぐか・どんな順で作るか・何が未確定か」が分かる。**
> 既存2システムは共に**バニラJS・純client-side・GitHub Pages**前提。本書はそれらを**1アプリ**に統合する設計。

---

## ⚡ 30秒サマリ

- **何を作るか**：「📅最適スロットを決める」→「🎬その枠の動画を作る」→「🦋予約/即時で投稿」→「結果(URL/状態/クリック)をスロットに書き戻す」→「🧪検証で時刻を確定」を**1つのアプリ・タブ切替**で回す統合ツール。
- **中核の発想**：**スロット(slot)が全ての中心オブジェクト**。スケジュール側のスロットには既に `title / video_id / url / scheduled_at / status` という"投稿管理"フィールドがある＝動画ツールの成果物（投稿URL・状態・クリック数）をそのまま書き戻せる。
- **決定済み**：①1アプリ統合 ②Bluesky中心(client-side) ③一気通貫 ④Supabase保存。
- **最大の論点（§7）**：**「予約投稿」**。Blueskyに予約投稿APIは無い。client-sideだけだと"アプリを開いている間"しか自動投稿できない。真の無人予約には小さなサーバーレス(cron)が要る。→ 段階導入で解決。
- **もう一つの要注意（§6.4）**：B系の検証KPIは**YouTube Studio前提**。Bluesky中心にするなら**検証指標をBluesky用に再定義**（いいね/リポスト/プロフィール遷移/Bitlyクリック/FANZA成約）。

## 1. 目的（なぜ統合するか）

2つのシステムは、同じ事業（成人寄りコミックス紹介の5秒動画でアフィリエイト成約）の**上流と下流**：

| | 役割 | 現状 |
| :--- | :--- | :--- |
| **B：スケジュール最適化** | "いつ・どの枠・どのジャンルで出すか"を決める（day-type分類・カレンダー・検証） | ローカル稼働・YouTube Shorts想定 |
| **A：動画メーカー** | "作って投稿し、成果を記録する"（5秒動画・Bluesky投稿・アフィリンク・Bitly記録） | GitHub Pages 公開・稼働中 |

別々だと「カレンダーで決めた枠を見ながら、別アプリで作って、手で投稿し、結果をまた手で台帳に書く」分断が起きる。**統合すると「スロットを起点に作成→投稿→記録→検証が途切れない」**＝運用工数とミスが激減し、検証データも自動で貯まる。

## 3. 統合アプリの構成（タブ）

1つの `index.html` ＋タブバーに集約（A系の3タブ実装＝`affiliate.js` の `TABS` を拡張）。

| タブ | 由来 | 内容 |
| :--- | :--- | :--- |
| 📅 **カレンダー** | B系 | 5週ローリング・day-type・スロット一覧・予約状態。**各スロットから「この枠を作る／投稿する」へ遷移** |
| 🎬 **動画作成** | A系 | 写真＋文字→5秒動画。選択中スロットの role/genre をヒント表示。完成→投稿へ |
| 🦋 **投稿** | A系 | 本文プレビュー＋Bluesky投稿（即時／予約）。投稿後スロットへ書き戻し |
| 🔗 **アフィリンク** | A系 | FANZAアフィリンク生成（本文へ貼る生リンク） |
| 🧪 **検証** | B系 | 4週検証（**Bluesky指標に再定義**）・割れ枠決着・CSV出力 |

> モバイル幅ではタブが5つになるため、タブバーは横スクロール or 2段折返しを採用（§8 UI課題）。

## 4. 中核データモデル：slot（統合の要）

B系 `schedule/schema.sql` の `slots` を**正本**とし、A系の投稿成果フィールドを追記して**1テーブルで計画も成果も持つ**。

### 4.1 既存（B系・計画側）
`id="<date>#<index>", date, slot_index, scheduled_at, day_type, role, genre, peak, status, verify_flag, alt_hypothesis, needs_review …`
status enum：`未着手 / 制作済・未予約 / 予約登録済 / 公開済 / 取り下げ`

### 4.2 追記（A系・成果側／本統合で追加）
| 列 | 型 | 意味 |
| :--- | :--- | :--- |
| `channel` | text | 投稿チャネル（当面 `bluesky` 固定。将来 youtube 等） |
| `post_uri` | text | Bluesky の at:// URI（投稿の一意ID） |
| `post_url` | text | `https://bsky.app/profile/.../post/...`（共有URL） |
| `short_url` | text | Bitly 短縮URL（クリック計測対象） |
| `bitly_id` | text | Bitly ID（クリック集計キー） |
| `affiliate_url` | text | 本文に貼った生アフィリンク（無改変＝af_id保持） |
| `image_ref` | text | 添付画像の参照（任意） |
| `click_count` | int | 投稿短縮URLの開封数（Bitly／§6.4） |
| `engagement` | jsonb | like/repost/reply 数（Bluesky API取得・任意） |
| `posted_at` | timestamptz | 実投稿時刻（自動／予約の確定時刻） |

> **データ保全の原則（B系踏襲・必須）**：`予約登録済`・`公開済` はテンプレ再計算で時刻/役割を**自動変更しない**（`needs_review` のみ）。`post_uri` 等の成果データは破棄しない。

### 4.3 状態遷移（統合後）
```
未着手 ─(動画作成)→ 制作済・未予約 ─(予約セット)→ 予約登録済 ─(scheduled_at到達/即時投稿)→ 公開済
                                   └────────────(即時投稿)──────────────────────────┘
公開済 ─(Bitly/API取得)→ click_count・engagement 更新 → 🧪検証へ
```

## 6. 一気通貫フローの設計（決定③）

### 6.1 スロット起点の制作
- 📅カレンダーのスロットに「🎬作る」ボタン。押すと 🎬タブへ遷移し、`slot.genre/role/day_type` を**ヒント表示**（例：「本命(土)／ギャル・ハーレム系」）。
- 動画完成時、現在のスロットへ `status=制作済・未予約` と素材参照を記録。

### 6.2 投稿（即時／予約）
- 🦋投稿タブで本文プレビュー（A系の既存UI）。アフィリンクは生リンクを本文末尾へ。
- **即時**：`blueskyPostRaw` で投稿 → `post_uri/post_url` 取得 → Bitlyで `post_url` 短縮 → slot を `公開済` に更新。
- **予約**：`scheduled_at` を持たせて `予約登録済`。実投稿は§7の方式で `scheduled_at` 到達時に実行。

### 6.3 書き戻し
- 投稿成功で slot に `post_uri/post_url/short_url/bitly_id/affiliate_url/posted_at/status` を保存（Supabase）。

### 6.4 計測・検証（**Bluesky用に再定義：重要**）
- B系の `Viewed率/平均視聴` は**YouTube Studio専用**でBlueskyでは取得不可。統合では検証KPIを差し替える：
  - **Bitlyクリック数**（投稿短縮URLの開封＝到達の代理指標）
  - **Blueskyエンゲージメント**（like/repost/reply：API取得可）
  - **プロフィール/関連遷移**（直接計測は限界。Bitly中心）
  - **FANZA成約**（FANZA管理画面が正＝手入力 or 後日連携）
- 4週検証の骨子（W1基準→**W2 時間帯AB（割れ枠決着）**→W3曜日→W4フック）は維持。指標だけ上記に置換。`verification_plan.json` に `metric_set: bluesky` を追加。

## 7. 最大の論点：予約投稿（要・方針確定）

**Bluesky には「予約投稿」APIが無い**（`createRecord` は即時）。`scheduled_at` に自動投稿するには実行主体が要る：

| 方式 | 仕組み | 長所 | 短所 |
| :--- | :--- | :--- | :--- |
| **A. client-side スケジューラ** | アプリを開いたタブが1分ごとに「期限到来スロット」を確認→投稿 | サーバー不要・今の構成のまま | **端末/タブを開いて起きている時だけ**動く（無人不可） |
| **B. サーバーレス cron（推奨・段階導入）** | GAS時間トリガー or Cloudflare Worker が Supabase の期限スロットを読み、Blueskyへ投稿 | **無人で正確**・端末非依存 | 小さなサーバー＋**そこにアプリPWを保持**（秘匿管理が必要） |
| **C. 半自動** | 予約時刻にブラウザ通知／リマインドし、人がワンタップ投稿 | 安全・実装軽い | 完全自動ではない |

- 決定④で **Supabase 採用済**＝Bには好相性（cronがDBを見るだけ）。
- **推奨段階**：Phase3で **A（開いている間の自動）＋C（通知）** を先に実装 → Phase5で **B（サーバーレスcron）** を足して無人化。
- ※ 「client-sideで完結」希望と「無人予約」は本質的に両立しないため、無人化時のみ最小のサーバーレスを1つ足す（YouTube/Xの時と同じ整理）。秘匿情報はそのサーバー側にのみ置く。

## 8. 技術・実装上の論点

1. **リポジトリ統合先**：稼働中の **A系repo（go5-maker-7b311d01）を母体**にB系を取り込むのが最短（既にGitHub Pages＋Bluesky稼働）。B系のPython（祝日・ビルド）も同梱。→ §9で確認。
2. **名前空間統一**：A系IIFE群を `window.SCH.*` に寄せ、衝突回避（B系流儀）。
3. **データ層**：`store.js` のアダプタを Supabase に一本化。slots/overrides/verification を同DB。**秘匿情報（アプリPW・Bitlyトークン）はDBに入れない**（端末localStorage／サーバーレス側）。
4. **タブUI**：5タブはモバイルで窮屈→横スクロール or 2段。📅カレンダーは横広→モバイルは週送り＋縦リスト。
5. **キャッシュ運用**：A系の `?v=N` を全体に適用。schedule埋込データは `build_app_data.py` 踏襲。
6. **テスト**：B系 `test_logic.js`(31項目)＋A系 `test_bluesky.js`/`test_affiliate.js` を統合リポジトリで一括実行。
7. **セキュリティ**：公開サイトに秘密を置かない原則は両系共通。Supabaseは**RLS**（B系 schema.sql 準拠）。アプリPWは端末のみ。

## 10.5 Phase 5 詳細設計（無人化＝サーバーレスcron）

**目的**：アプリ（タブ）を閉じていても、`scheduled_at` に Bluesky へ自動投稿する。Phase3はタブを開いている間限定だったが、Phase5でサーバー側が肩代わりする。

### 前提（ここで初めて必須になる）
- **Supabase 切替**（§9-4の「後で切替」をここで実施）。予約・本文・画像参照・認証をDBに置く（端末非依存にするため）。
- **サーバーレス実行主体**：Google Apps Script（時間トリガー）または Cloudflare Workers（Cron Triggers）。既存の記録用GASと同じ思想で、**秘匿情報はサーバー側だけ**に置く。

### データモデル追加（Supabase）
`reservations` テーブル：
| 列 | 意味 |
| :--- | :--- |
| `id` (uuid) | 予約ID |
| `slot_id` | 紐づくスロット |
| `scheduled_at` (timestamptz) | 投稿予定時刻 |
| `text` | 投稿本文（生アフィリンク込み・無改変） |
| `image_path` | 画像の保存先（Supabase Storage のパス。任意） |
| `status` | `pending / posting / posted / error` |
| `result_uri` / `result_url` | 投稿結果（書き戻し用） |
| `error` | 失敗理由 |
- **認証**：Bluesky ハンドル＋アプリパスワードは **DBに平文で置かない**。サーバーレスの秘密ストア（GASスクリプトプロパティ / Workers Secrets）に保持し、cronがそこから読む。画像はSupabase Storageに置き、cronが取得して `uploadBlob`。

### フロー
```
[クライアント] 予約作成 → Supabase reservations(pending) ＋ 画像をStorageへ
        ↓（端末は閉じてOK）
[サーバーレス cron 毎分] reservations で scheduled_at<=now かつ pending を取得
        → 画像取得 → Bluesky createSession→uploadBlob→createRecord（本文＝無改変アフィリンク）
        → status=posted, result_uri/url 保存
        ↓
[クライアント 次回起動時] reservations/slots を読み、検証ダッシュボードへ反映
```
- 冪等性：cronは1件ずつ `pending→posting` に原子的更新（`update ... where status='pending'` の戻り行数で排他）してから投稿。二重投稿防止。
- リトライ：`error` は一定回数まで再試行。最終失敗は通知（メール/Push）。

### 段階
- **5a**：Supabase 切替（store.js を supabase アダプタへ。schema.sql ＋ reservations 追加・RLS）。
- **5b**：予約作成をDB化（Phase3のin-memory→Supabase。画像はStorage）。
- **5c**：cron 実装（GAS時間トリガー or Workers cron）＋秘密ストア設定＋デプロイ手順書。
- **5d**：クライアントはcron投稿済みを読み込み表示（検証へ反映）。

### 必要な確認（5着手前）
1. サーバーレスの実行基盤：**GAS（無料・既存知見あり）** か **Cloudflare Workers（cronが分単位で安定）** か。
2. Supabase プロジェクト作成・鍵の用意（§9-4）。
3. 画像の保管：Supabase Storage を使うか（容量・公開設定）。
4. 失敗通知の手段（メール/Push）。

> ⚠️ Phase5は**クライアントだけでは完結しない**（無人実行＝サーバー必須）。Phase1〜4（client-side完結）とは前提が変わるため、上記インフラの用意が前提。

## 11. リスクと対策
- **無人予約 vs client-side完結の矛盾** → 段階導入（§7）。無人化時のみ最小サーバーレス。
- **Bluesky指標の限界** → 到達はBitly代理、成約はFANZA管理画面が正と明示（過大評価しない）。
- **2系統のデータ衝突** → slotを単一正本にし、A系の独立localStorageは設定/秘匿のみに限定。
- **大規模改修のデグレ** → Phase1は「移植＝挙動不変」を厳守し、既存テストを通してから機能追加。
- **規約/BAN** → 露骨表現・直接誘導文を助長しない（両系の既存方針を継承）。
